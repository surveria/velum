# Refuted findings

> These first-pass audit claims did NOT survive adversarial verification. They are kept for transparency, so it is visible what was examined and rejected.

## [exec-async] Direct-builtin fast paths for 'unresolved' identifiers guard with an incomplete shadow check

- Location: `src/runtime/binding/static_bindings.rs`:328
- Original claim: A fast path that resolves identifiers to builtins by name must be guarded by exactly the semantic resolution it bypasses; here the guard omits one of the places a shadowing binding can live (the global object), so a user override installed as a property (sloppy `parseInt = f` at top level creates exactly that, via assign_bytecode_or_create_sloppy_global → define_global_object_data_property) can plausibly be ignored on subsequent calls. The duplicated allowlist tables can also drift: builtin names appear in one table and not the other today (e.g. MATH_NAME, JSON_NAME only in builtin_value).

**Why refuted:** The cited code shape is real (two match tables, guard checking only `globals` + local scopes), but both legs of the "why bad" reasoning fail. First, the claimed override vector does not exist: sloppy `parseInt = f` never reaches define_global_object_data_property, because get_or_materialize_binding_bytecode (declaration.rs:25, static_bindings.rs:388-399) materializes the builtin into `builtin_globals` as an immutable Const cell (insert_global_builtin, native/core.rs:332), so the assignment throws "assignment to constant" — a deliberate, tested engine semantic (tests/global_numeric_constants_smoke.rs:53-58). Second, even for an own property installed via `globalThis.parseInt = f`, the fast path cannot diverge from the resolution it bypasses: the slow path itself (eval_bytecode_identifier, bytecode/ops/mod.rs:81-97) consults binding scopes and builtin_value BEFORE the global-object own-property fallback, so bare-identifier reads/calls of builtin names ignore global-object own properties with or without the fast path — the guard covers exactly the locations (globals, locals from current_local_frame_start, with-envs checked by callers) where the slow path could resolve differently. Third, the "drift today" evidence is explained: the names present only in builtin_value (MATH, JSON, ATOMICS, REFLECT, PERFORMANCE, globalThis, NaN, Infinity) are exactly the non-callable builtins, and direct_builtin_callable_value is deliberately callable-only; a future omission from the callable table is fail-safe because the caller (runtime/mod.rs:523) falls through to the binding lookup and yields the same value.

## [hack-hunt] Array-like length semantics split across seven allowlisted functions in three files

- Location: `src/runtime/native/builtins/array/generic.rs`:444
- Original claim: One concept — 'the length of an array-like, clamped for this consumer' — has seven owners in three modules with three different clamping/error policies; Function.prototype.apply (bound.rs) and array callbacks can drift from Array built-ins independently. The allowlist stops growth but the split itself has no assigned AS item, unlike every other frozen duplicate.

**Why refuted:** The allowlist and line numbers are real, but the architectural indictment fails on all three load-bearing points. First, the seven functions are not seven owners of one concept: only array_like_length, its two-line receiver-check wrapper array_like_length_for_callback, and array_like_length_from_value read an array-like length; the rest are a length setter, the spec's push/unshift 2^53-1 overflow check, a usize-to-Number formatter, and a max constant. Second, there are no divergent clamping policies: every reader delegates to the single canonical Context::to_length (conversion.rs:143) plus the shared length_to_usize (conversion.rs:168), differing only in the error string for 32-bit usize overflow, so apply/callbacks cannot drift from Array built-ins in ToLength semantics — the same allowlist guard rejects any second definition. Third, the split does have an assigned AS item: AS-03b1b (complete, PR #413; architecture-stabilization-plan.md:827,860-870; semantic-architecture-inventory.md:310,668) explicitly documents this exact design — one specification owner with per-consumer engine-resource checks kept separate from JavaScript semantics — so the allowlist enforces a finished invariant rather than freezing an unresolved duplicate.
