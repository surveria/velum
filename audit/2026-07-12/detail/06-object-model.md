# Object model: ObjectHeap, properties, semantic facade

> Детальный отчёт аудита архитектуры rs-quickjs. Подсистема: `object-model`. Сгенерировано многоагентным аудитом 2026-07-12; каждая находка уровня medium/high прошла адверсариальную проверку отдельным агентом. Материал на английском (исходный текст аудита).

## Subsystem assessment

The object/property/semantic_object stack is a disciplined three-layer design: physical storage (ObjectHeap: SlotArena of Object records, an interned ShapeTable, dense/sparse array storage), a semantic pre-dispatch facade (semantic_object/*) that resolves Proxy/function/global/string/typed-array exotics before handing caches an explicit 'ordinary ObjectHeap tail', and per-site monomorphic caches (static_names) guarded by receiver shape plus one VM-global prototype-lookup version. Invariants are defended aggressively — every id access is checked, storage growth goes through a reservation ledger, prototype walks carry cycle budgets — and the documented AS-01..AS-09 program has genuinely consolidated equality, conversion, and internal-method dispatch into single owners. The remaining weaknesses cluster in three places. First, the Object god-record is real but well-fenced; the worse structural problem is the ShapeTable: append-only, deduplicated by linear full-slice scan, with a whole-layout copy on every existing-property write and per-property transitions during seal/freeze — quadratic time and permanent quadratic memory that sits under every property operation. Second, semantics still leak by property *name string*: '__proto__', 'length', and canonical-numeric-index recognition are re-implemented by string comparison in at least five modules each, with an uncatchable engine error as the backstop when a path forgets, and for-in enumeration exists twice (physical collect_keys vs semantic keys) with a shared shadowing bug and a proxy-in-prototype-chain bypass. Third, invalidation is maximally coarse: one global epoch flushed by any structural mutation anywhere (including every Array#shift), which is safe but means the cache tier degrades to zero under ordinary allocation-heavy workloads. Side tables pairing ObjectId to promise/collection/generator stores are correctly swept but the association plumbing is copy-pasted three times with two different slot-reuse strategies. Overall: the semantic layering is sound and unusually honest about its debt; the physical shape/enumeration/string-dispatch substrate underneath it is the part that will not scale and is mostly undocumented.

## Strengths

- Every fast path observed (packed array ops, cacheable lookups, native-call caches) has an explicit guard and a genuine fallback to the generic semantic path; guards check storage kind, attributes, sparse keys, and indexed prototype pollution before bulk operations (e.g. holey_properties_without_indexed_prototype in array/fast.rs).
- The semantic_object facade's 'pre-dispatch + ordinary tail' contract is consistently applied: caches only ever fill from CacheablePropertyLookup produced after Proxy/string/global/typed-array dispatch is resolved, and every Uncacheable outcome re-enters the same generic finish path.
- Storage accounting discipline is exceptional: object creation, property growth, shape creation, and byte-buffer resize all reserve/commit/rollback through one ledger, and GC sweep reconciles payload totals; no unwrap/panic/unchecked indexing anywhere in 12k lines.
- Attribute state uses typed two-variant enums (PropertyWritable/PropertyEnumerable/PropertyConfigurable, ObjectExtensibility) instead of boolean soup, and ObjectProperty::set_value structurally cannot violate non-writable data or accessor slots.
- Prototype traversals uniformly carry an explicit cycle budget (PrototypeTraversalBudget) rather than trusting acyclicity, and the GC's association sweep (gc.rs sweep_dead_associations) keeps ObjectId-keyed side tables consistent with arena slot reuse.
- The self-documentation is real: the god-record, side-table allowlist, global-object split, function-store extensibility gap, and native-constructor newTarget patching are all recorded in docs/semantic-architecture-inventory.md with named AS owners and mechanical guards in scripts/check-architecture-boundaries.sh.

## Confirmed findings (adversarially verified)

### [HIGH] ShapeTable is append-only with linear full-slice dedup: O(S*k) per transition, O(k^2) writes and permanent quadratic memory on seal/freeze

- **Kind:** `inelegant` · **Location:** `src/runtime/object/shape.rs:216` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** shape_for_properties: `self.shapes.iter().position(|shape| shape.properties() == properties)` — every new shape linearly compares against every existing shape's full layout slice. transition_after_update (line 148) copies the entire property list into a fresh Vec on EVERY write to an existing property (called from set_named_property_value, object/mod.rs:485, even when nothing changes). seal/freeze (integrity.rs:84-102) run transition_after_update once per property, creating up to k-1 intermediate shapes; a shape with k properties stores k layout entries plus k offsets, and shapes are never removed — sweep_unmarked_objects (base.rs:91) does not touch self.shapes.

**Why it matters:** There is no transition tree or hash index, so shape lookup/creation cost scales with total shapes ever created times layout width; building one object with k properties is O(k^2) plus k table scans, and Object.freeze on a k-property object permanently retains ~k^2/2 layout entries charged as CacheEntry. Because every named-property write funnels through transition_after_update, the whole property-write path inherits an O(k) allocation. This is the storage substrate under everything else in the subsystem, so it silently caps object counts and property widths the engine can handle.

**Verifier notes:** shape.rs:216-220 does exactly what the claim says: shape_for_properties linearly scans every shape ever created with full-slice equality, and the table (ObjectHeap.shapes, base.rs:48) is global and append-only — sweep_unmarked_objects (base.rs:91) and gc.rs never touch it, and ShapeTable has no removal API. transition_after_update (shape.rs:148-172) allocates and fills a k-element Vec before checking `changed`, and is called unconditionally on every existing-property write from set_named_property_value (mod.rs:485), so the write path carries an O(k) allocation even for pure value writes. seal/freeze (integrity.rs:84-102) transition once per property, minting up to k novel shapes of width k, each permanently charging 2k+1 CacheEntry units (shape.rs:225-233, reserve+commit with no release) — the claimed ~k^2/2 retained entries is actually an underestimate (~2k^2). The only softening facts (slice == short-circuits on length, unchanged writes skip the table scan, dedup helps monomorphic workloads, ledger limits bound memory by erroring) refine constants but do not defeat the architectural point.

**Verifier correction:** Two quantitative refinements: (1) Rust slice equality rejects different-width shapes in O(1), so the O(S*k) scan cost concentrates among same-width shapes (scan is still Omega(S) per new-shape creation); plain value writes that change nothing return before the scan and pay only the O(k) Vec copy. (2) The freeze retention estimate is conservative — sealing/freezing a k-property object mints up to k new shapes each storing k layout entries plus k offsets, ~2k^2 permanently ledger-charged entries, not ~k^2/2. Neither refinement weakens the finding.

**Suggestion:** Replace linear dedup with a transition edge map keyed by (parent ShapeId, PropertyKey, attributes) plus a hash of the layout for the update/remove cases; skip transition_after_update entirely when attributes are unchanged (compare before cloning); consider sweeping unreferenced shapes during GC since ObjectStructureSnapshot already tracks per-object shape.

### [HIGH] __proto__ is a name-string special case scattered across four layers instead of an accessor, with an uncatchable engine error on cycle

- **Kind:** `hack` · **Location:** `src/runtime/object/prototype.rs:25` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** prototype_get_in_chain: `if property.name() == PROTOTYPE_PROPERTY { return self.prototype_value(id)... }` on the miss path; ObjectHeap::set (heap.rs:375) and delete (heap.rs:385) repeat the string check; semantic_object.rs:208 repeats it again; static_names/mod.rs re-declares the constant (line 24) and string-compares it at six cache sites to keep cached writes/RMW/deletes away from it. set_prototype_value (prototype.rs:78) returns Error::runtime("prototype cycle is not allowed") — a non-catchable engine error — on the `obj.__proto__ = v` assignment path.

**Why it matters:** B.2.2.1 defines __proto__ as an accessor on Object.prototype; modeling it as name recognition in the physical layer makes it non-shadowable-by-design and wrong at the edges: Object.create(null).__proto__ returns Null via prototype_value instead of undefined (no inherited accessor exists), getOwnPropertyDescriptor(Object.prototype,'__proto__') cannot see it, and a cycle created via plain assignment (`a.__proto__ = b` where b inherits from a) aborts the script with an uncatchable Runtime error where the spec requires a catchable TypeError. Every new write/delete/cache path must remember the string guard or silently create an own '__proto__' data property.

**Verifier notes:** Every cited site checks out: prototype.rs:25/35 return prototype_value on any '__proto__' miss (so Object.create(null).__proto__ yields null, not undefined); heap.rs:375/385 and semantic_object.rs:208 repeat the string check; static_names/mod.rs:24 privately re-declares the constant already exported at object/mod.rs:53 and guards exactly six cache sites (268, 327, 368, 427, 577, 599), with two more in static_names/read.rs and a fifth uncounted layer in native/function/direct.rs:19/72/158. No __proto__ accessor is ever installed on Object.prototype (grep shows only constant declarations), so getOwnPropertyDescriptor cannot see it and delete cannot disable it (heap.rs:385-393 returns true without removing the magic). The uncatchable-cycle claim is verified end-to-end: plain assignment routes through the legacy path (assignment.rs:642) to ObjectHeap::set -> set_prototype_value -> Error::runtime (prototype.rs:78), and runtime_exception_value (control/assertions.rs:9-24) returns None for Error::Runtime so bytecode_error_completion (execution.rs:459-463) re-raises it as Err instead of Completion::Throw, which try/catch (try_catch.rs) can never catch — while the non-extensible branch of the same function and Object.setPrototypeOf both produce catchable TypeErrors, proving the divergence is accidental. The only counterpoint — Annex B being normative-optional — does not rescue it, since omitting __proto__ would require plain data-property behavior, not name-magic with wrong edges. Severity high is fair: a two-line pure-JS script aborts the VM uncatchably, and every future write/delete/cache path must remember the string guard.

**Verifier correction:** Claim verified in full; if anything it undercounts the scatter — native/function/direct.rs:19/72/158 is a fifth layer with its own private re-declaration of the "__proto__" constant, and bytecode/ops/assignment.rs:24 declares it a fourth time as LEGACY_PROTO_PROPERTY. The catchability split is directly demonstrable in-code: set_prototype_value's non-extensible branch (prototype.rs:86) throws catchable Error::type_error while its cycle branch (prototype.rs:78) throws uncatchable Error::runtime, and Object.setPrototypeOf on the identical cycle condition takes try_set_prototype_value and throws a catchable TypeError (object_static.rs:273-277).

**Suggestion:** Install __proto__ as a real accessor pair on %Object.prototype% through the existing native-accessor machinery (as done for species getters in AS-09q), delete all six string guards, and route the setter through semantic_try_set_prototype so cycle failure becomes TypeError.

### [HIGH] for-in fails to let non-enumerable own properties shadow enumerable prototype properties

- **Kind:** `risk` · **Location:** `src/runtime/object/property/keys.rs:63` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed (corrected severity: medium)

**Evidence:** collect_keys calls object.extend_enumerable_keys(...) at each prototype level, which only appends keys whose property is_enumerable(); push_unique_key dedups only against already-yielded keys. The semantic duplicate (semantic_object/keys.rs:19, semantic_enumerable_property_keys) likewise collects only semantic_own_enumerable_string_keys per level. Neither records non-enumerable own keys as visited.

**Why it matters:** EnumerateObjectProperties requires every own property key to be added to `visited` regardless of enumerability, so a non-enumerable own 'x' must suppress an enumerable 'x' on the prototype. Here `var o = Object.create({a:1}, {a:{value:2, enumerable:false}}); for (var k in o)` yields 'a' (from the prototype) when it must yield nothing. Both enumeration implementations share the divergence, so no path is correct.

**Verifier notes:** At src/runtime/object/property/keys.rs:50-70, collect_keys's `visited` holds only ObjectIds for cycle detection; each prototype level appends only is_enumerable() keys via extend_enumerable_keys (which even early-returns at line 86 if the object has no enumerable own keys), and push_unique_key dedups solely against already-yielded keys — so non-enumerable own keys never suppress same-named enumerable prototype keys. The semantic duplicate (src/runtime/semantic_object/keys.rs:12-35, used for proxies via runtime/mod.rs:576) has the identical flaw: it dedups with keys.contains() against only previously collected enumerable keys. The for-in consumer (src/runtime/bytecode/control/for_in.rs:45) snapshots this list with no per-iteration re-validation, and non-enumerable properties are fully creatable via Object.defineProperty/Object.create, so the cited repro genuinely yields 'a' where the spec's EnumerateObjectProperties (which adds every own key to visited regardless of enumerability) yields nothing.

**Verifier correction:** Defect is fully real in both enumeration paths, but the trigger is narrow: it requires an enumerable prototype property shadowed by a same-named non-enumerable own property, a configuration only reachable via defineProperty/Object.create descriptors (built-in prototype properties are already non-enumerable, so ordinary objects are unaffected). The consequence is an extra key in for-in output — no crash, unsafety, or data corruption — so medium (spec-conformance bug, test262-visible) is fairer than high.

**Suggestion:** Track a separate visited set of all own keys (not just yielded keys) at each level in both collect_keys and semantic_enumerable_property_keys — or better, collapse to one owner first (see the duplicate-enumeration finding).

### [MEDIUM] Packed-to-holey conversion has no density heuristic and holey slack is invisible to storage accounting

- **Kind:** `risk` · **Location:** `src/runtime/object/array/storage.rs:482` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** insert_dense_property: writing at any position < max_properties beyond the packed length converts to Holey and `holey.resize_with(position+1, || None)`. ArrayIndex::dense_position (index.rs:55) treats every index below max_properties as dense. property_count counts only Some entries, and Object::storage_payload_bytes (accounting.rs:73-93) counts only regexp text and byte buffers — array element storage is never charged as payload bytes, and post-creation growth is only ever recomputed at GC sweep.

**Why it matters:** `let a = []; a[max_object_properties - 1] = 1;` allocates a Vec of max_properties Option<ObjectProperty> slots (tens of MB at default-scale limits) while the ledger records exactly 1 ObjectProperty and zero payload bytes. For an engine whose whole safety story is checked, pre-committed resource accounting (AS-05b2), this is a hole where a two-element sparse array bypasses the memory limits the ledger exists to enforce.

**Verifier notes:** The code facts are all accurate: insert_dense_property (storage.rs:480-487) does a full-width Packed→Holey resize with no density heuristic; dense_position (index.rs:55-64) treats any position < max_properties as dense; property_count only counts Some slots; and storage_payload_bytes (accounting.rs:73-93) charges only regexp and byte-buffer bytes, never array-element backing store. So the accounting genuinely never meters the holey Vec's None-slot slack as bytes. However, the magnitude/severity framing is overstated and misses bounding structure. The holey Vec is hard-bounded by max_object_properties: dense_position returns None above the limit, routing those elements to sparse/named storage (each charged one record), so one array's dense Vec can never exceed max_properties slots. At the actual default (DEFAULT_MAX_OBJECT_PROPERTIES = 4096, limits.rs:13) with Option<ObjectProperty> on the order of tens of bytes, a single a[4095]=1 allocates ~4096 slots ≈ a few hundred KB, not the claimed "tens of MB at default-scale limits" (that needs max_properties in the millions). The logical property-record count is also pre-committed in real time via reserve_property_growth/commit (mod.rs:179-198, 559-561); what goes unmetered is backing-Vec bytes generally — the design meters property storage by record count, not backing-store bytes (named-property Vec capacity slack is equally unmetered), so this is not a holey-specific bypass. The genuine residual issue is a bounded record-vs-slot amplification that only bites an embedder who treats ObjectProperty count as a memory proxy while leaving byte limits unlimited (the default). That warrants low, not medium.

**Verifier correction:** Real but overstated. Confirmed: writing any index < max_properties beyond the packed prefix triggers a full-width Packed→Holey conversion (storage.rs:480-487) with no density heuristic, and the resulting None-slot slack is never charged as payload bytes (only regexp/byte-buffer bytes are, accounting.rs:73-93) while property_count records just the live entries. Corrections to the claim: (1) the dense Vec is hard-bounded by max_object_properties because dense_position (index.rs:55-64) sends indices >= max_properties to sparse/named storage, so a single array's backing Vec caps at ~max_properties slots — at the real default of 4096 (limits.rs:13) that is roughly a few hundred KB, not "tens of MB"; the "tens of MB at default-scale" magnitude is inaccurate. (2) It is not an unbounded bypass: total unmetered slack is bounded by max_objects x max_object_properties slots, the same limits the accounting already enforces. (3) The logical property-record count IS pre-committed in real time (reserve_property_growth -> reserve_count(ObjectProperty) -> commit, mod.rs:179-198); the design deliberately meters property storage as record counts, and backing-Vec bytes (including named-property Vec capacity slack) are simply not part of the byte model — so this is a general accounting characteristic, not a holey-array-specific hole. Practical residual risk: a record-vs-slot amplification (up to ~max_properties-fold) that only matters if an embedder relies on ObjectProperty count as a memory proxy while leaving byte limits at their unlimited default. A density heuristic before the packed-to-holey conversion would still be a reasonable hardening, but severity is low.

**Suggestion:** Route indices beyond a density threshold (e.g. position > 2*dense_len + 64, the usual JSC/V8-style heuristic) to the existing sparse named-property path instead of keying density solely on max_properties, and/or charge holey capacity to a payload-byte category at conversion time.

### [MEDIUM] A Proxy in the middle of a prototype chain is bypassed by for-in and key enumeration

- **Kind:** `risk` · **Location:** `src/runtime/object/property/keys.rs:50` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** Context::enumerable_keys (runtime/mod.rs:572-585) takes the semantic trap-aware path only when the receiver itself is a proxy; otherwise it calls ObjectHeap::keys -> collect_keys, which walks `object.prototype` physically and calls extend_enumerable_keys on each record — for a proxy record that means enumerating its (empty) physical properties and following its stored prototype field, never invoking ownKeys/getOwnPropertyDescriptor/getPrototypeOf traps.

**Why it matters:** AS-09ad deliberately made [[Get]]/[[HasProperty]] re-enter semantic dispatch on every prototype hop precisely so mid-chain proxies stay observable, but enumeration was left on the physical walker: `for (k in Object.create(new Proxy(target, handler)))` fires no traps and sees no keys. Two enumeration implementations with different proxy behavior is the exact 'parallel path' failure mode the AS program exists to eliminate.

**Verifier notes:** The cited code behaves exactly as claimed. Context::enumerable_keys (mod.rs:572-585) routes to the trap-aware semantic_enumerable_property_keys only when the top-level value is itself a proxy; otherwise it calls ObjectHeap::keys -> collect_keys (keys.rs:50-70), a pure ObjectHeap method with no Context access that physically walks object.prototype and calls extend_enumerable_keys per record. A proxy record is Object::ordinary() with enumerable_property_count==0 and physical prototype None, so has_enumerable_own_keys() (mod.rs:687) is false, extend_enumerable_keys early-returns, and the physical walk halts without ever invoking ownKeys/getOwnPropertyDescriptor/getPrototypeOf traps. This is the sole path for-in uses (for_in.rs:45). I verified the AS-09ad contrast is real: finish_ordinary_property_read/presence (semantic_object.rs:198-223, 289-307) re-enter semantic dispatch on every prototype hop and re-check is_proxy, so [[Get]]/[[HasProperty]] observe a mid-chain proxy while enumeration does not — a genuine parallel-path divergence. A proxy prototype is reachable via Object.create/setPrototypeOf (prototype.rs:113 stores the proxy ObjectId in the physical prototype field). Medium severity is fair: an observable spec-conformance divergence confined to the niche case of a proxy sitting mid prototype-chain during for-in, not memory-unsafe or crashing.

**Suggestion:** Make collect_keys dispatch each prototype hop through semantic_get_prototype / semantic_own_enumerable_string_keys (mirroring the AS-09ad ordinary-finish recursion), then delete one of the two enumeration implementations.

### [MEDIUM] Single global prototype_lookup_version epoch: any structural mutation anywhere flushes every property cache in the VM

- **Kind:** `microopt-debt` · **Location:** `src/runtime/object/heap.rs:449` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** bump_if_structure_changed bumps the one VM-global PrototypeLookupVersion whenever any object's shape/property-count/prototype/extensibility changes; PropertyLookupGuard::is_valid (lookup.rs:33-37) compares that global counter. Bump sites include every property add/delete on any object, every Array#shift/unshift even on the fast path (array/front.rs:28,63), and ObjectHeap::define_property bumps unconditionally (property/descriptor.rs:613) even when bump_if_structure_changed on the next line would have detected no change.

**Why it matters:** The guard has no per-object or per-chain dependency, so a workload that constructs objects with properties in a loop (every fresh object's first property add is a 'structure change') or uses arrays as queues invalidates all static-name ICs continuously — the cache tier is only effective in steady-state code that never allocates shaped objects. It is safe but shaped exactly backwards from real workloads, and the unconditional define_property bump is a belt-and-suspenders band-aid on top of the snapshot comparison.

**Verifier notes:** The code matches the claim: `PrototypeLookupVersion` is a single u64 on `ObjectHeap` (src/runtime/object/base.rs:56), `bump_if_structure_changed` (heap.rs:449-458) bumps it on any object's shape/property-count/prototype/extensibility change, and `PropertyLookupGuard::is_valid` (lookup.rs:33-37) requires exact equality with that global counter, so one bump anywhere invalidates every static-name IC slot VM-wide (static_names/cache.rs:18). Bump sites confirmed: every shape-changing set (heap.rs:378-382 — including a fresh object's first property add), every delete (heap.rs:395-399), the dense fast paths of shift/unshift (front.rs:28,63), seal/freeze unconditionally (integrity.rs:15,21), GC sweep (base.rs:119), and define_property bumps unconditionally at descriptor.rs:613 immediately before the conditional bump at 614 — a literal double bump when structure changed, and a global flush even for no-op redefinitions, since attribute changes already transition the shape (shape.rs:148-172), new dense elements change property_count (mod.rs:712-716), and accessor-ness is re-checked at read/write time. The workload reasoning holds: object-allocation loops and array-queue code bump the epoch continuously so monomorphic ICs on stable receivers never survive, each miss forcing a full prototype-chain re-walk (lookup.rs:223-242). Correctness is unaffected (caches fail open to the slow path), so medium severity for microopt-debt is fair.

**Verifier correction:** Two minor precision notes that do not change the finding. (1) The guard is not literally without per-object dependency: is_valid also checks the receiver's shape (lookup.rs:35). But because the global-version check is ANDed in, that per-object component cannot save any cache from an unrelated bump, so the flush-everything conclusion stands; what is genuinely missing is per-chain dependency tracking. (2) The unconditional bumps in array shift/unshift fast paths (front.rs:28,63) are load-bearing under the current design, not gratuitous: dense element moves change lookup results (e.g. a cached Missing at a holey front index becomes present after unshift) without any shape or snapshot change, so the global bump is the only invalidation available — they are evidence of the design's cost, as the claim argues, not an independently removable redundancy. The define_property bump at descriptor.rs:613, by contrast, does appear removable: attribute updates transition the shape via transition_after_update (shape.rs:148-172, attributes include writable/enumerable/configurable and setter presence per descriptor.rs:434-447), dense-element additions change property_count in the snapshot, value writes read through slots, and accessor conversions are rejected at cache-read/write time, with native-call caches guarded by a per-property version bumped in define (descriptor.rs:581) — though removing it warrants a careful audit of accessor payload replacement paths.

**Suggestion:** Scope invalidation: bump only when the mutated object can be on some cached chain (e.g. per-object or per-prototype version stamps, or exempt fresh objects that no guard has ever pinned), and remove the redundant unconditional bump in define_property.

### [MEDIUM] Two parallel OrdinarySet implementations: descriptor-recursive reflect path vs disposition-walk path whose physical layer cannot report failure

- **Kind:** `inelegant` · **Location:** `src/runtime/object/property/descriptor.rs:449` · **Status in project docs:** documented (AS-plan)
- **Verification verdict:** confirmed

**Evidence:** The abstract Set (property_call.rs:75) uses semantic_reflect_property_write (mutation.rs:207): descriptor-driven recursion returning bool, with its own 'length' interception at mutation.rs:295. The bytecode assignment path uses semantic_property_write -> finish_semantic_property_write -> write_ordinary_object_property_with_accessors (accessor.rs:48): a separate accessor_write_target chain walk (prototype.rs:140), a second 'length' interception (accessor.rs:79), then physical Object::set whose ObjectProperty::set_value silently no-ops on non-writable (descriptor.rs:449-458) and whose non-extensible branch returns Ok(()) (object/mod.rs:488-490); finish_semantic_property_write then reports Ok(true) unconditionally.

**Why it matters:** OrdinarySet semantics exist twice with different mechanics that must be kept observably identical by hand: one signals failure by return value, the other by silent swallowing at three different depths (set_value, non-extensible early return, NoSetter disposition). Any new exotic write rule must be added to both, and the sloppy path structurally cannot distinguish 'wrote' from 'rejected', which is why the strict-mode work (AS-09aa) had to route around it entirely.

**Verifier notes:** Every citation checks out: OrdinarySet exists twice — the descriptor-recursive reflect path (mutation.rs:207, length interception at 295, used by Set at property_call.rs:75) and the disposition-walk path (finish_semantic_property_write mutation.rs:120-139 → write_ordinary_object_property_with_accessors accessor.rs:48, chain walk prototype.rs:140, second length interception accessor.rs:79) whose physical layer swallows failure at exactly the three cited depths (set_value descriptor.rs:449-458 no-ops on non-writable, non-extensible returns Ok(()) at object/mod.rs:488-490, NoSetter returns Ok(()) at accessor.rs:61) and whose ObjectTail arm returns Ok(true) unconditionally (mutation.rs:136). The routed-around claim is corroborated by the fork in assignment.rs:180/209+640: strict writes and sloppy plain-object writes go through the reflect path with SetFailureBehavior, leaving the failure-blind path live only for sloppy arrays, __proto__, sloppy non-Object receivers, and builtin callers. The hand-sync hazard is real, not hypothetical: exotic rules (length, typed arrays) are duplicated in both paths, and they already diverge mechanically — non-writable array length throws via set_array_length (array/length.rs:16-18) on the disposition path but returns false at mutation.rs:289-291 on the reflect path.

**Verifier correction:** One overstatement to temper: "the bytecode assignment path uses semantic_property_write → finish_semantic_property_write" describes only the legacy fork — since the strict-mode work, the default bytecode assignment (all strict writes plus sloppy plain-object writes) already routes through the reflect path (assignment.rs:180-192/209-230); the disposition-walk path survives for sloppy array targets, __proto__ writes, sloppy non-Object receivers, and internal builtin callers (json.rs:278, proxy.rs:119, object_prototype.rs:150, array/generic.rs:491). Also, the swallowing is documented intent for sloppy-mode semantics (accessor.rs:21-24) and the [[Set]] boolean is unobservable in that domain, so this is a maintenance/duplication defect, not a live correctness bug — medium severity for kind=inelegant is fair.

**Suggestion:** Make the reflect-style recursion the single [[Set]] owner and reduce accessor_write_target/Object::set to storage primitives that return success booleans; the cache tail contract already isolates callers from this change.

### [MEDIUM] Array 'length' semantics scattered as string comparisons across five modules with an uncatchable engine-error backstop

- **Kind:** `hack` · **Location:** `src/runtime/object/mod.rs:384` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** Object::set: `if self.array_length.is_some() && property_name == ARRAY_LENGTH_PROPERTY { return Err(Error::runtime("array length assignment is not supported")) }`. The actual assignment semantics live in accessor.rs:79 (ordinary writes), mutation.rs:295 (reflect writes), semantic_object/descriptor.rs:74 (defineProperty), and array/length.rs (shrink/rollback owner); object/mod.rs separately name-checks 'length' in get_own/has_own/delete; the ARRAY_LENGTH_PROPERTY constant is independently re-declared in at least four files.

**Why it matters:** Array-exotic length is intercepted by property-name string equality at every semantic entry point rather than owned once; any path that forgets the check falls through to a non-catchable Runtime error rather than a spec behavior, i.e. the backstop converts a missed case into an engine abort instead of a wrong-but-catchable result. Same pattern (and same risk) as the __proto__ scattering.

## Additional low-severity findings (not separately verified)

### [LOW] Uint8Clamped conversion round-trips an f64 through decimal string formatting to become a u8

- **Kind:** `hack` · **Location:** `src/runtime/object/typed_array.rs:790` · **Status in project docs:** UNDOCUMENTED

**Evidence:** to_uint8_clamp: after correct ties-to-even rounding, `rounded.to_string().parse::<u8>().unwrap_or(0)` converts the in-range f64 to u8 by formatting it as decimal text and parsing it back, silently mapping any parse failure to 0.

**Why it matters:** A numeric clamp implemented via string formatting is lint-evasion shaped (the project bans `as` truncation casts, but the sibling to_float32 shows #[allow] is the accepted tool); it allocates on every Uint8ClampedArray store, and the unwrap_or(0) is a silent failure default hiding any future edge case rather than surfacing it.

**Suggestion:** `u8::try_from(rounded as u32)` under the same targeted allow used by to_float32, or integer-domain rounding — value is already proven within [0,255].

## Optimization opportunities (post-cleanup)

### [MEDIUM] Plain-string property access transcodes the entire string to UTF-16 per access

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/property/mod.rs:181` · **Status in project docs:** UNDOCUMENTED

**Evidence:** string_property/string_has_property/string_enumerable_keys/string_property_value/string_length_value_if_string all start with `value.encode_utf16().collect::<Vec<_>>()` for a Value::String receiver — even for `.length` or a single index; string_property_index additionally allocates `index.to_string()` to verify canonicality.

**Why it matters:** `s[i]` in a loop over a UTF-8 Value::String is O(len) allocation+transcode per iteration, i.e. O(n^2) for a scan that should be O(n); `s.length` on a 1MB string allocates 2MB to return a number. The dual String/HeapString representation forces every virtual-property consumer to normalize eagerly instead of lazily.

**Suggestion:** Answer `.length` and index containment from UTF-8 directly when the string is ASCII (the common case, checkable in O(1) amortized with a cached flag), or convert Value::String receivers to HeapString once at the property boundary and cache it.

### [MEDIUM] Canonical numeric-index recognition allocates and double-parses on every property access; typed-array check runs its string parse before the payload check

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/object/typed_array.rs:658` · **Status in project docs:** UNDOCUMENTED

**Evidence:** ObjectHeap::typed_array_property_index first evaluates `typed_array_property_index(property, usize::MAX)` — parse::<f64>, then `format_ecmascript_number(number) != property` (allocates a String), then parse::<usize> — BEFORE checking whether the object has a typed_array payload; semantic_property_read (semantic_object.rs:144) runs this for every non-proxy object read. ArrayIndex::parse (array/index.rs:45) similarly does `value.to_string() != property` — one String allocation per successful parse — and is called on every get/set/delete/has/define of any array or indexed name.

**Why it matters:** The hottest paths in a JS engine (indexed element access) each pay one or two heap allocations purely to verify that a digit string is canonical, and the typed-array pre-dispatch pays the full parse/format cost for plain objects and arrays that can never match. This is a correctness check implemented at the wrong cost tier and it taxes the generic path that all fast paths fall back to.

**Suggestion:** Replace both canonical checks with an allocation-free byte scan (nonempty, no leading zero unless '0', all digits, fits limit — the standard CanonicalNumericIndexString fast test), and reorder ObjectHeap::typed_array_property_index to check `self.object(id)?.typed_array.is_some()` first.

### [MEDIUM] Monomorphic one-entry ICs that re-walk the full chain on refill, and cache classification embedded in the generic read path

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/object/prototype.rs:22` · **Status in project docs:** UNDOCUMENTED

**Evidence:** prototype_get_in_chain calls cacheable_property_value first — building a guard and walking the chain classifying cacheability — and on Uncacheable walks the chain a second time via prototype_property_value_in_chain; array receivers are always Uncacheable (has_uncacheable_own_property, lookup.rs:544), so every generic array element get does the classification walk then the real walk. Each StaticPropertyAccessId has exactly one Cell<Option<CacheablePropertyLookup>> (cache.rs:18); a polymorphic site overwrites it every observation, and every miss reruns cacheable_property_lookup over the whole prototype chain.

**Why it matters:** The generic semantic path is defined *through* the cache machinery rather than the cache layering over it — receivers that can never cache still pay guard construction and classification on every access, and 2-shape polymorphic sites (extremely common) get worst-of-both: perpetual full-chain lookups plus cache-write traffic.

**Suggestion:** Once the shape table has transition-based identity: (a) give ordinary get/has a direct walk that does not consult cacheability, (b) widen per-site caches to 2-4 entries keyed by receiver shape, (c) let arrays participate by caching the 'not an own index, hit on prototype at depth d' case which the current uncacheable-if-any-element rule forbids.
