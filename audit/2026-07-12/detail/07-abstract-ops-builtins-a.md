# Abstract operations + core builtins (Array/String/Object/JSON/Math/Number)

> Детальный отчёт аудита архитектуры rs-quickjs. Подсистема: `abstract-ops-builtins-a`. Сгенерировано многоагентным аудитом 2026-07-12; каждая находка уровня medium/high прошла адверсариальную проверку отдельным агентом. Материал на английском (исходный текст аудита).

## Subsystem assessment

The subsystem is far better than a typical prototype: AS-02/AS-03 produced a real semantic facade (Get/Set/Call/GetMethod/ToPrimitive/ToNumber/ToString/equality/iterators each have one nominal owner in runtime/abstract_operations), and most builtins visibly route through it with disciplined resource accounting. The claim of single ownership holds for conversions and equality, holds mostly for iteration, and breaks down at two fault lines. First, the value model: strings exist in two representations (UTF-8 Value::String vs UTF-16 HeapString with an eager lossy UTF-8 shadow copy), functions are four separate Value variants that don't participate in object semantics, and primitive wrapper objects don't exist — so builtins choose per-method which semantics they get, yielding lone-surrogate corruption, padStart counting the wrong units, uncatchable Error::runtime failures for legal JS (Object.create with a function prototype, push.call on a primitive), and combinatorial match-arm duplication. Second, the array subsystem has three parallel implementations per method (generic builtin, ObjectHeap storage method, packed specialization), and the guard discipline is asymmetric: bytecode fast paths check for accessors in the chain while the builtin storage paths silently read accessors as undefined, the for-of array fast path doesn't guard against a patched %ArrayIteratorPrototype%.next, and the sort comparator bytecode-sniffer escapes the AS-08 kill switch entirely and can feed std::sort a non-total comparator (panic risk). The regexp-adjacent string methods (match/search/replace) predate the facade: they treat string patterns as literals, skip GetSubstitution in two of four replace paths, and bypass the @@match/@@replace protocol that split/replaceAll correctly implement. JSON leans on serde_json for a grammar that is not ECMAScript's. The stabilization plan documents the value-model variants, the duplicate-operation allowlist, and the sort/callback recognizers as design, but the specific divergences above (accessor guards, kill-switch gap, literal-pattern match, serde dialect, lossy UTF-8 consumers) are undocumented and are where the foundation actually leaks.

## Strengths

- Abstract operations are genuinely centralized for the common cases: builtins consistently call self.to_string/to_number/to_integer_or_infinity/to_length rather than hand-rolling coercions, and equality (strict/SameValue/SameValueZero) has exactly one bit-level-correct owner (equality.rs) that both storage search paths and builtins import.
- IteratorClose semantics are carefully right: original-throw precedence over return-lookup failures, resource-limit errors deliberately excluded from swallowing (iterator.rs:525-611), and yield* delegation handles return/throw/close cases with explicit done-state transitions.
- Number formatting rests on a sound foundation: shortest round-trip digits are delegated to Rust's proven float formatter, ECMAScript layout rules are implemented separately (value/kind.rs), and toFixed/toPrecision derive exact decimal digits via full-precision expansion with correct round-half-up — no dtoa hacks in the core path.
- Math.sumPrecise is a real fixed-width (33x64-bit) exact accumulator with proper signed-zero and infinity-flag semantics, not a naive compensated sum.
- Reflect.* goes almost entirely through the semantic facade (semantic_property_read_with_receiver, semantic_construct, semantic_own_property_keys) — a good template for what the rest of the builtins should look like.
- Resource accounting (self.step(), check_string_len, checked arithmetic everywhere) is threaded through essentially every generic loop, and the no-unwrap/no-indexing discipline is genuinely upheld across ~13k lines reviewed.
- The generic array paths implement spec ordering faithfully (HasProperty/Get/Set/Delete sequences for shift/unshift/splice/copyWithin, species construction, isConcatSpreadable), and the stable merge sort propagates abrupt comparator completions correctly.
- AS-08's optional_optimizations_enabled kill switch plus pure-bytecode-expression classification (callback_fast_path.rs) is a principled specialization design — where it is actually applied.

### [HIGH] String.prototype.match/search treat string patterns as literal substrings; replace skips GetSubstitution

- **Kind:** `hack` · **Location:** `src/runtime/native/builtins/string_regexp.rs:195` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** string_match_plain: `if !needle.is_empty() && !text.contains(needle) { return Ok(Value::Null) }`; search: `text.find(&self.string_argument_text(pattern)?)` (line 81); replace string path calls replace_first_plain which does `output.push_str(replacement)` with no $-processing (lines 111, 546-554), and string_regexp_replace_global also pushes the replacement literally (line 230) while a parallel `regexp_substitution` (line 306) implements $&/$`/$'/​$n only for the *_value variants.

**Why it matters:** Spec requires match/search to compile a string argument into a RegExp ("axb".match("a.b") matches; "abc".search(".") is 0) and requires GetSubstitution for every replace path ("aa".replace(/a/g, "$&!") must expand $&). This file contains two replace implementations that have already drifted: one with substitution, one without. Silent wrong answers on extremely common code.

**Verifier notes:** Every cited behavior is real and unguarded. (1) match: src/runtime/native/builtins/string_regexp.rs:42-45 routes non-regexp patterns to string_match_plain (line 195), which does a literal `text.contains(needle)` check instead of RegExpCreate, and even on success returns a bare [needle] array with no index/input properties. (2) search: lines 79-83 do a literal `text.find(...)` — and `find` returns a Rust byte offset, not a UTF-16 code-unit index, so even literal matches report wrong positions after non-ASCII prefixes. (3) replace: the string-pattern path (lines 109-114 → replace_first_plain, lines 546-554) and both regexp-with-string-replacement paths (string_regexp_replace_first line 211, string_regexp_replace_global line 230) push the replacement literally, while the parallel *_value variants (lines 242-282) implement full $-substitution via regexp_substitution (line 306) but are reachable only through RegExp.prototype[Symbol.replace] (regexp.rs:224-241) — eval_string_prototype_replace never dispatches to @@replace, so `"aa".replace(/a/g, "$&!")` yields "$&!$&!". Worse than claimed: eval_string_prototype_replace unconditionally stringifies args[1] (lines 101-105), so a function replacement passed to String.prototype.replace is converted to its source text instead of being called; and replaceAll in string_modern.rs (line 157, plain_replacement_substitution lines 430-473, pinned by tests/modern_string_apis_smoke.rs:10) DOES perform $&/$`/$' substitution, confirming a third drifted parallel implementation. Both dispatch tiers in function/direct/string.rs (lines 55-72, 153-170) route directly into these functions with no guard, and no test documents the literal behavior as intentional.

**Verifier correction:** Claim confirmed as written; scope is actually broader: (a) String.prototype.replace also stringifies function replacements (string_regexp.rs:101-105), inserting function source text instead of invoking the callback; (b) search returns byte offsets rather than UTF-16 code-unit indexes for string patterns; (c) match's plain result also omits the index/input/groups properties; (d) there are three drifted replacement implementations — regexp_substitution (with $-processing, used only by Symbol.replace), the literal push_str paths used by String.prototype.replace, and plain_replacement_substitution in string_modern.rs used by replaceAll (which does substitute, so replace and replaceAll disagree on identical inputs).

**Suggestion:** Route String.prototype.match/search/replace through the spec protocol (@@match/@@search/@@replace via GetMethod) with a single GetSubstitution owner; delete the literal-substring shortcuts.

### [HIGH] Packed numeric sort fast path feeds a non-total comparator to std sort_by (panic risk with NaN)

- **Kind:** `risk` · **Location:** `src/runtime/object/array/storage.rs:237` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** `elements.sort_by(|left, right| { ... Self::numeric_sort_ordering(left, right, descending) })` where numeric_sort_ordering returns Ordering::Equal whenever left-right is NaN. An array [3, NaN, 1] is all-Number so it takes this path; Equal(NaN,3) + Equal(NaN,1) + Less(1,3) violates transitivity. builtins/array/sort.rs:349 duplicates the same ordering function for toSorted.

**Why it matters:** Since Rust 1.81 the std sort implementations detect Ord violations and panic ("comparison function does not implement a total order"). The project bans panics; a script `[1, NaN, 0].sort((a,b)=>a-b)` could abort the process. The generic merge sort handles NaN deterministically, so the fast path can also produce a different element order than the disabled-optimization path.

**Verifier notes:** storage.rs:237-245 feeds std sort_by a comparator (numeric_sort_ordering, storage.rs:259-273) that returns Equal whenever left-right is NaN, and the gates (default_number_property_value, packed_numeric_sort_values) admit Value::Number(NaN), so a packed all-number array containing NaN sorted with a bytecode-recognized (a,b)=>a-b comparator (sort.rs:187-219) reaches this non-total comparator; the duplicate at sort.rs:349 feeds sort_by at sort.rs:174 for the toSorted fast path. The comparator breaks transitivity (cmp(NaN,x)=Equal for all x, cmp(1,3)=Less), the project is edition 2024 (toolchain >= 1.85, post-1.81 driftsort/ipnsort which document may-panic on non-total comparators), there is no catch_unwind anywhere, and Cargo.toml has the panic=deny lint plus CLAUDE.md's explicit no-panic policy. The typed-array sort at typed_array/methods.rs:743-753 already implements a correct NaN-total comparator, confirming this site simply missed the established pattern; the fast-path/slow-path order divergence is also real, though spec-permitted since ECMA-262 leaves order implementation-defined for inconsistent comparators.

**Verifier correction:** Core defect confirmed, with two corrections to the evidence: (1) the cited reproducer [1, NaN, 0].sort((a,b)=>a-b) will NOT panic — 3-element slices go through insertion sort, which never triggers std's best-effort Ord-violation detection; the panic risk applies to larger packed numeric arrays (roughly >20 elements) containing NaN, where detection is data-dependent but the input is script-controllable, so an adversary can shape an array to trip it. (2) "abort the process" holds only if the embedder builds with panic=abort; by default the panic unwinds through the engine, which still violates the project's Result-only/no-panic contract (Cargo.toml panic=deny lint, CLAUDE.md:138). Fix is trivial: reuse the NaN-total ordering pattern already present in typed_array_numeric_order (src/runtime/native/builtins/typed_array/methods.rs:743) or f64::total_cmp-style handling in both storage.rs:259 and builtins/array/sort.rs:349. The fast/slow-path order divergence is real but spec-permitted (implementation-defined order for inconsistent comparators), so it is a consistency wart, not an independent spec violation.

**Suggestion:** Sort with a total order (e.g. f64::total_cmp after mapping NaN to a fixed rank, or reuse the engine's own merge sort on the packed buffer) and add a NaN-containing array to the fast-path equivalence tests.

### [HIGH] Array.prototype.sort comparator bytecode-sniffing is not gated by the AS-08 optimization kill switch

- **Kind:** `microopt-debt` · **Location:** `src/runtime/native/builtins/array/sort.rs:187` · **Status in project docs:** documented (AS-plan)
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** compile_numeric_sort_comparator pattern-matches the callee's bytecode ([LoadBinding, LoadBinding, NumberBinary(Sub), Complete(Return)]) and replaces the whole observable sort with objects.sort_packed_default_numeric_array_if_array. Every other specialization checks `self.optional_optimizations_enabled()` (callback_fast_path.rs:18, bytecode/array.rs:20/37/53...); neither eval_packed_numeric_array_sort nor the storage sort does.

**Why it matters:** AS-08's 'disabled-mode equivalence' audit is the project's safety argument for specialization, and this path escapes it: the comparator is never called (zero comparator invocations vs n·log n in the generic path), semantics are re-implemented a third time (builtins merge sort, storage sort_by, plus numeric_sort_ordering duplicated in two files), and the one mechanism designed to prove equivalence cannot exercise it.

**Verifier notes:** The factual anchor is accurate: compile_numeric_sort_comparator (src/runtime/native/builtins/array/sort.rs:187) sniffs the comparator's bytecode, eval_packed_numeric_array_sort replaces the sort without ever invoking the comparator, no optional_optimizations_enabled() check exists on this path, and numeric_sort_ordering is duplicated in native/builtins/array/sort.rs:349 and object/array/storage.rs:259. But the 'why bad' premise is contradicted by a documented invariant the auditor missed: AS-08b explicitly audited and retained dense-array/native-builtin packed specializations as 'operation-shaped rather than source-shaped' (docs/semantic-architecture-inventory.md:597, 616-627 names the numeric sort a-b/b-a recognizer specifically; docs/architecture-stabilization-plan.md AS-08b evidence lists 'general pure callback expression plans' as an accepted guard category), with the documented safety argument for this category being semantic guards plus ordinary-fallback-on-guard-miss, not the disabled-mode gate. The claim 'every other specialization checks the flag' is also false: nothing under src/runtime/native/builtins/ checks it — flatten.rs (which likewise compiles callback bytecode for flatMap), callbacks.rs, and copy.rs packed paths are all ungated by the same design; the gated sites cited belong to the separately classified cache/fast-path/bytecode-shortcut categories.

**Verifier correction:** The absence of the AS-08 mode gate on the numeric-sort comparator recognizer is a deliberate, documented AS-08b classification (operation-shaped packed algorithm with semantic guards and an ordinary merge-sort fallback on guard miss), shared by the whole native-builtin packed family, not an un-audited escape. Residual debt is real but minor: numeric_sort_ordering is duplicated across native/builtins/array/sort.rs and object/array/storage.rs and could be unified, and one could argue comparator-bytecode recognition sits closer to the gated 'function fast path' category than to storage-shape guards — a classification worth revisiting, not a high-severity defect.

**Suggestion:** Gate both packed-sort entry points on optional_optimizations_enabled(), share one numeric_sort_ordering, and add the sort path to the AS-08 disabled-mode differential suite. (The a-b/b-a recognizer itself is documented in docs/semantic-architecture-inventory.md; the missing gate is not.)

### [HIGH] Builtin array storage fast paths silently misread accessor elements (guard the bytecode paths have, builtins lack)

- **Kind:** `microopt-debt` · **Location:** `src/runtime/object/array/mod.rs:337` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** array_property_value_by_index / get_own_array_index resolve elements via ObjectProperty::value(), which returns Value::Undefined for accessor payloads (property/descriptor.rs:340-345); packed_array_index_of uses data_value_ref() which is None for accessors (search.rs:72-80); packed_array_join treats accessors as Undefined (mod.rs:287-301). By contrast the bytecode-facing entry points explicitly bail: array_index_has_accessor_in_chain (array/bytecode.rs:10-27, used at :55 and :91).

**Why it matters:** `const a=[1,2,3]; Object.defineProperty(a,1,{get:()=>42}); a.indexOf(42)` returns -1 without calling the getter; pop/reverse/join/includes/lastIndexOf on such arrays return wrong values and skip observable getter calls. This contradicts the inventory's claim that storage helpers 'prove ... descriptor ... conditions before a bulk operation'. Two guard regimes for the same optimization is exactly how fast paths rot.

**Verifier notes:** The claim is accurate at every cited location. Accessor properties genuinely live in packed dense array storage (Object::define_array_property at src/runtime/object/property/descriptor.rs:732-793 converts/inserts accessor payloads in place, and packed_array_properties at storage.rs:75-86 checks only sparse-keys/Packed/length, never accessors), while the builtin entry points (native/builtins/array/mod.rs:182-341) gate only on array_len_if_array. So packed_array_index_of/includes/last_index_of skip accessor slots via data_value_ref()==None, packed_array_join renders them as Undefined, pop returns property.value()==Undefined, and array_property_value_by_index misreads accessors on receiver and prototype chain — the repro (indexOf returning -1 without calling the getter) holds exactly, in contrast to the correctly guarded bytecode paths (array_index_has_accessor_in_chain, bytecode.rs:10-27). The only inaccuracy is minor: the in-place bulk fast paths (reverse/shift/unshift/sort) do bail on accessors via has_default_array_attributes (descriptor.rs:412-421), but their element-wise fallbacks still misread accessors as Undefined and set_value silently no-ops on accessor slots, so reverse is still wrong via the fallback path — i.e., there are three inconsistent guard regimes, not two, which reinforces rather than weakens the architectural point.

**Verifier correction:** Confirmed as stated, with one refinement: reverse's dense bulk path is actually guarded (has_default_array_attributes returns false for accessors), but its element-wise fallback (reverse_array_pair -> array_property_value_by_index -> ObjectProperty::value) still reads accessors as Undefined without invoking getters, and ObjectProperty::set_value silently no-ops on accessor slots. Also additional unguarded surface beyond the claim: packed_array_values_if_array (array/mod.rs:366-374, used by flatten/callback fast paths) maps accessors to Undefined too, and the non-configurable-accessor pop fallback ignores the failed delete where the spec requires TypeError. Severity high is fair: silent spec-violating results and skipped observable getter calls across many core Array builtins, directly contradicting docs/project-plan.md:576's claim that descriptor-sensitive inputs keep the generic fallback, and inconsistent with the correctly guarded bytecode paths.

**Suggestion:** Make packed/holey eligibility require has_default_array_attributes (as the sort path already does via packed_default_array_values_if_array) or reuse array_index_has_accessor_in_chain before every storage-level element read in the builtin paths.

### [HIGH] JSON.parse and QuoteJSONString delegated to serde_json with mismatched semantics

- **Kind:** `hack` · **Location:** `src/runtime/native/builtins/json.rs:118` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: medium)

**Evidence:** `serde_json::from_str(&text)` for the whole parse; numbers use `value.as_f64().ok_or(runtime error)` (lines 170-173); stringify_json_string is `serde_json::to_string(value)` over a &str (line 646) after HeapString::as_str() has already replaced lone surrogates with U+FFFD (storage/string_heap.rs:49-53).

**Why it matters:** serde_json enforces a 128-level recursion limit, rejects out-of-range numbers (JSON.parse('1e999') must be Infinity), and rejects lone-surrogate \uD800 escapes that JSON.parse must accept; JSON.stringify('\uD800') must emit "\\ud800" (well-formed JSON.stringify) but emits U+FFFD. The JSON grammar is borrowed from a library whose dialect differs from ECMA-404-as-ECMAScript-sees-it, and the string type cannot round-trip the values anyway.

**Verifier notes:** Every cited mechanism checks out: json.rs:118 parses via serde_json::from_str with only the preserve_order feature (Cargo.toml:33), so the default 128-level recursion limit applies, 1e999 is rejected (NumberOutOfRange → SyntaxError instead of Infinity), and lone-surrogate \uD800 escapes are rejected instead of accepted; json.rs:645-650 quotes strings via serde_json::to_string over HeapString::as_str(), which is documented (string_heap.rs:49) to replace lone surrogates with U+FFFD, so JSON.stringify('\uD800') emits "\u{FFFD}" instead of the ES2019-required "\ud800", and json_parse_text (json_meta.rs:38-46) applies the same lossy conversion to parse input. No guard, pre-processing, or documented-deviation note exists. However, the claim's assertion that 'the string type cannot round-trip the values anyway' is wrong — StringHeap stores exact UTF-16 code units (as_utf16/intern_utf16/is_well_formed), so lone surrogates are losslessly representable and the defect is fixable within the existing string type; also the as_f64 error branch (json.rs:170-173) is dead code with default features. Since all failure modes are edge cases (unpaired surrogates, overflow literals, >128 nesting) with no security or common-path impact, high overstates it.

**Verifier correction:** Core claim confirmed: JSON.parse/QuoteJSONString delegation to serde_json (default features + preserve_order) produces real, undocumented ECMA-262 conformance failures — SyntaxError on >128 nesting, on 1e999 (must be Infinity), and on lone-surrogate \u escapes (must be accepted), plus silent U+FFFD corruption of unpaired surrogates in both stringify output (spec requires "\ud800") and parse input via the lossy as_str()/to_string path. Two corrections: (1) the engine's string type CAN round-trip lone surrogates — StringHeap stores exact UTF-16 (as_utf16, intern_utf16), only the JSON path picks the lossy UTF-8 view, so this is fixable without new string infrastructure; (2) severity is medium, not high — escaping for well-formed strings matches QuoteJSONString exactly and ordinary JSON round-trips correctly, so the blast radius is test262-visible edge cases and rare real-world payloads, with no security or memory-safety angle.

**Suggestion:** Write the ~200-line spec JSON parser over the engine's UTF-16 strings (reusing string_to_number-style numeric handling) and an in-house QuoteJSONString; both are small and remove a semantic dependency on an external dialect.

### [HIGH] Dual string representation (UTF-8 Value::String vs UTF-16 HeapString) with per-method choice of view

- **Kind:** `inelegant` · **Location:** `src/storage/string_heap.rs:92` · **Status in project docs:** documented (AS-plan)
- **Verification verdict:** confirmed

**Evidence:** StringData eagerly stores BOTH `units: Rc<[u16]>` and a lossy `text: String` for every heap string; builtins split arbitrarily: charAt/slice/includes use string_receiver_utf16, but trim/toLowerCase/toUpperCase (string.rs:339-394), padStart/padEnd (string_extra.rs:211), JSON, encodeURI, parseInt, and the default sort comparator (sort.rs:313 string_argument_text) use the lossy UTF-8 view. abstract_equality needs four extra match arms just for BigInt×{String,HeapString} (equality.rs:38-51).

**Why it matters:** Any string containing a lone surrogate is silently corrupted to U+FFFD the moment it flows through a UTF-8 consumer ('\uD800'.trim(), encodeURI('\uD800') should throw URIError, default sort ordering of lone surrogates). Two representations of one concept double memory per string, force combinatorial match arms engine-wide, and make 'which view is this method using' a per-call correctness decision.

**Verifier notes:** StringData (src/storage/string_heap.rs:92-99) does eagerly store both `units: Rc<[u16]>` and a lossy `text: String` (built via `from_utf16_lossy` for ill-formed strings), and `JsString::as_str` is documented as "replacing lone surrogates with U+FFFD". The per-method split is real: string.rs lines 106-336 use `string_receiver_utf16`, while trim/trimStart/trimEnd/toLowerCase/toUpperCase (string.rs:345-390), padStart/padEnd (string_extra.rs:211), the default sort comparator (array/sort.rs:313 via `string_argument_text`), encodeURI/parseInt (global.rs:149,198-203 via `to_string`), and JSON.rawJSON (json_raw.rs:97) all flow through `to_string_primitive` (conversion.rs:358), which unconditionally takes the lossy `as_str()` view with no `is_well_formed` guard anywhere on that path. Lone surrogates genuinely enter the system (lexer `append_utf16_value` pushes raw u16 for `\uD800` escapes; Value::String, a Rust String, cannot hold them, so they are always HeapString), so `'\uD800'.trim()` returns '�', `encodeURI('\uD800')` returns '%EF%BF%BD' instead of the spec-mandated URIError (encode_uri_value takes `&str`, making that error unreachable for unpaired surrogates), and the default sort compares 0xFFFD instead of 0xD800. equality.rs:38-51 has exactly the four extra BigInt×{String,HeapString} arms claimed, and the StringData doc comment ("JavaScript semantics use `units`") is an invariant the builtins themselves violate, confirming rather than refuting the claim. Severity high is fair for a systemic per-call correctness hazard causing silent corruption and spec divergence across many builtins in an ECMAScript engine.

**Verifier correction:** Two minor calibrations, neither weakening the finding: (1) "double memory per string" is approximate — actual overhead is units (2 bytes/unit) plus UTF-8 text, and the heap's byte accounting (`storage_bytes`) counts only `text.len()`, so the UTF-16 half is also invisible to the max_bytes limit; (2) the BigInt×HeapString equality arms use the lossy view but happen to yield correct results, since U+FFFD makes BigInt parsing fail exactly as a lone surrogate would — the arms are bloat as claimed, but not themselves a correctness bug.

**Suggestion:** Collapse to one canonical representation (UTF-16 with an optional ASCII/Latin-1 compact form, or WTF-8) and derive UTF-8 views lazily; the inventory (line 312) documents the split but not the corrupting consumers.

### [HIGH] Error::runtime (uncatchable engine failure) used for user-reachable JS conditions

- **Kind:** `risk` · **Location:** `src/runtime/native/builtins/object_static.rs:474` · **Status in project docs:** documented (AS-plan)
- **Verification verdict:** partially-confirmed

**Evidence:** `Object.create` with a function prototype: Error::runtime("Object.create callable prototypes are not supported"); Object.setPrototypeOf on functions/undefined → Error::runtime (lines 279-289, 492); Object.getPrototypeOf(5) → Error::runtime (object.rs:123-126); Array generics on primitive receivers → ensure_array_like_object Error::runtime (array/generic.rs:501-512); `new Array(1.5)` → Error::runtime("invalid array length") instead of RangeError (array/mod.rs:562-571). try/catch converts only Completion::Throw (bytecode/control/try_catch.rs:113); Error::Runtime propagates as an engine failure.

**Why it matters:** Legal JavaScript (Object.create(function(){}), Object.setPrototypeOf(fn,null), Object.getPrototypeOf('x'), [].push.call(5), new Array(1.5)) terminates the evaluation with an uncatchable engine error rather than a spec TypeError/RangeError or the spec's non-error behavior. This inverts the AS-04 boundary: engine-failure taxonomy is being used to paper over missing ToObject/wrapper-object and function-prototype-mutation semantics.

**Verifier notes:** The uncatchability mechanics are exactly as claimed: runtime_exception_value (src/runtime/control/assertions.rs:9-24) converts only Error::JavaScript/JavaScriptError into throwable values, bytecode_error_completion (src/runtime/bytecode/execution.rs:453-468) propagates Error::Runtime as Err terminating evaluation, and try_catch.rs:113 catches only Completion::Throw — the AS-04a doc itself says Runtime errors "bypass JavaScript catch". Four of the five cited example families are confirmed reachable from legal JS: Object.create(function(){}) (object_static.rs:473-475), Object.setPrototypeOf with undefined target or non-object prototype (object_static.rs:282-284, 492), Object.getPrototypeOf on any primitive (object.rs:123-126, since semantic_object_ref returns None for primitives with no ToObject fallback), Array generics on primitive receivers via ensure_array_like_object (array/generic.rs:501-512, reached from eval_direct_array_push's fallthrough), and new Array(1.5) (array/mod.rs:562-571) which should be a catchable RangeError. However, one headline example is wrong: Object.setPrototypeOf(fn, null) succeeds — semantic_try_set_prototype (semantic_object/prototype_integrity.rs:61-64) handles Value::Function via set_function_static_parent and returns Some(true), making the Error::runtime arm at object_static.rs:285-289 unreachable dead code for function targets, and NativeFunction targets get a catchable TypeError. The function-prototype gap behind Object.create is also documented as known storage debt (docs/semantic-architecture-inventory.md:214) owned by AS-05, so part of this is tracked debt rather than an unnoticed inversion.

**Verifier correction:** Core claim holds: Error::Runtime is uncatchable by JS (runtime_exception_value excludes it; bytecode_error_completion re-raises it as an engine Err; try/catch only handles Completion::Throw), and legal JS reaches it at Object.create(function(){}) (object_static.rs:473-475), Object.setPrototypeOf(undefined,...) Correction: Object.setPrototypeOf(fn, null) does NOT fail — semantic_try_set_prototype returns Some(true) for Value::Function via set_function_static_parent, so object_static.rs:285-289 is unreachable for function targets, and NativeFunction targets produce a catchable TypeError, not an engine error. Note also that the function-valued-prototype limitation behind the Object.create case is documented "unsupported storage debt" (docs/semantic-architecture-inventory.md:214) owned by AS-05, so part of this is tracked interim debt; the remaining sites (getPrototypeOf on primitives, array-generic receivers, Array(1.5)) do violate the completed AS-04a contract that spec-required exceptions be typed JS throws.

**Suggestion:** Audit every Error::runtime in builtins: user-reachable ones become Error::type_error/exception; the underlying gaps (no primitive wrappers, functions not supporting [[SetPrototypeOf]], functions as separate Value variants) are AS-02-owned but the mis-typed errors are not recorded anywhere.

## Additional low-severity findings (not separately verified)

### [LOW] toFixed evaluates fractionDigits before ThisNumberValue, inverting spec order (its siblings get it right)

- **Kind:** `inelegant` · **Location:** `src/runtime/native/builtins/number_format.rs:24` · **Status in project docs:** UNDOCUMENTED

**Evidence:** eval_number_prototype_to_fixed: digits conversion (line 24) and the RangeError check (25-30) run before number_receiver_value(this_value) (line 32). Spec order is ThisNumberValue first. eval_number_prototype_to_exponential does receiver first (line 49) — the two siblings disagree.

**Why it matters:** Observable via `Number.prototype.toFixed.call({}, {valueOf(){throw 'x'}})`: spec throws TypeError from ThisNumberValue, this code runs the valueOf (side effect) and throws its exception. Ordering bugs in one of three near-identical functions are copy-paste drift.

**Suggestion:** Hoist a shared prologue (receiver first, then argument coercion) across toFixed/toExponential/toPrecision.

### [LOW] f64→integer conversions routed through decimal string formatting and reparsing

- **Kind:** `inelegant` · **Location:** `src/runtime/abstract_operations/conversion.rs:182` · **Status in project docs:** UNDOCUMENTED

**Evidence:** finite_nonnegative_integer_to_usize: `format!("{integer:.0}").parse::<usize>()`; same pattern in length_from_integer (line 274), usize_to_number (line 192-195), array_length_from_number (builtins/array/mod.rs:569), number radix arg (number.rs:364), number_finite_integer (number.rs:410), number_to_usize (string_regexp.rs:653).

**Why it matters:** A numeric conversion in the hottest conversion layer (ToLength feeds every array method) allocates a String and runs a decimal parser to avoid an `as` cast. It is slow, obscures the actual range contract, and is copy-pasted seven times with per-site error strings.

**Suggestion:** One checked f64→u64 helper: verify 0.0..=2^53, then `value as u64` under a single documented #[allow] (matches count_from_digits precedent in number_format.rs:111-116), or bit-manipulate the mantissa.

### [LOW] get_async_iterator performs a duplicate observable @@iterator lookup on fallback

- **Kind:** `inelegant` · **Location:** `src/runtime/abstract_operations/iterator.rs:201` · **Status in project docs:** UNDOCUMENTED

**Evidence:** get_async_iterator: after `self.iterator_method(iterable)?` returns None (line 201), the fallback `self.get_iterator(iterable)` (line 206) calls iterator_method again internally (lines 153/159/165/183).

**Why it matters:** GetMethod(@@iterator) is observable through proxies and getters; spec performs it once. Also duplicates the sync/async wrapping decision across two functions.

## Optimization opportunities (post-cleanup)

### [MEDIUM] String methods copy the entire receiver per call; naive O(n·m) substring search

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/native/builtins/string.rs:473` · **Status in project docs:** UNDOCUMENTED

**Evidence:** string_receiver_utf16 → to_utf16_string → `text.as_utf16().to_vec()` (or encode_utf16().collect for UTF-8 strings): charAt of a 1 MB string copies 2 MB before reading one unit. string_index_of_from is `(position..=length).find(|i| starts_with_at(...))` (lines 549-558); replace_all_positions uses windows().position() (string_modern.rs:410-428); advance_utf16_index re-encodes the whole string per empty regexp match (string_regexp.rs:628). StringData additionally stores an eager UTF-8 copy of every string (string_heap.rs:92-99).

**Why it matters:** Once the string representation is unified (see dual-representation finding), receivers can be borrowed Rc<[u16]> slices — zero-copy charAt/slice/iteration — and substring search can use two-way/memchr-class algorithms. The eager dual encoding roughly doubles string heap usage today for no semantic benefit.

**Suggestion:** Pass &[u16] (Rc-backed) receivers into string builtins; drop StringData.text to lazy; adopt a real substring search. These follow naturally after the representation is fixed.

### [MEDIUM] Quadratic key handling in enumeration and per-element string keys in hot iteration

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/semantic_object/keys.rs:20` · **Status in project docs:** UNDOCUMENTED

**Evidence:** semantic_enumerate_property keys: `if !keys.contains(&key) { keys.push(key) }` inside the loop over all own keys of every prototype — for-in over an object with k properties is O(k^2) String comparisons. IteratorSource::ArrayIndex steps via `let key = index.to_string(); self.get_named(&array, &key)` (abstract_operations/iterator.rs:269-274) — one heap allocation plus interned-name lookup per element of every for-of over an array. JSON circularity uses `stack.contains(&id)` per object (json.rs:658).

**Why it matters:** These are the spec-faithful generic paths that everything falls back to when guards miss; with the fast-path guards needing to get stricter (accessor/next-patch findings), the generic paths' constants matter more, and O(n^2) for-in is reachable today with a 10k-property object.

**Suggestion:** HashSet (or sorted-vec) dedup for enumeration; an index-keyed Get entry point that skips string materialization for canonical array indices; a small hash set once the JSON stack passes ~32 entries.
