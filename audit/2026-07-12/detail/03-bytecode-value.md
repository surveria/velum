# Bytecode model, Value, storage, errors

> Детальный отчёт аудита архитектуры rs-quickjs. Подсистема: `bytecode-value`. Сгенерировано многоагентным аудитом 2026-07-12; каждая находка уровня medium/high прошла адверсариальную проверку отдельным агентом. Материал на английском (исходный текст аудита).

## Subsystem assessment

This scope is unusually disciplined for a prototype: typed newtype ids everywhere, checked arithmetic and try_reserve on every allocation path, an instruction-aligned span side table validated at construction, owner-stamped strings/symbols via an Rc-capability VmIdentity, and typed error channels that genuinely separate JS throws from engine/resource failures. The project's self-documentation is exceptional — most known debt (12-variant Value, 4 id spaces, Test262Error fallback, direct-native speculation, quickened numeric opcodes) is recorded with AS-xx owners and mechanical no-growth guards. The central architectural weakness is the instruction set itself: BytecodeInstruction is a ~110-variant cross-product of (operation x reference-form x call-convention) rather than an orthogonal core with composable reference operands — 19 call/construct shapes, 6 update shapes, 6 compound shapes, and a 7-opcode family added just for private names show the set is closed under the language only by enumeration, so every new syntactic form multiplies opcodes and touches every structural consumer. That consumer cost is already visible: the 800-line metrics module hand-duplicates the tree walk six times and has silently drifted out of sync with the instruction set (DoWhile, With, DestructurePattern, and super-assignment blocks are invisible to all counts, masked by a catch-all `_ => 0`). Optimization operands (NativeCallTarget speculation, ArrayLength/ArrayIndex name-based quickening) are encoded inside the semantic instructions rather than in a side tier, which AS-08 gates behind runtime guards but leaves entangled with the shareable bytecode representation. The dual String(String)/HeapString(JsString) value representation and the doubly-stored, half-accounted string/atom tables are the main undocumented storage debt. Overall: the semantic layering (AS-02/03/04) is real and enforced; the representation layer beneath it (ISA shape, string duality, side-table counting) is where enumeration-style design will keep taxing every future feature.

## Strengths

- BytecodeBlock pairs every instruction with a SourceSpan and rejects unknown or mixed source identities at construction (src/bytecode/block.rs:41-63), so diagnostics can never silently lose provenance.
- Quickened numeric opcodes (BytecodeNumericBinaryOp etc.) each carry a total generic_binary()/generic_unary() mapping back to the semantic BinaryOp/UnaryOp (src/bytecode/numeric.rs), keeping exactly one semantic source of truth with the specialization as a pure relabeling.
- Storage arenas (StringHeap, SymbolTable, AtomTable) enforce count and byte limits pre-commit with checked_add/try_reserve and reconcile counters on sweep with explicit underflow errors - no silent accounting drift in the happy path.
- JsString/JsSymbol are stamped with an Rc-capability VmIdentity (src/ownership.rs) compared by pointer identity, making cross-VM aliasing structurally impossible for these primitives rather than probabilistically unlikely.
- Error enum cleanly separates Lex/Parse/Runtime/JavaScript/ResourceLimit channels; with_context deliberately never rewrites a thrown JS value, and resource limits cannot become catchable.
- SourceId is a deterministic domain-separated 128-bit content hash with explicit length framing (src/source.rs:40-56), stable across VMs and recompilation without retaining source text.
- The project's own inventory and stabilization plan document nearly all major debt with owners and mechanical guards - an audit of this codebase mostly confirms the map instead of redrawing it.
- No unsafe, unwrap, expect, panic, or unchecked indexing anywhere in the audited ~4.6k lines; every slot access is Option-checked with a typed error.

## Confirmed findings (adversarially verified)

### [HIGH] Instruction set is a syntax cross-product, not an orthogonal core

- **Kind:** `inelegant` · **Location:** `src/bytecode/types.rs:440` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** BytecodeInstruction enumerates 19 call/construct shapes (CallBinding, CallValue, CallStaticMember, CallComputedMember, their 4 Spread twins, CallPrivateMember(+Spread), CallSuper(+Spread), CallSuperMember(+Spread), CallComputedSuperMember(+Spread), Construct, ConstructValue, ConstructValueSpread), 6 Update* shapes, 6 Compound* shapes, 6 Delete/Assign shapes - one per (operation x reference-form x convention) combination. Adding private names alone required 7 new opcodes (PrivateMember, PrivateAssign, CompoundPrivateProperty, UpdatePrivateProperty, CallPrivateMember, CallPrivateMemberSpread, PrivateIn). The `in` operator has three encodings: Binary{op:In, property_access:Some(_)} (line 543), InStaticProperty (line 547), and PrivateIn (line 611).

**Why it matters:** The ISA is closed under the language only by enumeration. Every new reference form or call convention multiplies variants, and each variant must be threaded through every structural consumer (interpreter dispatch, continuation records, linear planner, six metrics traversals). This is the direct cause of the metrics drift bug and makes the instruction set the most expensive place in the engine to evolve. A factored design (reference operand + generic Call/Update/Compound cores) would grow additively, not multiplicatively.

**Verifier notes:** The enum at src/bytecode/types.rs:440 contains exactly the claimed cross-product: 19 call/construct variants (verified by exact count), 6 Update* and 6 Compound* forms, 7 private opcodes (lines 583-613), and three `in` encodings (lines 543, 547, 611) plus a fourth in the linear planner (BytecodeLinearOp::InStaticPropertyBinding). The threading cost is real: interpreter dispatch re-enumerates variants across multiple routing levels with `_ => Err(runtime mismatch)` fallbacks (so misses fail at runtime, not compile time, despite mod.rs:36 claiming exhaustiveness), the linear planner maintains a parallel mirror enum, and the six metrics traversals end in `_ => 0` catch-alls. Critically, the asserted metrics drift is independently verifiable in code: DoWhile and With (emitted at compiler/control.rs:24,86) appear in no metrics arm and not in is_leaf_instruction, so their nested blocks silently count 0 in all six metrics; SuperPropertyAssign/CompoundSuperProperty value blocks and DestructurePattern fall through the same way, and ComputedSuperMember is misclassified as a leaf despite carrying an expression block. Two peripheral inaccuracies: "6 Delete/Assign shapes" is actually 9 (4 Delete + 5 Assign), and continuation records are generic Function/Block frames that do not grow per reference-form variant, so that consumer is overstated.

**Verifier correction:** Two corrections to the evidence, neither weakening the core claim: (1) "6 Delete/Assign shapes" is actually 4 Delete + 5 Assign = 9 variants; (2) continuation records (runtime/bytecode/continuation.rs) are generic Function/Block frames — the per-variant threading burden falls on interpreter dispatch, the linear planner's parallel BytecodeLinearOp enum, and the six metrics traversals, not on continuation records. Additionally, part of the variant multiplication (embedded NativeCallTarget, ArrayIndexMember/ArrayLength, Number* ops) is deliberate fast-path specialization, so a fully factored reference-operand design has a hot-path indirection trade-off. The drift mechanism is concretely present today: DoWhile, With, SuperPropertyAssign, CompoundSuperProperty, and DestructurePattern nested blocks are silently uncounted by all six metrics via the `_ => 0` catch-alls in src/bytecode/metrics/mod.rs, and ComputedSuperMember is classified as a leaf despite carrying a nested block.

**Suggestion:** After AS-09, introduce a Reference/AccessMode operand type consumed by single generic Call/Construct/Update/CompoundAssign/Delete instructions; keep the current fused forms only as a compiler-internal encoding if profiling justifies them, behind one lowering function.

### [MEDIUM] Bytecode metrics silently omit five nested-block instructions behind a dead exhaustiveness guard

- **Kind:** `risk` · **Location:** `src/bytecode/metrics/mod.rs:165` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** Every traversal ends `instruction if instruction.is_leaf_instruction() => 0, _ => 0` - the catch-all makes the leaf classification unenforceable. DoWhile{body,condition}, With{body}, DestructurePattern{pattern}, SuperPropertyAssign{value: BytecodeBlock}, and CompoundSuperProperty{value: BytecodeBlock} appear nowhere in metrics/mod.rs (grep confirms zero hits), so their nested blocks fall to `_ => 0`. ComputedSuperMember{expression: BytecodeBlock} is explicitly classified as a leaf (line 590) despite carrying a block.

**Why it matters:** instruction_count, binding_operand_count, etc. are the public CompiledScriptUsage API and oracles for ~20 smoke tests, and they are silently wrong for any script containing do-while, with, destructuring, or super assignment. Worse, the `_ => 0` arm guarantees every future nested-block instruction also miscounts with no compiler or test signal - the exact failure mode already happened five times.

**Verifier notes:** The cited code matches the claim exactly: all six metric traversals in src/bytecode/metrics/mod.rs end with `instruction if instruction.is_leaf_instruction() => 0` followed by `_ => 0`, making the leaf guard behaviorally dead and defeating match exhaustiveness. DoWhile, With, DestructurePattern, SuperPropertyAssign, and CompoundSuperProperty appear nowhere in the metrics module (verified by grep across mod.rs, traversal.rs, targets.rs), all carry BytecodeBlock children per src/bytecode/types.rs, and all are actually emitted by the compiler (compiler/control.rs:24,86; member.rs:16,265; expression.rs:183; mod.rs:270), so their nested blocks silently contribute 0. ComputedSuperMember is indeed listed as a leaf at mod.rs:590 despite its `expression: BytecodeBlock` being populated by compile_expression. The counters feed the public CompiledScriptUsage (lib.rs:33 re-export) and exactly 20 test files assert against the bytecode_* getters; no documentation, debug assert, or alternate guard mitigates this — and the claim slightly undercounts, since UpdateSuperProperty's BytecodeSuperProperty::Computed also carries an uncounted expression block.

**Verifier correction:** Claim fully verified, and it slightly understates the gap: UpdateSuperProperty is a sixth unhandled variant with a nested block (BytecodeSuperProperty::Computed { expression: BytecodeBlock }), and SuperPropertyAssign/CompoundSuperProperty additionally miss that computed-super-property expression block on top of their `value` block. Medium severity is fair — metrics do not affect execution semantics, but the public CompiledScriptUsage API silently underreports and the `_ => 0` catch-all guarantees unnoticed drift for future variants.

**Suggestion:** Delete the `_ => 0` arm so the match is exhaustive (the leaf arm should enumerate leaves explicitly), or better, replace the six hand-walks with one for_each_child_block visitor; add the five missing instructions either way.

### [MEDIUM] Compile-time quickening by source spelling is baked into the semantic ISA (ArrayLength, ArrayIndex* families)

- **Kind:** `microopt-debt` · **Location:** `src/bytecode/types.rs:614` · **Status in project docs:** documented (AS-plan)
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** Every static member read compares the property name: `if property.name().as_str() == ARRAY_LENGTH_PROPERTY { emit(ArrayLength) } else if let Some(index) = compile_array_index(...) { emit(ArrayIndexMember) }` (src/compiler/member.rs:87-90). BytecodeArrayIndex::parse (types.rs:255-261) canonicalizes any numeric-spelled property name into dedicated ArrayIndexMember/ArrayIndexAssign/UpdateArrayIndexProperty/CompoundArrayIndexProperty opcodes, each carrying BOTH the parsed index and the original BytecodeProperty for fallback.

**Why it matters:** The specialization decision is made once at compile time from name spelling, with no runtime feedback, and lives in the shared instruction encoding instead of an optimization tier: every `.length` read on any non-array object still enters the array-specialized handler, and the redundant (property, index) operand pair is state that can desynchronize. It also quadruples the Update/Compound/Assign opcode families (the ArrayIndex row exists solely for this).

**Verifier notes:** The cited code matches the claim: compile-time spelling checks in src/compiler/member.rs (lines 49, 87-93, 175, 302) and control.rs:437 select ArrayLength/ArrayIndex* opcodes, and those opcodes carry both the parsed index and the original BytecodeProperty. However, the "why bad" reasoning largely fails: every specialized handler in src/runtime/bytecode/array.rs is gated behind a runtime optimizer flag (optional_optimizations_enabled) and dynamically verifies the receiver is an array/typed-array/string before using the index, falling through to the identical generic path (get_static_property_value with the property operand) on mismatch — so semantics never depend on the compile-time guess and a non-array `.length` read pays only a few cheap checks. The generic path itself has per-site inline caching keyed by StaticPropertyAccessId, contradicting "no runtime feedback"; the (property, index) pair cannot practically desynchronize since BytecodeArrayIndex's only constructor is parse(&property) with a round-trip check and both operands are immutable after compile; and "quadruples the opcode families" is wrong — the ArrayIndex row adds one variant per family (5 opcodes total), not 4x.

**Verifier correction:** What survives is mild ISA-cleanliness debt: the shared instruction encoding carries a spelling-derived, duplicated (property, index) operand pair, and the per-site specialization is permanent with no re-quickening, so mispredicted sites pay a small constant guard cost. But the specialization is a guard-verified hint gated behind a runtime optimization flag, always falls back to the exact generic semantic path, and the generic path already has inline-cache runtime feedback — so there is no correctness hazard and negligible performance hazard. Downgrade to low (micro-opt debt / encoding aesthetics), and drop the "quadruples opcode families" and "can desynchronize" assertions.

**Suggestion:** Fold ArrayLength/ArrayIndex* into the generic StaticMember/ComputedMember instructions and move array specialization to the runtime cache tier AS-08 already owns (shape-guarded IC on the ordinary-object tail), where it can adapt per-site.

### [MEDIUM] NativeCallTarget speculation operands embed a ~150-variant builtin-method enum inside call instructions, keyed by source name

- **Kind:** `microopt-debt` · **Location:** `src/bytecode/types.rs:639` · **Status in project docs:** documented (AS-plan)
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** CallBinding/CallBindingSpread/CallStaticMember/CallComputedMember/Construct all carry `native: Option<NativeCallTarget>`. The compiler fills it via `NativeCallTarget::from_binding_name(name.as_str())` and `from_property_name(property.as_str())` (src/compiler/call.rs:26,41), including for computed string/number literals (call.rs:189-196). NativeCallTarget (src/api/native_call.rs, 387 lines) flatly enumerates ArrayPush, MathAbs, JsonParse, ... per method.

**Why it matters:** Optimization state is entangled with the semantic, immutable, shareable bytecode: the hint is a compile-time guess from spelling (`anything.push()` is speculated as Array.prototype.push) that runtime guards must re-validate on every call, and every new accelerated builtin permanently grows a compiler-visible enum plus two name tables. A per-Context call-site cache (which already exists as CallValueCache) makes the encoded hint redundant.

**Verifier notes:** The factual description is accurate: CallBinding/CallBindingSpread/CallStaticMember/CallComputedMember/Construct (src/bytecode/types.rs:459,637-661) embed Option<NativeCallTarget> filled from source spelling (src/compiler/call.rs:26,41,56,112,189-197), and the enum has ~163 variants with name tables. But the reasoning has three flaws: (1) the hint is only re-validated per-call on the CallBinding path — member calls check per-access-site caches first (src/runtime/native/function/direct.rs:38,76-93) and consult the hint only on misses; (2) CallValueCache serves only CallValue instructions keyed by StaticCallSiteId and never covers the hint-bearing instructions — the redundancy argument survives only via different caches (access-site/binding native-kind caches, and the hint-free kind-derivation fallback at direct.rs:47-52 and eval_cached_native_dynamic_member_call); (3) the operand is not purely optimization state: NativeCallTarget::Eval on CallBinding implements spec-mandated direct-eval detection (src/runtime/mod.rs:493-497), which is syntactic and unrecoverable from runtime caches, and the operand is immutable static data — mutable optimization state lives in per-Context caches, so bytecode shareability is not compromised. The valid kernel is the maintenance surface: every new accelerated builtin grows a compiler-visible enum, a name table, the parallel NativeFunctionKind mapping, and dispatch matches.

**Verifier correction:** The structural facts hold (hint operands on five call instructions, ~163-variant compiler-visible enum plus name tables, spelling-based speculation), and the enum-growth maintenance cost is real. But the hint is guarded by canonical-identity checks and consulted only on cache misses for member calls; CallValueCache does not cover these instructions (different caches do, which is what makes the hint mostly redundant); and NativeCallTarget::Eval carries spec-required direct-eval semantics that cannot be replaced by any runtime cache. Static immutable hints in shared bytecode plus per-Context mutable caches is a conventional interpreter design, so this is low-severity maintainability debt, not entangled optimization state.

**Suggestion:** Once AS-10 profiling exists, measure whether the encoded hint outperforms the existing call-site cache; if not, delete the operand and the enum, keeping direct-native dispatch purely in the VM-local cache tier.

### [MEDIUM] PushLiteral(Value) lets VM-local ids type-check into immutable shareable bytecode

- **Kind:** `risk` · **Location:** `src/bytecode/types.rs:444` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** PushLiteral carries the full 12-variant Value, which includes ObjectId/FunctionId/HeapString/Symbol id-bearing variants; so do Expr::Literal(Value) (src/ast/expression.rs:41) and BytecodeDirectThrow::Literal(Value) (src/bytecode/fast_path.rs:10). **Why it matters:** CompiledScript is VM-independent by design; a single future code path that builds a PushLiteral from a runtime Value (e.g. constant folding, a cache, or a serializer) would smuggle a VM-local slot id across VM boundaries and type-check cleanly. The project already has the correct 5-variant portable type (OwnedValue) but does not use it where the invariant actually lives.

**Verifier notes:** Every cited fact verified: PushLiteral(Value) at src/bytecode/types.rs:444, the 12-variant Value with id-bearing variants at src/value/kind.rs:8-21, Expr::Literal(Value) at src/ast/expression.rs:41, BytecodeDirectThrow::Literal(Value) at src/bytecode/fast_path.rs:10, parser emitting only the 5 portable literal kinds at src/parser/expression.rs:422-442, the documented invariant at docs/semantic-architecture-inventory.md:455, and CompiledScript being a public reusable cross-Context artifact. No structural guard, assertion, or validation pass exists; the compiler clones the AST Value verbatim (src/compiler/expression.rs:34-36). The auditor missed one partial mitigation: PushLiteral execution routes through runtime_value/checked_value (src/runtime/bytecode/instruction_stack.rs:24-25, src/runtime/values.rs:235-265), which identity-checks HeapString/Symbol and rejects foreign ones at runtime — but Function/NativeFunction/HostFunction/Object ids are explicitly passed through unchecked, and FunctionId/ObjectId are bare unbranded usize indices, so the most dangerous variants would resolve silently to the wrong record in another VM (and per src/runtime/gc.rs:579 are not even durable across GC in the same VM). The nuance tempers but does not defeat the risk, so the claim is confirmed with a corrected note.

**Verifier correction:** Accurate as stated, with two refinements. (1) There is a partial dynamic backstop the claim omits: PushLiteral execution goes through Context::runtime_value -> checked_value (src/runtime/values.rs:235-265), which rejects HeapString/Symbol values whose VmIdentity differs from the executing VM, so those two variants would fail loudly rather than smuggle silently. (2) OwnedValue has 6 variants (it includes String), not 5, and is not a drop-in literal type since bytecode string literals use PushString(StaticString); the natural enforcing type would be the 5-variant subset the parser actually emits. Today the invariant holds by convention only: the compiler is the sole constructor of persistent PushLiteral and clones parser-built literals verbatim; all runtime PushLiteral mentions are pattern-match recognizers, not constructors.

**Suggestion:** Introduce a BytecodeLiteral enum (Undefined/Null/Bool/Number/BigInt) for PushLiteral, Expr::Literal, and BytecodeDirectThrow; the invariant becomes a type instead of a guard-script assertion.

### [MEDIUM] Dual string representation: Value::String(String) and Value::HeapString(JsString) split one concept across ~200 runtime dispatch sites

- **Kind:** `inelegant` · **Location:** `src/value/kind.rs:14` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** Value has both String(String) (UTF-8, unowned, cannot represent lone surrogates, no VM identity) and HeapString(JsString) (owned exact UTF-16 + cached UTF-8). Cross-variant PartialEq re-encodes the entire Rust string to UTF-16 per comparison (kind.rs:55-59). src/runtime has 102 `Value::String(` and 95 `Value::HeapString(` match sites.

**Why it matters:** Every string consumer pays double dispatch; equality between the two forms is O(n) re-encoding; and Value::String payloads live entirely outside StringHeap, so they are invisible to the string byte limits, the intern index, and the AS-07 sweep - an unowned, unbounded, unaccounted allocation class next to a meticulously accounted one. The inventory records the two variants but no AS item owns their unification or the accounting gap.

**Verifier notes:** The structural facts check out: src/value/kind.rs:14-15 has both variants, src/runtime contains exactly 102 `Value::String(` and 95 `Value::HeapString(` sites, cross-variant PartialEq (kind.rs:55-59) compares content, and no AS item owns unification. But the "why bad" reasoning misses the architecture's normalization boundary: `Context::runtime_value` (src/runtime/values.rs:22-27) interns `Value::String` into the accounted StringHeap and is called at ~77 sites covering property writes (property/static_names/mod.rs:260/298/360/399), property reads (static_names/read.rs:38/198/211, dynamic.rs:27), binding declarations, parameters, and function fast paths — `Value::String` is a deliberate transient representation, notably an in-place mutable concat accumulator (values.rs:63-133) that avoids re-interning every intermediate. "Unbounded" is wrong: every admission and growth path enforces `limits.max_string_len` (checked_value/check_string_len at values.rs:235-237/314-323, push_concat_text, concat_utf16_values), and all retaining containers are count-limited in the storage ledger. "Invisible to the AS-07 sweep" is by design, not a leak — the payload is inline Rust-owned and dropped with its container, exactly like Number/BigInt (gc.rs:350-355 treats it as a primitive; BigInt heap payloads are equally outside the byte ledger). The equality claim also overstates: the cross-variant arm uses the lazy, allocation-free, short-circuiting `encode_utf16()` iterator, and the same-variant HeapString path is itself O(n) full UTF-16 slice comparison (string_heap.rs:136-140), not O(1) id equality, so cross-variant comparison is not asymptotically anomalous.

**Verifier correction:** Real residue: (1) ~197 dual-dispatch match sites in src/runtime are a genuine, if deliberate, maintenance cost; (2) a few paths persist Value::String without runtime_value normalization — Map/Set entries store raw key/value (src/runtime/collections.rs:231-252) and bootstrap properties such as Value::String("Object") at src/runtime/object/heap.rs:264 — and those bytes appear in no byte ledger (only slot counts), unlike HeapString bytes which are exactly accounted (string_heap.rs:249-258). Aggregate memory remains bounded (count limits x max_string_len), so this is a coarseness gap in byte accounting, not an unbounded allocation class. The inventory documents both variants but no AS item owns either unification or extending byte accounting to inline string payloads in collection entries.

**Suggestion:** Migrate toward a single interned/heap string representation (small-string inline optimization can come later as a representation experiment per the plan); at minimum, route string-producing runtime paths through StringHeap so limits and sweep see them.

**Verifier notes:** The code facts check out: StringData eagerly stores both UTF-16 units and a full UTF-8 String (string_heap.rs:92-117), storage_bytes() counts only text.len() (131-133), and insert_string enforces max_bytes against it (249-258), so physical content retention is ~3x the accounted bytes for ASCII. What remains is a logical-vs-physical metric gap that is documented design, not an accounting defect.

**Verifier correction:** The mechanics are accurate — storage_bytes() counts only the UTF-8 rendering while each heap string also retains its UTF-16 units, so physical content retention is ~3x the enforced max_bytes for ASCII. But this is the documented AS-05b2b contract, not a violation of it: payload bytes are defined as logical 'owned UTF-8 length', explicitly excluding physical layout and duplicate references to shared payload (which also covers the Rc-cloned intern key — no payload is duplicated there), and architecture.md separately defers physical heap budgets to future work. Residual issue is docs/API clarity: embedders may read the HeapString byte limit as a memory bound, and for a UTF-16-authoritative heap the UTF-16 code-unit length would arguably be the more natural logical metric. Worth a doc note or a future physical-budget limit, not a contract-undermining defect.

**Suggestion:** Count units (2*len) plus text bytes in storage_bytes, or make the UTF-8 rendering lazy/optional for non-well-formed and rarely-displayed strings.

### [LOW] Every thrown value eagerly formats a display string via diagnostic Display, duplicating state

### [LOW] Three coexisting encodings for short-circuit control: structured blocks, Keep-jumps, and a dedicated NullishCoalescing instruction

## Optimization opportunities (post-cleanup)

### [MEDIUM] AtomTable interning is O(n^2) via sorted-Vec insertion and stores every name twice

- **Kind:** `perf-opportunity` · **Location:** `src/storage/atom.rs:85` · **Status in project docs:** UNDOCUMENTED

**Evidence:** intern() binary-searches `entries: Vec<AtomEntry>` then `self.entries.insert(position, AtomEntry::new(name, id))` - an O(n) element shift per new atom, O(n^2) to intern n atoms. Each name is cloned into both `names: Vec<String>` (line 84) and the AtomEntry (line 85), while `bytes` accounting adds `name.len()` once (line 73-76).

**Why it matters:** Atoms are interned for every identifier and property name during compilation, so large programs pay quadratic shifting; the double storage halves the accuracy of the byte accounting and doubles real memory. Both loops are in the same function: binary_search_by (line 103-105) plus Vec::insert (line 85).

**Suggestion:** Replace with HashMap<Rc<str>, AtomId> for the index and Vec<Rc<str>> for id->name: O(1) intern, single storage, exact byte accounting - a drop-in change since AtomId is already an index into names.

### [LOW] String intern re-encodes to UTF-16 per lookup; Symbol registry is a linear-scan Vec

- **Kind:** `perf-opportunity` · **Location:** `src/storage/symbol.rs:165` · **Status in project docs:** UNDOCUMENTED

**Evidence:** StringHeap::intern_owned allocates `text.encode_utf16().collect::<Vec<_>>()` before every index probe, even on a hit (string_heap.rs:202); SymbolTable::for_key, key_for, and has_registry_key each do `self.registry.iter().find(...)` over Vec<(JsString, SymbolId)> (symbol.rs:118-122, 165-168, 179-183).

**Why it matters:** Interning is on the hot path of property-name creation, and Symbol.for/keyFor degrade linearly with registry size. Both are avoidable constant-factor and asymptotic costs that will show up once AS-10 profiling starts.

**Suggestion:** Key the string index by a raw-entry hash of the UTF-8 form (or use a hashbrown raw-entry API over units without pre-collecting); back the Symbol registry with a HashMap keyed by the interned StringId.
