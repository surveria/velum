# Cross-cutting hack census, guard allowlists, docs-vs-code drift

> Detailed architecture-audit report for rs-quickjs. Subsystem: `hack-hunt`. Generated by a multi-agent audit on 2026-07-12; every medium/high finding was adversarially verified by an independent agent reading the cited code.

## Subsystem assessment

The layering story is real, not aspirational: greps confirm runtime/bytecode are parser-AST-free, semantics have single allowlisted owners (equality, conversion, iteration, semantic_object facade), and the hack census by comment markers is effectively zero — the only frozen source-name special case is the Test262Error unbound-constructor fallback (2 references, allowlisted, but sitting on the hot `new` path and baked into the core ErrorName enum). The 2,750-line boundary script is the project's most distinctive artifact: roughly 30 check families spanning genuine invariants (empty legacy-facade and harness-opcode allowlists that shrank to zero), load-bearing debt freezes (Test262Error, the 7-function array-like-length split, 8 linear modules, 3 fast-path files), and ~150 verbatim-pinned source lines plus full struct-field freezes (48 Context fields, 12 Value variants) that mirror implementation text and will fight every refactor; it also uses GNU-only shell so it cannot run on the macOS dev platform. Workload-shaped code extends beyond the documented packed-numeric-array reduction: masked-index accumulate and masked-index `in` superinstructions, the flatMap callback-tree evaluator, and the literal-throw Try recognizer are all benchmark-idiom-shaped, though module-allowlisted. The most consequential undocumented defects are optimization-tier discipline leaks: a second per-call function-body matcher that ignores the Disabled-mode gate and charges zero steps, the flatMap fast path with no gate and non-step-neutral decline, and a numeric binary kernel copied five times whose shared bare-powf exponentiation diverges from ECMAScript and from QuickJS's explicit non-IEEE special case. Structurally, the deepest inelegances are the hybrid bytecode IR (structured tree instructions coexisting with flat jumps, forcing hand-written per-construct resume-phase records) and Rc<Mutex> binding cells that put a lock on every local-variable access in a single-threaded VM. Ownership and invariant documentation quality is otherwise the best I have seen at this scale — nearly every finding a reviewer can surface is already recorded with an AS owner, which makes the handful of undocumented mode-gate and step-parity violations stand out as the real foundation risks.

## Strengths

- Marker hygiene is exceptional: zero TODO/FIXME/HACK/XXX/'for now'/'workaround'/'special case' comments in ~101k lines of project-authored src (all 111 'temporar*' hits are legitimate domain terms like TemporaryThis/TransientTemporary); the debt ledger genuinely lives in docs/ with AS owners instead of in scattered comments.
- Verified doc claims held under grep: src/runtime and src/bytecode contain zero crate::ast/parser/lexer imports; Value::Error, ErrorObject, and text-classified 'ReferenceError:' construction are gone; the compiler has zero source-name comparisons and zero harness opcodes ('print' appears only as the native builtin's own registered name); async resume re-enters the single run_bytecode_state driver with no second interpreter.
- Several allowlists in the 2,750-line boundary script are now empty and enforced as empty (legacy property/call facades, legacy iterator facades, compiler source-name comparisons, harness opcodes, workload-shaped control recognizers) — the guard demonstrably shrank debt rather than only freezing it, and its self-test mode mutation-tests every allowlist.
- The vendored regress fork is under real control: sha256 manifest verifies clean (all files OK), the manifest covers the complete flat src set, provenance and the single semantic deviation (Unknown/Zzzz script aliases) are documented in VENDORED.md, and the root crate consumes it only via a path dependency with prohibit-unsafe.
- Fast-path discipline is mostly real where it matters: the pre-setup function fast path charges exact step parity, linear plans decline on with-environments and consult one optimizer gate, the numeric-array-reduction plan stores partial state before propagating a step-limit error, and guard misses route to shared semantic owners (eval_bytecode_number_binary, semantic_call, number_strict_equality) rather than private semantics — the exceptions are enumerated as findings.
- Resource accounting is unusually principled: 26 storage categories with an O(1) ledger independently reconciled against recount snapshots, checked arithmetic everywhere, and every fast path reasoning explicitly about step charging.

## Confirmed findings (adversarially verified)

### [HIGH] Second function-body fast-path matcher ignores the optimizer mode gate and charges zero runtime steps

- **Kind:** `microopt-debt` · **Location:** `src/runtime/function/fast_path.rs:243` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** eval_bytecode_function_fast_path (fast_path.rs:243-313) pattern-matches 7 body shapes and returns completions directly; none of its arms call charge_runtime_steps() or self.step(). Its only call site, parameters.rs:287-291 (eval_function_body_after_setup), gates solely on `self.current_with_environments().is_empty()` — there is no optional_optimizations_enabled() check anywhere on this path. Contrast: the sibling pre-setup path charges `charge_runtime_steps(fast_path.step_count)` in every arm (fast_path.rs:146,150,154,161,...) and its compiler is mode-gated (function/mod.rs:254), and callback_fast_path.rs:18 checks the mode. docs/architecture.md:82-92 claims Disabled mode routes 'function fast paths ... through their generic semantic fallbacks' and that 'Guard misses must preserve completion, output, and error behavior.'

**Why it matters:** OptimizationMode::Disabled is the project's documented equivalence/diagnosis tool; a specialized path that still runs in Disabled mode makes that tool unsound for exactly the class of bug it exists to isolate. Independently, a step-limited VM diverges: a `return a+b` body costs 0 steps on this path but N steps on the generic interpreter, so RuntimeLimits behavior (a documented error channel) depends on which path fired.

**Verifier notes:** Every cited fact verified: eval_bytecode_function_fast_path (fast_path.rs:243-313) matches 7 body shapes and returns completions with no charge_runtime_steps()/step() call in any arm or helper; its sole call site (parameters.rs:287-291) gates only on current_with_environments().is_empty(), and an exhaustive search of all optional_optimizations_enabled() call sites shows none on this call chain — in Disabled mode the mode-gated pre-setup path (mod.rs:254, pre_setup.rs:47) is inert, so every matching call runs this ungated specialization. The generic interpreter charges one step per instruction (execution.rs:367, 434), and the sibling pre-setup path charges fast_path.step_count in all arms, so the zero-step accounting is a genuine divergence in the documented runtime-steps limit error channel, contradicting docs/architecture.md:82-92 which says Disabled routes function fast paths through generic semantic fallbacks. Severity high is defensible given the project's explicit architectural invariant, though user-visible impact is limited to step-limit behavior and Disabled-mode diagnostic soundness (values/completions are otherwise intended to be identical).

**Verifier correction:** Second after-setup matcher eval_bytecode_function_fast_path (src/runtime/function/fast_path.rs:243) is reachable in OptimizationMode::Disabled (only gate is the with-environment check at parameters.rs:287) and charges zero runtime steps, unlike the mode-gated, step-charging pre-setup path; this violates the documented Disabled-mode contract (docs/architecture.md:82-92) and makes step-limit errors path-dependent.

**Suggestion:** Gate eval_bytecode_function_fast_path behind optional_optimizations_enabled() and charge bytecode.body().instructions().len() steps in every arm, mirroring the pre-setup path; or delete this second matcher and let the pre-setup FunctionFastPath be the single owner.

### [HIGH] Exponentiation uses bare f64::powf in five places, diverging from ECMAScript and the QuickJS reference for |base|==1 with non-finite exponent

- **Kind:** `risk` · **Location:** `src/runtime/numeric.rs:81` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed (corrected severity: medium)

**Evidence:** Five independent evaluators compute Pow as `left.powf(right)` with no special-casing: numeric.rs:81 (apply_number_binary, the generic owner), bytecode/ops/mod.rs:212, function/fast_path.rs:479, bytecode/linear/numeric_chain.rs:572, native/builtins/array/flatten.rs:373. IEEE-754 pow (which Rust powf follows) returns 1 for pow(±1, ±Inf) and pow(1, NaN); ECMAScript Number::exponentiate requires NaN for both. QuickJS's js_pow contains an explicit `!isfinite(exp) && fabs(base)==1 -> NaN` branch commented 'not compatible with IEEE 754'; no equivalent branch exists anywhere in this repo (grep for is_nan/is_finite near the powf sites is empty).

**Why it matters:** `(-1) ** Infinity` and `1 ** NaN` return 1 instead of NaN on every execution tier, so the bug is self-consistent and invisible to the optimizer-on/off equivalence tests the project relies on; only Test262 exponentiation edge-case variants or QuickJS differential runs would catch it.

**Verifier notes:** All five cited sites compute operator Pow as bare `left.powf(right)` with no NaN/infinity guard: src/runtime/numeric.rs:81 (apply_number_binary, the generic owner reached via numeric_binary), src/runtime/bytecode/ops/mod.rs:212, src/runtime/function/fast_path.rs:479, src/runtime/bytecode/linear/numeric_chain.rs:572, and src/runtime/native/builtins/array/flatten.rs:373; no constant-folding path or upstream guard intercepts these, and no test exercises `** Infinity` or `** NaN` (tests/engine_cases/exponentiation_parentheses.js contains neither token). Decisively, the repo already implements exactly the required special case — src/runtime/native/builtins/math.rs:612-617 `math_pow` returns NaN when `exponent.is_nan() || (is_exact_abs_one(base) && exponent.is_infinite())` — but it is reachable only via Math.pow native dispatch (native/function/direct.rs:314, 780), never via the `**` operator. Since ECMAScript defines both Math.pow and `**` in terms of the same Number::exponentiate, the engine is observably self-inconsistent: `Math.pow(-1, Infinity)` yields NaN while `(-1) ** Infinity` yields 1, and `1 ** NaN` yields 1 instead of NaN on every execution tier. The IEEE-754/C99 pow behavior attributed to Rust powf (pow(±1, ±inf)=1, pow(+1, NaN)=1) and the QuickJS js_pow reference branch are both accurately described.

**Verifier correction:** The defect is real, present on all tiers, and even worse than claimed in one respect: Math.pow already has the correct guard (math.rs:612-617), so Math.pow(x, y) !== x ** y is directly observable, proving the deviation was known for the builtin but never applied to the operator. However, severity is better rated medium than high: the affected input domain is extremely narrow (|base| exactly 1 with non-finite or NaN exponent), there is no crash, unsoundness, or data-corruption risk, and real-world code virtually never hits it — the practical impact is Test262/QuickJS-differential conformance failures. The natural fix is to route all five operator sites through the existing guarded math_pow helper (or a shared free function).

**Suggestion:** Add one shared `number_exponentiate(f64, f64) -> f64` in the abstract-operations conversion/numeric owner with the |base|==1 && !exp.is_finite() -> NaN case, and make all five sites (ideally collapsed per the duplication finding) call it.

### [MEDIUM] Numeric binary-op kernel duplicated across five evaluators outside the semantic-duplicate allowlist

- **Kind:** `inelegant` · **Location:** `src/runtime/bytecode/ops/mod.rs:197` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** The same Add/Sub/Mul/Div/Rem/Pow match block exists in numeric.rs:75-93 (apply_number_binary), bytecode/ops/mod.rs:206-237 (eval_bytecode_number_binary), function/fast_path.rs:464-488 (fast_number_binary), linear/numeric_chain.rs:~572, and array/flatten.rs:359-382 (eval_flat_map_array_number_binary). The copies already differ: ops/mod.rs implements the bitwise/shift ops inline, fast_path.rs and flatten.rs bail on them; flatten.rs's copy skips the checked_value() call the others make. The check-architecture-boundaries.sh 'semantic duplicates' allowlists freeze equality/conversion/length owners but have no allowlist for this numeric kernel, so nothing prevents a sixth copy.

**Why it matters:** Five copies of one arithmetic kernel is exactly the 'specialization that duplicates semantics rather than layering over it' failure mode the stabilization plan forbids (architecture-stabilization-plan.md rule 7 area); the powf divergence above is already replicated 5x, and any future fix or BigInt/quickening change must be applied five times or the tiers silently disagree.

**Verifier notes:** All five copies exist exactly as cited (numeric.rs:75-113, ops/mod.rs:197-241, fast_path.rs:464-488, numeric_chain.rs:561-593, flatten.rs:359-382), left.powf(right) is replicated in all five, and the check-architecture-boundaries.sh semantic-duplicate allowlists (equality, conversion, invocation, semantic-object, indexing, property-call, iterators) indeed have no entry for this arithmetic kernel — unlike equality, where fast paths must delegate to the shared number_strict_equality owner. However, the claim's harm evidence is substantially overstated: the "copies already differ" divergences are behaviorally inert (fast_path.rs/flatten.rs bailing on bitwise ops is the sanctioned guard-miss fallback to the generic path per stabilization rule 4, and checked_value is a provable no-op for Value::Number per values.rs:235-265); the "BigInt change applied five times" argument is wrong because all five copies guard on Value::Number and fall through to numeric_binary (numeric.rs:21-42), which already owns BigInt in one place; and new-op additions cannot silently diverge because the matches are exhaustive, so the compiler forces edits at every site. The residual genuine hazard is narrow: a semantic edit to an existing arm (realistically only Pow, whose IEEE powf behavior diverges from ES Number::exponentiate at ±1**±Infinity and 1**NaN) must be applied five times, partially mitigated by the documented optimizer-on/off equivalence coverage.

**Verifier correction:** Five f64 arithmetic kernel copies exist as cited and lack the allowlist protection given to the equality/conversion kernels, so extracting a shared f64 kernel (as was done for number_strict_equality) is a legitimate cleanup. But the current copies do not observably disagree: op-subset bailouts are sanctioned guard-declines to the generic path, checked_value is a no-op for numbers, bitwise conversions already delegate to shared number_to_i32/number_shift_count helpers, BigInt is owned solely by the generic numeric_binary, and exhaustive matches make new-op divergence a compile error. The real exposure is limited to divergent future edits of existing arms — essentially the shared powf spec deviation — tempered by optimizer-on/off equivalence tests.

**Suggestion:** Extract one const-inlineable free function for the Number-only binary kernel (and one for compare/equality, which callback_fast_path already partially shares via number_strict_equality) and have all tiers call it; add it to the boundary script's duplicate-owner allowlists.

### [MEDIUM] Array.prototype.flatMap packed-numeric fast path: no optimizer-mode gate, private mini-evaluator, and double step-charging on mid-run decline

- **Kind:** `microopt-debt` · **Location:** `src/runtime/native/builtins/array/flatten.rs:98` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** eval_direct_array_flat_map (flatten.rs:52-60) unconditionally tries eval_packed_numeric_array_flat_map; grep for optional_optimizations_enabled in flatten.rs is empty (callback_fast_path.rs:18 has the gate this file lacks). The path compiles the callback's bytecode into a FlatMapArraySource tree (lines 260-328) evaluated by a private eval_flat_map_array_number_binary (line 359) instead of the shared eval_bytecode_number_binary. Inside the per-element loop it charges `self.step()` and `charge_runtime_steps(pattern.step_count)` (lines 110-111), then on any non-Number source result it `return Ok(None)` (line 116), after which the caller re-runs the full generic flatMap and charges all steps again.

**Why it matters:** Three violations of the project's own fast-path contract in one site: Disabled mode does not decline it (inventory line 597 claims direct-native shortcuts decline in disabled mode), it duplicates semantics instead of layering (its own powf copy), and the decline path is not step-neutral, so a step-limited program can fail only when the fast path partially ran — e.g. `[1,2,3].flatMap((x,i,a) => [a*2])` charges the fast iterations, bails on the Object operand, then recharges generically.

**Verifier notes:** The code facts check out: flatten.rs:58 unconditionally tries the packed-numeric flatMap path with no optional_optimizations_enabled gate, it evaluates a compiled expression tree via a private eval_flat_map_array_number_binary (line 359), and lines 110-116 charge steps before a possible Ok(None) decline that the caller recharges generically. However, the project's own inventory (docs/semantic-architecture-inventory.md, AS-08b bullets below line 597) explicitly sanctions this exact path as a retained operation-shaped specialization ("flat-map array construction compiles a general parameter/literal numeric expression tree... both retain ordinary callback fallbacks"), and sort.rs's comparator recognition is equally ungated, so the "violates the fast-path contract" leg largely dissolves. The shared eval_bytecode_number_binary is pub(super) inside runtime/bytecode (inaccessible from runtime/native), the sanctioned gated callback fast path has the identical private powf match (function/fast_path.rs:464-488), and the generic path (numeric.rs:81) also uses powf, so the duplication carries no divergence risk. The step leak is real but iteration-invariant — a declining pattern always declines at element 0 because arg kinds (Number, Number, Object) never change — so the over-charge is O(1) (1 step + the tiny callback's instruction count), not the O(n) "fast iterations" the claim implies.

**Verifier correction:** Array.prototype.flatMap packed-numeric fast path (flatten.rs): the ungated status and private number-binary evaluator match the documented AS-08b contract (inventory explicitly retains flat-map tree compilation and sort comparator recognition as operation-shaped, guard-fallback specializations; the gated shared eval_bytecode_number_binary is pub(super) and unreachable from runtime/native, and the sanctioned callback fast path in function/fast_path.rs:464-488 uses the same private powf match with identical arithmetic to the generic numeric.rs:81). Two residual nits: (1) the decline path charges one iteration's steps (self.step() + pattern.step_count at lines 110-111) before returning Ok(None) at line 116, after which the generic path recharges — but the decline is iteration-invariant and always occurs at element 0, so the leak is O(1); charging after successful completion (as sibling eval_packed_array_flat does at lines 175-178) would make it step-neutral; (2) a pattern referencing the array parameter inside an arithmetic op (e.g. (x,i,a)=>[a*2]) compiles but can never succeed at runtime — rejecting it in compile_flat_map_array_callback would eliminate the leak entirely. Minor tension remains with architecture.md's statement that disabled mode routes "function fast paths" through generic fallbacks, since flatMap's callback tree still runs in disabled mode while map/filter's gated eval_pure_function_callback_fast_path does not; worth a one-line doc or gate decision, not a medium defect.

**Suggestion:** Gate on optional_optimizations_enabled(); validate all sources are Number-producing at compile time (reject any tree containing Param(2)/non-numeric operands) so the run can never decline mid-loop; route arithmetic through the shared numeric kernel.

### [MEDIUM] Benchmark-idiom superinstructions beyond the one documented reduction plan: masked-index accumulate, masked-index `in`, and literal-throw try bodies

- **Kind:** `microopt-debt` · **Location:** `src/runtime/bytecode/linear/mod.rs:310` · **Status in project docs:** documented (AS-plan)
- **Verification verdict:** confirmed

**Evidence:** compile_add_array_element_to_binding_with_mask (linear/mod.rs:310-339) recognizes the exact 9-instruction window for `t = t + a[i & MASK]`; compile_in_array_index_mask_binding (linear/in_operator.rs:26-60) recognizes the exact 5-instruction window for `(i & MASK) in a`; bytecode/fast_path.rs:16-31 (BytecodeDirectThrow) recognizes try-blocks whose first two instructions are `Push<literal>; Complete(Throw)` and stores that on the Try instruction (body_direct_throw, control.rs:127-134). docs/architecture.md:99-102 presents the packed-numeric-array reduction as 'One broad linear reduction plan' and the inventory (line 616) classifies retained specializations as 'operation-shaped rather than source-shaped'.

**Why it matters:** A bit-masked array-index accumulate and a bit-masked `in` probe are not general operations — they are the shape of one benchmark kernel (crypto/hash-table style loops). Each such window is another hand-proved guard set (binding aliasing, builtin shadowing, cell materialization) that must be re-verified whenever binding or array semantics change; the module-level allowlist freezes the files but not the pattern count inside them, so this shape census only exists in code.

**Verifier notes:** All three recognizers exist exactly as cited: a 9-instruction window for `t = t + a[i & MASK]` (linear/mod.rs:310-339), a 5-instruction window for `(i & MASK) in a` (linear/in_operator.rs:26-60), and a Push+Complete(Throw) try-body recognizer (bytecode/fast_path.rs:16-31) stored on the Try instruction (types.rs:772, runtime control.rs:127-143). The docs advertise only "One broad linear reduction plan" (architecture.md:98-102) and AS-08b evidence claims "one reusable operation earned replacement" / "retained only reusable bytecode operations" (stabilization-plan.md:2206, inventory.md:594), while none of these three idiom windows is documented anywhere; the architecture guard (check-architecture-boundaries.sh:1901-1922) freezes only the file sets, not intra-file pattern counts, and no focused tests for any of the three windows exist in tests/ or in-source unit tests. The core defect — undocumented benchmark-idiom shape census living only in code behind a file-level allowlist — is accurate.

**Verifier correction:** Three qualifications temper the framing without changing the verdict. (1) The "operation-shaped rather than source-shaped" quote (inventory line 616/597) classifies dense-array/native built-in paths, not linear superinstructions; the applicable inventory claim for these files is line 594's "AS-08b retained only reusable bytecode operations", which the masked-index windows still sit in tension with. (2) "Another hand-proved guard set" is overstated: the fused evals compose the same generic helpers as the unfused ops (eval_bytecode_number_binary, eval_dynamic_array_index_member with generic property fallback, assign_bytecode_cell), so the per-window re-verification burden is fusion equivalence (evaluation order, elided stack states, resume/step accounting) rather than duplicated semantics, and no builtin-shadowing guard exists because no builtin is involved. (3) The direct-throw recognizer is broader than "literal-throw" — it accepts any effect-free push (Literal/String/Undefined), charges two resource steps matching the replaced instructions, and applies only to unscoped, non-resuming bodies, making it the most operation-like of the three; the two masked-index windows are the strongest examples of benchmark-kernel shape. Severity medium is fair: no correctness defect (guard misses fall back to structured execution), but three undocumented exact-window recognizers with no focused tests contradict the documented one-broad-plan posture the guards were built to enforce.

**Suggestion:** Either generalize (a windowed expression-tree fuser like the flatMap one, applied uniformly) or record each window's motivating workload in the inventory so a future cleanup can measure and delete the ones whose benchmark no longer matters.

### [MEDIUM] The boundary guard pins ~150 exact source-text literals and cannot even run on the development platform

Also note in fairness: the enum-variant, struct-field, and file-list allowlists are implemented as normalized structural checks (awk body extraction, compare_set) consistent with the inventory's stated preference, and the no-growth freeze is documented intentional policy; the finding applies specifically to the ~150 verbatim grep -F statement literals and the exact block.clone() count, which pin incidental token shape (variable names, loop-header style, ?; placement) beyond the documented invariants.

**Suggestion:** Replace verbatim-line pins with structural checks (a tiny Rust xtask using syn to assert enum variant sets, struct fields, and function-owner locations), keep the genuinely behavioral greps, and swap find -printf/sed -i for portable equivalents.

## Optimization opportunities (post-cleanup)

### [MEDIUM] Linear plans and the reduction plan are recompiled from bytecode on every loop entry

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/bytecode/control.rs:282` · **Status in project docs:** UNDOCUMENTED

**Evidence:** eval_bytecode_while calls compile_bytecode_linear_plan(condition) and (body) at lines 282-283 each time the While instruction executes; eval_bytecode_for_loop compiles the reduction plan (459-465) and three linear plans (607-627) per loop entry. BytecodeLinearPlan<'a> borrows the block and captures BindingCells, so it structurally cannot outlive one execution — a function called N times pattern-matches its loop bytecode N times (compile_bytecode_linear_peephole tries ~12 recognizers per instruction, linear/mod.rs:125-185).

**Why it matters:** The dominant cost of the whole quickening layer is paid per call instead of once; for short hot loops the recognizer scan can rival the loop body. This is a direct consequence of doing quickening in the runtime over borrowed blocks rather than as a compiler pass.

**Suggestion:** Now that AS-06/AS-08 are stable, lower the linear ops at compile time into the (already immutable, Rc-shared) BytecodeBlock as a parallel op table — cells resolve at entry like today, but pattern recognition happens once per program. Alternatively cache plans keyed by Rc pointer identity of the block in the Optimizer owner with the existing identity-cache invalidation hook (gc.rs invalidate_identity_caches).

### [MEDIUM] Every binding is an Rc<Mutex<Binding>> heap cell; single-threaded VM pays a lock per local variable access

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/binding/scope.rs:530` · **Status in project docs:** UNDOCUMENTED

**Evidence:** `pub struct BindingCell(Rc<Mutex<Binding>>);` (scope.rs:530). All execution tiers read locals via cell.value()/cell.assign() (linear/mod.rs:526-571, fast_path.rs:365, numeric_array_reduction.rs:104-110), each taking a parking_lot lock on a cell that is provably single-threaded (it sits under Rc, which is !Send). RetainedValueRegistry (Rc<Mutex<RetainedValueState>>) and TransientRootRegistry (Rc<Mutex<TransientRootState>>, transient_roots.rs:15) repeat the pattern.

**Why it matters:** A lock plus a per-binding heap allocation on the hottest operation in an interpreter, with the Rc/Mutex combination itself signalling a confused threading story (the Mutex buys nothing under Rc except panic-free interior mutability). It also forces every fast path to capture and validate cells rather than indexing a frame.

**Suggestion:** With the activation stack now the single call-state owner (AS-06a1), store non-captured locals as slots in the ActivationFrame (plain Vec<Value> indexed by the existing BindingOperand::Local slot) and reserve heap cells for captured/upvalue bindings only; replace remaining Rc<Mutex<_>> with a no-unwrap Cell/try-RefCell wrapper.

### [MEDIUM] TransientRootScope::drop scans the entire live root vector; nested scopes make unwinding quadratic

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/transient_roots.rs:179` · **Status in project docs:** UNDOCUMENTED

**Evidence:** Drop for TransientRootScope runs `state.roots.retain(|root| root.scope != self.scope)` (transient_roots.rs:179) — O(all live transient roots) per scope exit. Scopes are created per operand window, per call (invocation.rs, 2 sites), per iterator (iterator.rs), and per descriptor/Proxy temporary (object.rs, proxy.rs), so a recursion of depth d with k roots per frame costs O(d*k) per unwind step, O(d^2*k) total. The retain is even pinned verbatim in the boundary script (script:1205).

**Why it matters:** Accidental O(n^2) on the call-unwind path, and every add_values/drop also takes the Mutex; deep recursion or iterator-heavy code pays it constantly, not just at GC safepoints.

**Suggestion:** Scopes are created and dropped LIFO even though their roots interleave; store per-scope Vec<Value> buckets in a stack (drop = pop + ledger release, O(own roots)), or record each scope's first index and swap_remove from a per-scope index list.

- **Array-like length semantics split across seven allowlisted functions in three files** (`src/runtime/native/builtins/array/generic.rs`) — The allowlist and line numbers are real, but the architectural indictment fails on all three load-bearing points.
