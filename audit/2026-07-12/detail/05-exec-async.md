# Execution model: activations, async, generators, promises, bindings

> Detailed architecture-audit report for Velum. Subsystem: `exec-async`. Generated by a multi-agent audit on 2026-07-12; every medium/high finding was adversarially verified by an independent agent reading the cited code.

## Subsystem assessment

The runtime core is a bytecode-first, safe-Rust VM whose call state genuinely has one owner: Context.activation_frames plus a single flat locals stack, with await/generator suspension implemented by physically moving frame suffixes into DetachedFunctionExecution values held by promise reactions or generator records — a sound single-ownership design, but one whose park/unpark seams (detach-by-FunctionId search, rebase-only-first-frame, dual private-environment tracking, super-this in shared Rc cells) are held together by language-level early-error rules and reviewer discipline rather than types. Layering is otherwise strong and audited: semantic abstract operations (AS-02/03) are the single owners of property/call/construct behavior, every optimization (call caches, static binding locations, function fast paths) is gated behind one optimizer mode with generic fallbacks, and the storage ledger makes resource accounting a checked invariant. The weakest abstractions are environmental: globals live in two stores (BindingScope cells and lazily materialized global-object properties) reconciled by name-special-cased sync functions; the flat scope stack has no variable-environment/lexical-environment split, so sloppy-eval var hoisting and Annex B assignment target 'the innermost scope' heuristically, and one Annex B walk even crosses activation boundaries; and BindingCell is an Rc<Mutex> whose read path pays lock + clone + cycle-detection allocation on every variable access. Async support did grow parallel structures — six PromiseReaction variants, four hand-rolled native continuation state machines, and duplicated settle-or-repark dispatch between first-call and resume paths — plus an overloaded Completion enum whose ten flattened variants force every consumer to reject impossible states with stringly errors. Kludge count is modest and mostly recorded (Test262Error fallback, object side tables, derived-super specialization are documented allowlisted debt); the most valuable undocumented items are the Box::leak fake-unreachable branch in scope-index escalation, the incomplete shadow guards on unresolved-builtin fast paths, Proxy-tunneled private access, and the run_jobs error path that strands parked-continuation storage. Overall quality is high for a prototype at this scale — invariants are checked loudly at seams and debt is inventoried — but the foundation-level fixes that would unlock the rest are a real environment-record model (varEnv/lexEnv split, global record as one owner) and a typed suspension protocol separated from spec completions; both would delete more special cases than they add.

## Strengths

- Single ownership of call state is real: Context.activation_frames + one flat locals stack (AS-06) is the only owner; detach/reattach for await and generators MOVES the same BindingScope/ActivationFrame values rather than copying, so parked state never has two live owners.
- Discipline around the no-panic rule is near-total: every arithmetic step is checked_add/checked_sub with typed limit errors, every index access is get()-based with explicit 'disappeared' errors, and error paths roll back ledger reservations (push_call_activation, remember_promise_object, allocate_pending_promise_record).
- The storage ledger (AS-05) gives the engine an unusual invariant: every variable-size owner (bindings, frames, reactions, jobs, associations) is counted with reserve/commit/rollback and reconciled at teardown, so accounting drift is detectable rather than silent.
- Optimization layering (AS-08) is genuinely enforced: every fast path (call-value cache, static binding cache, direct native calls, function fast paths) is gated on optional_optimizations_enabled() with a documented disabled-mode-equivalence contract, and guard misses return to the shared semantic operation rather than a second implementation.
- Runtime never touches the AST: the compile-time IR boundary holds — the runtime consumes only bytecode, BindingLayout, and static ids, and an architecture guard rejects parser/AST imports under src/runtime.
- Self-documentation of debt is exceptional: the inventory and stabilization plan record the Test262Error fallback, object side tables, derived-super specialization, global-binding layer split, and per-item guards that fail on growth — most engines carry this debt silently.
- Invariant checks at ownership seams are defensive and specific: pop_call_activation verifies frame kind, local_base, scope-stack height, and settled continuations; resume_function_execution errors on parked child activations; module evaluation depth is checked in both directions.

## Confirmed findings (adversarially verified)

### [HIGH] assign_annex_b_var scans local scopes across activation-frame boundaries

- **Kind:** `risk` · **Location:** `src/runtime/binding/declaration.rs:57` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** let cell = self.locals.iter().rev().filter_map(|scope| scope.get(atom)).find(|cell| cell.kind() == DeclKind::Var)... — no `.skip(self.current_local_frame_start())`, unlike get_binding_by_atom (line 365-372) and every other lookup in the file, which clamp to the active frame.

**Why it matters:** Every other local-scope walk in the runtime respects the current activation's local_base so one function cannot see a caller's locals. This one walk can reach a caller's (or any suspended ancestor's) `var` cell with the same name. If the Annex B hoisted var is ever absent in the current frame (the hoist has legitimate skip conditions around parameter/lexical conflicts), the block-scoped function name is assigned into another activation's variable — cross-frame state corruption with no error.

**Verifier notes:** The cited code shape is accurate: assign_annex_b_var (src/runtime/binding/declaration.rs:57-62) walks all of self.locals without the .skip(current_local_frame_start()) clamp used by get_binding_by_atom and every other lookup, and caller locals are physically reachable. However, the corruption scenario is unreachable: StoreAnnexBVar is only emitted when the annex-B variable binding resolves in the layout (compile_annex_b_function_update returns early on BindingOperand::Unresolved, src/compiler/function.rs:204-207), and that same non-Unresolved condition on the same layout puts the binding into the function's hoist plan (src/compiler/hoist.rs:240-252), which every entry path executes before body bytecode (parameters.rs:230/252/271, script_execution.rs:129/137/147), creating a DeclKind::Var cell in the current frame (or erroring before the body runs). The auditor's premise — that the hoist's skip conditions (lexical conflicts, binding_layout/builder/annex_b.rs:149-158) can fire while the store instruction still executes — is wrong, because those skip conditions make the operand Unresolved, which suppresses emission of the store instruction itself. The innermost-first walk therefore always finds the current frame's Var cell before crossing an activation boundary; what remains is a defense-in-depth inconsistency, not a reachable defect.

**Verifier correction:** assign_annex_b_var lacks the .skip(self.current_local_frame_start()) frame clamp that every sibling lookup in declaration.rs and static_binding_lookup.rs applies. This is currently unexploitable: StoreAnnexBVar emission and the runtime var hoist are gated by the identical layout-resolution condition, so the target Var cell always exists in the current frame when the instruction runs. Adding the clamp is worthwhile hardening — it makes this lookup consistent with the rest of the file and converts a hypothetical future invariant break from silent cross-frame assignment into the existing "Annex B var binding is missing" runtime error — but it is not a live cross-frame corruption bug.

**Suggestion:** Add .skip(self.current_local_frame_start()) like every sibling lookup, and keep the existing 'binding is missing' error as the fallback.

### [HIGH] Global variables have two owners (BindingScope globals/builtin_globals vs global object properties) reconciled by name-special-cased sync functions

- **Kind:** `inelegant` · **Location:** `src/runtime/globals.rs:344` · **Status in project docs:** documented (AS-plan)
- **Verification verdict:** confirmed

**Evidence:** sync_global_object_binding_property (only syncs when name == GLOBAL_THIS_NAME), sync_global_object_property_binding (only DeclKind::Var cells or globalThis), global_object_property_descriptor hardcodes NaN/Infinity writability by string match (lines 317-325) and SIDE-EFFECTS a define_property on descriptor read (line 333); with_environment.rs:60-65 and static_bindings.rs:370-386 add two more manual sync call sites; declaration.rs hoist_function (212-221) writes both stores.

**Why it matters:** In the spec the global environment record IS the global object; here the same logical binding lives in a BindingScope cell and, lazily, as an object property, kept coherent by four hand-written sync functions triggered from scattered call sites and keyed on hardcoded names. Any path that updates one store without the ritual desynchronizes them. Concrete plausible failure: `globalThis.parseInt = f` defines only the object property; a later bare `parseInt` resolves through builtin_globals (static_binding_lookup.rs builtin_global_binding_at_location checks globals/locals but never global-object own properties), still calling the builtin. Similarly unresolved_binding_name_is_shadowed (static_binding_lookup.rs:243-251) never consults global-object properties before the direct-builtin fast paths fire.

**Verifier notes:** Every cited detail matches the code: globals.rs:344/365 sync functions are name-special-cased exactly as described (globalThis-only, or Var-cells-plus-globalThis), globals.rs:317-340 hardcodes NaN/Infinity attributes by string match and performs a define_property side effect during a descriptor read, and the sync ritual is invoked from even more scattered sites than claimed (6 in property/static_names/mod.rs, 2 in bytecode/ops/assignment.rs, plus with_environment.rs:64, static_bindings.rs:367-386, typed_array.rs:186, and hoist_function writing both stores at declaration.rs:211-221). The concrete failure trace holds: globalThis.parseInt = f only defines the object own property (sync_global_object_property_binding no-ops for non-Var, non-globalThis names), while bare-name resolution (eval_bytecode_identifier at bytecode/ops/mod.rs:81-97 and the call path at runtime/mod.rs:507-528) consults binding cells and builtin_value() before falling back to global-object own properties, and the guards (unresolved_binding_name_is_shadowed at static_binding_lookup.rs:243-251, builtin_global_binding_at_location at :387-399) check only globals/locals, never object properties — so the stale builtin is returned. No guard, invalidation, documented invariant, or test covers this direction; existing tests only exercise the read direction and non-builtin names, so severity=high is fair for an architectural defect with an observable spec divergence (e.g. the common globalThis.Date = MockDate mocking pattern silently fails).

**Verifier correction:** Confirmed, and the claim if anything understates the scatter: sync_global_object_property_binding has 9 call sites (6 in property/static_names/mod.rs, 2 in bytecode/ops/assignment.rs, 1 in with_environment.rs) plus sync_global_object_binding_property call sites in static_bindings.rs and typed_array.rs. The desync is observable both for overwrite (globalThis.parseInt = f; parseInt still calls the builtin via builtin_value/direct-builtin fast paths, which run before unresolved_global_property_value) and for delete (delete globalThis.parseInt; bare parseInt should throw ReferenceError but still resolves to the builtin cell). No test or doc covers the write/delete direction; tests/global_this_smoke.rs only asserts the read direction (globalThis.parseInt === parseInt) and fresh non-builtin names.

**Suggestion:** Introduce one GlobalEnvironmentRecord owner whose var/function bindings are the object's properties (spec shape), with the interned-cell representation as a cache keyed off the object's shape version rather than a co-equal store; delete the per-name sync functions.

### [MEDIUM] Box::leak 'unreachable' branch in ScopeIndex::make_owned to satisfy the no-panic rule

- **Kind:** `hack` · **Location:** `src/runtime/binding/scope.rs:93` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** fn unreachable_owned() -> (&'static mut Vec<AtomId>, &'static mut Vec<BindingEntry>) { ... (Box::leak(Box::new(Vec::new())), Box::leak(Box::new(Vec::new()))) } — called from the `Self::Shared(_) => unreachable_owned()` arm of make_owned() (line 87) after the Shared arm was rewritten to Owned.

**Why it matters:** The no-unwrap/no-panic rule was satisfied by returning leaked, disconnected empty vectors instead of an error. Today the branch is structurally unreachable, but if a refactor ever makes it reachable, binding inserts would silently write into leaked throwaway vectors — bindings would vanish with no error, the worst possible failure mode for an engine. It also leaks memory on every hypothetical call. The pattern trades a loud impossible panic for silent wrong behavior.

**Verifier notes:** The code at src/runtime/binding/scope.rs:91-100 is exactly as claimed: unreachable_owned() returns two Box::leak'd empty Vecs and is called from the Shared arm of make_owned() (line 86), which is structurally unreachable only because lines 75-80 rewrite Shared to Owned first; the doc comment explicitly says it exists to keep the match exhaustive "without panicking paths", and Cargo.toml denies clippy::panic/unwrap_used/expect_used, confirming the no-panic-rule motivation. The hazard analysis holds: all three call sites (insert_or_replace line 286, insert_new_at_slot line 358, try_insert_new_at_slot line 401) already return Result, so an Err fallback was trivially available, and each pushes the binding cell into self.slots and charges the storage ledger before mutating the returned vectors — so a reachable branch would commit the slot while the atom index updates vanish into leaked vectors, silently making the binding unfindable. No guard or invariant beyond the in-function rewrite exists; medium severity is fair for a deliberate silent-wrong-behavior fallback in an engine, though the auditor correctly notes it is unreachable today.

**Verifier correction:** Fix is cheap: make make_owned() return Result and replace the Shared arm with Err(Error::runtime("scope index remained shared after copy-on-write")) — all three callers already propagate Result, eliminating both the leak and the silent-loss path.

**Suggestion:** Restructure make_owned so the type system proves the invariant: e.g. `let (a, b) = match self { Shared(d) => (d.slot_atoms.to_vec(), d.bindings.to_vec()), Owned{..} => return ... }` or have make_owned return Result and thread the error, eliminating the fake-unreachable arm entirely.

### [MEDIUM] BindingCell = Rc<Mutex<Binding>> with per-read alias-cycle walk; three inconsistent interior-mutability disciplines for call state

- **Kind:** `inelegant` · **Location:** `src/runtime/binding/scope.rs:560` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** BindingCell(Rc<Mutex<Binding>>) (line 530); BindingCell::value() (lines 560-575) allocates `let mut visited = Vec::new()`, pushes a clone of the cell, takes the parking_lot lock, and clones the whole BindingState on EVERY binding read — cycle detection is O(chain^2) with `visited.iter().any(same_cell)`. Meanwhile FunctionSuperBinding uses RefCell<Option<Value>> + Cell<bool> (class_support.rs:17-18) and Function uses Cell<bool> params_remembered (mod.rs:167).

**Why it matters:** A single-threaded VM (Rc is !Send) pays a Mutex lock, an Rc clone, a state clone, and a heap allocation on the hottest operation in the engine — reading a variable — purely to defend against alias cycles that only module `import` linking (alias_to, the sole Alias creator) can create, and are already rejected there via same_cell/already-linked checks. Re-entrant lock acquisition would deadlock rather than panic, which is strictly worse than RefCell's panic for debugging. Three different interior-mutability idioms (Mutex, RefCell, Cell) for adjacent concepts means no single discipline can be audited.

**Verifier notes:** scope.rs:530/560-575 match the claim exactly: every binding read clones the cell, heap-allocates via the first visited.push, takes a parking_lot Mutex lock (useless for thread safety since Rc is !Send), and clones BindingState, with O(chain^2) same_cell scanning. This is verifiably the interpreter's variable-read path (LoadBinding at bytecode/linear/mod.rs:526 plus ~44 sibling sites, including fast_path.rs). The cited three-idiom inconsistency (Mutex vs RefCell vs Cell) exists at class_support.rs:17-18 and function/mod.rs:237. The only inaccuracy is mechanism attribution: alias_to's same_cell check alone would not prevent 2-cycles; cycles are actually impossible because the compiler rewrites exports-of-imports to Indirect (compiled_script/mod.rs:155-172) and resolve_export (module.rs:413-505) flattens chains with its own cycle-detecting resolving set, so every alias target is a terminal non-import cell — alias chains are at most 1 hop, which makes the runtime cycle walk defend against an unconstructible state, strengthening the claim's conclusion.

**Verifier correction:** One evidence correction: cycles are not prevented by alias_to's same_cell/already-linked checks (those only block 1-cycles); they are prevented because compile_module_named rewrites exports-of-imports to Indirect exports and resolve_export flattens indirect chains under a cycle-detecting resolving set, so alias_to targets are always terminal non-import cells and alias chains are at most 1 hop. This makes the per-read visited-Vec walk doubly dead: not only are cycles impossible, chains longer than one hop are too. Minor nits: Vec::new() itself does not allocate (the first push does, still on every read), and the BindingState clone is mostly the Value clone the caller needs anyway. The deadlock concern is theoretical — all lock scopes in scope.rs are tight with no re-entrant calls under the lock.

**Suggestion:** Fast-path the Initialized case: lock once, if Initialized clone the Value and return — allocate the `visited` vector only after the first Alias hop. Better: flatten alias chains at module-link time (aliases are immutable after linking), removing BindingState::Alias from the read path entirely; then Binding can become a plain enum behind one chosen cell type.

### [MEDIUM] Completion is one flattened 10-variant enum spanning statement control, function control, and suspension protocol, with four hand-written projections

- **Kind:** `inelegant` · **Location:** `src/runtime/control/completion.rs:9` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** enum Completion { Normal, Throw, Return, ReturnDirect, Break{..}, Continue{..}, Suspended(PromiseId), GeneratorStart, Yielded, YieldedIteratorResult } plus into_result / into_function_result / into_call_completion / into_native_value_result, each enumerating all 10 arms with slightly different mappings. Consumers repeat the full 10-arm rejection dance: module.rs:579-611, promise/mod.rs:224-257 and 675-709, mod.rs:670-693, generator.rs:483-522.

**Why it matters:** Invalid states are representable everywhere: a module evaluation can 'receive' GeneratorStart, an async resume can 'receive' Break, and each site guards this with stringly runtime errors instead of types. Every new completion variant (the suspension family was added for AS-06) forced edits at every consumer, and the Return vs ReturnDirect and Yielded vs YieldedIteratorResult pairs encode caller-specific protocol details in the shared type. This is the classic flattened-union smell where layered types (AbruptCompletion vs SuspensionSignal vs LoopSignal) would make impossible states unrepresentable.

**Verifier notes:** The code matches the claim exactly: Completion at src/runtime/control/completion.rs:9-26 has the 10 variants described, and the four projections (into_result/into_function_result/into_call_completion/into_native_value_result, lines 39-110) are near-duplicate exhaustive matches differing only in a couple of arms and owner-specific error strings. All five cited consumer sites (module.rs:579-611, promise/mod.rs:224-257 and 675-709, mod.rs:670-693, generator.rs:483-522) repeat full-enum matches that reject impossible states with stringly runtime errors ("generator completion escaped module", "async function entered generator start", etc.), and grep shows ~25 more sites matching the suspension family. The Return/ReturnDirect and Yielded/YieldedIteratorResult pairs are distinguished only by the async-generator/delegation machinery (async_delegate.rs:34-45, generator.rs:489-502) and or'd together everywhere else, confirming the caller-specific-protocol point; docs/architecture.md corroborates the suspension family arrived with AS-06b.

**Verifier correction:** Confirmed with context the auditor omitted: the Normal/Throw/Return/Break/Continue core mirrors the ECMAScript Completion Record, so that portion of the flattening is spec-aligned; the smell is specifically the suspension family (Suspended/GeneratorStart/Yielded/YieldedIteratorResult) leaking into every consumer. Partial mitigations exist — Completion::suspends_execution() groups the suspension family, and BytecodeOutcome (bytecode/execution.rs:14-34) already layers suspended vs completed outcomes at the bytecode boundary — but module.rs:581 flattens BytecodeOutcome back into Completion and re-matches all 10 arms, which reinforces rather than refutes the finding. Also note the exhaustive matches are a deliberate Rust tradeoff (compiler-forced coverage on new variants) and layering would add wrapping in the interpreter hot path, so this is a conscious design cost, not an oversight; severity medium is fair for a pervasive (~30-site) inelegance with no correctness impact.

**Suggestion:** Split into a spec-shaped Completion (Normal/Throw/Return/Break/Continue) and a separate ExecutionOutcome { Completed(Completion) | Suspended(SuspendKind) } returned only by the resumable-execution layer; the four projections collapse into From impls and most consumer match arms disappear.

### [MEDIUM] Completion::YieldedIteratorResult payload is overloaded: real iterator-result object for sync generators, ad-hoc {"0": awaitFlag, "1": value} pair object for async generators

- **Kind:** `hack` · **Location:** `src/runtime/generator/async_generator.rs:505` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** finish_async_generator_completion: `Completion::YieldedIteratorResult(result) => { let await_value = self.get_named(&result, "0")?; let value = self.get_named(&result, "1")?; if await_value == Value::Bool(true) {...} }` — versus the sync path (generator.rs:494-502) which treats the same variant's payload as a genuine iterator result object to hand to the caller.

**Why it matters:** One enum variant carries two structurally different payloads distinguished only by which resume path is executing, and the async encoding smuggles a protocol flag through property reads on a synthesized object (an allocated object plus two property lookups per delegated yield). Nothing in the type system prevents the sync consumer from receiving the async pair encoding or vice versa; a bytecode change to delegated-yield lowering would fail at runtime, not compile time.

**Verifier notes:** Both cited sites match the claim: finish_async_generator_completion (src/runtime/generator/async_generator.rs:505-526) unpacks the YieldedIteratorResult payload via get_named "0"/"1" and branches on the "0" slot being Bool(true), while the sync finish_generator_completion (src/runtime/generator.rs:494-502) treats the same variant's payload as a genuine iterator-result object and returns it directly to the caller after only an is-object check. The two encodings are produced by iterator.rs:503 (real result object) and async_delegate.rs:210-214 (create_array_from_elements([Bool(await_flag), value])), both funneled through the single untyped Completion::YieldedIteratorResult(Value) variant (completion.rs:25) with no doc comment or type-level distinction; the pairing holds only because producer and consumer both key off current_function_is_async_generator() at runtime. No guard would catch a cross-encoding: the pair array is an object so the sync guard passes and it would leak to user code, and reading "0" off a real iterator result yields undefined, so the async path would silently yield undefined rather than error.

**Verifier correction:** Two minor corrections: (1) the sync path is at src/runtime/generator.rs:494-502, not generator/generator.rs; (2) the claim understates the failure mode — a cross-encoding would largely NOT fail at runtime but silently misbehave, since the async pair array passes the sync consumer's is-object guard (it would be handed to user code as the iterator result), and the async consumer reading property "0" off a genuine iterator result gets undefined (!= Bool(true)), silently yielding undefined. **Suggestion:** Add a typed variant (e.g. YieldedDelegated { needs_await: bool, value: Value } or a dedicated struct) instead of encoding the pair as an object with keys "0"/"1".

## Optimization opportunities (post-cleanup)

### [LOW] Flatten module import alias chains at link time to make every binding read a single lock-free slot load

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/binding/scope.rs:634` · **Status in project docs:** UNDOCUMENTED

**Evidence:** alias_to is called only from module linking (module.rs:237 import_cell.alias_to(cell)); after linking, alias targets never change, yet BindingCell::value pays the visited-Vec + chain-walk cost on every read of every binding in the program (lines 560-575).

**Why it matters:** The entire engine's variable-read path carries cycle-detection allocation and chain-walk logic to serve a case (import aliases) that is fully determined and immutable at module link time — a permanent per-read tax paying for a one-time linking concern.

**Suggestion:** Once the read-path cycle walk is removed (see the BindingCell finding), resolve Alias to the terminal cell at link time (the module DAG is fully known; self-alias and re-link are already rejected). Binding collapses to {Uninitialized | Initialized(Value)} and reads become one lock + clone — benefiting every variable access in the engine, not just modules.

### [LOW] Replace object-index-parallel Vec<Option<Id>> side tables (promise/generator/collection) with payload slots or maps sized by live population

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/mod.rs:118` · **Status in project docs:** documented (AS-plan)

**Evidence:** collection_object_slots, generator_object_slots, promise_object_slots are Vec<Option<Id>> resized to the highest bound object index (promise/mod.rs:468-469, generator.rs:280-281, collections.rs:153-154); gc.rs:662-680 and async_trace.rs:242-273 iterate the ENTIRE tables (O(max object id)) every collection/snapshot, and accounting.rs:589-602 counts by full iteration.

**Why it matters:** Memory is O(highest object index ever bound) rather than O(live associations), and every GC pass, async-trace snapshot, and accounting reconciliation scans the full sparse tables including empty slots — cost that grows with total objects allocated, not with promises/generators in flight.

**Suggestion:** The side tables are already an allowlisted debt category ('object side tables' guard). Move the association into an optional slot on the object payload, or a compact map keyed by ObjectId+generation; GC/trace passes then visit only live associations and the stale-slot bug class disappears if object ids are ever recycled.

### [LOW] Cache the active frame's derived facts (frame start, this, new_target, super, upvalues) instead of reverse-scanning activation_frames per access

- **Direct-builtin fast paths for 'unresolved' identifiers guard with an incomplete shadow check** (`src/runtime/binding/static_bindings.rs`) — The cited code shape is real (two match tables, guard checking only `globals` + local scopes), but both legs of the "why bad" reasoning fail.
