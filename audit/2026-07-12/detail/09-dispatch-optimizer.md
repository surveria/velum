# Native dispatch, caches, fast paths, optimizer boundary

> Detailed architecture-audit report for rs-quickjs. Subsystem: `dispatch-optimizer`. Generated by a multi-agent audit on 2026-07-12; every medium/high finding was adversarially verified by an independent agent reading the cited code.

## Subsystem assessment

The optimization layer is genuinely layered in most places: a semantic core (semantic_object internal methods + abstract_operations) is fronted by four optimization families — (1) per-call-site caches (StaticNameAtomCacheHandle: atom, property-lookup, native-call, and call-value slots), (2) a direct-native-call layer (compiler-embedded NativeCallTarget speculation guarded by registry-id equality), (3) loop-local linear plans with peephole superinstructions and numeric quickening, and (4) function-body fast paths (compile-time FunctionFastPath plus a second call-time bytecode matcher). Roughly 9k lines (~6%) of the runtime is optimization machinery, gated by 31 explicit OptimizationMode checks, with 132 eval_direct_* entry points. Guard discipline is mostly excellent: property caches carry receiver+shape+global-prototype-epoch guards, native-call caches carry per-property version guards, registry ids are GC roots, and collect_garbage eagerly wipes identity-bearing caches while object sweep bumps the global lookup epoch — the docs' invalidation claim checks out. The failures are at the edges, and they are exactly the kind the customer fears: exponentiation semantics were never centralized (five powf copies, four spec-incorrect, the only correct one hidden inside Math.pow); one call-time function fast path ignores OptimizationMode entirely and skips step accounting, so Disabled mode does not actually exercise one dispatch surface; Array.prototype.flatMap contains its own unguarded private specializer that re-implements both binding resolution and numeric semantics; and the native-call layer rests on two hand-mirrored 100+ variant enums plus a hand-numbered 330-slot registry whose integrity is maintained purely by author discipline, with one silent catch-all fallback. Net judgment: the optimization layer is trustworthy where AS-08 touched it (caches, linear plans, quickened ops re-enter shared semantics), and untrustworthy where specialization grew inside builtins or predates the isolation program.

## Strengths

- Cache guards are real and validated on every use: CacheablePropertyLookup pins receiver id + shape + a global prototype_lookup_version (object/property/lookup.rs:14-45), native-call caches add a per-property write-version check (lookup.rs:278-317), and misses always fall back to the semantic path.
- GC invalidation is centralized and verifiable: collect_garbage clears all native-call/call-value identity caches before sweeping (gc.rs:588, 623-638), object sweep bumps the global prototype lookup epoch so shape/slot caches cannot survive id reuse (object/base.rs:119), and registered native-function ids are GC roots (roots.rs:268), so the docs' id-reuse claim holds.
- Quickened numeric ops and linear-plan ops re-enter shared semantic evaluators rather than forking them: eval_bytecode_number_binary/compare/equality fall back to the generic binary evaluator for non-number operands (bytecode/ops/mod.rs:197-283), and linear peephole ops call the same assign/update/property helpers as the interpreter.
- OptimizationMode::Disabled is honored at 31 explicit gates including plan compilation (linear/segment.rs:28), cache lookup (static_names/mod.rs:75-79), direct native calls (native/function/direct.rs:34, 66, 152), and builtin-constant shortcuts (binding/static_bindings.rs:310), with an Optimizer counter snapshot making hit/miss/slow-path behavior observable.
- The direct-native speculation is guarded by identity, not name: a compiler-embedded NativeCallTarget only takes effect when the resolved callee's NativeFunctionId equals the registry entry for that kind (direct.rs:417-429), so monkey-patched builtins fall back correctly.
- The project honestly self-documents the optimization architecture: the inventory names caches and direct paths as guarded backends, records the Test262Error construct special case, and AS-08 explicitly owns quickening/inline-cache isolation with a disabled-mode equivalence gate.

## Confirmed findings (adversarially verified)

### [HIGH] Exponentiation semantics duplicated five times; four copies are spec-incorrect (IEEE powf instead of Number::exponentiate)

- **Kind:** `risk` · **Location:** `src/runtime/numeric.rs:81` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** numeric.rs:81 `BinaryOp::Pow => left.powf(right)` (the GENERIC operator path); same raw powf in bytecode/ops/mod.rs:212, function/fast_path.rs:479, bytecode/linear/numeric_chain.rs:572, native/builtins/array/flatten.rs:373. The only spec-correct version is private to Math.pow: math.rs:612-616 `fn math_pow` handles `exponent.is_nan() || (is_exact_abs_one(base) && exponent.is_infinite()) => NAN`.

**Why it matters:** ECMAScript Number::exponentiate differs from IEEE pow: `1 ** NaN` must be NaN (powf returns 1) and `(-1) ** Infinity` must be NaN (powf returns 1). Because the abstract operation was never centralized, every quickened/fast path re-copied the wrong implementation, so `**` disagrees with Math.pow engine-wide — the exact 'specialization duplicates semantics' failure AS-03/AS-08 were meant to prevent.

**Verifier notes:** All five cited sites verified as raw f64::powf with no NaN/±1-infinite-exponent handling: the generic path (numeric.rs:81 via numeric_binary/apply_number_binary) and the four quickened copies (bytecode/ops/mod.rs:212, function/fast_path.rs:479, bytecode/linear/numeric_chain.rs:572, array/flatten.rs:373) guard only on operands being Value::Number, which NaN/Infinity satisfy, so no path bails to a correct implementation — there is none for the ** operator. The only spec-correct version (math.rs:612-616 math_pow, returning NaN for NaN exponents and |base|==1 with infinite exponent) is private and reachable only via Math.pow dispatch. Since Rust powf follows IEEE/C99 pow (pow(1,NaN)=1, pow(-1,inf)=1) while Number::exponentiate mandates NaN, `1 ** NaN` and `(-1) ** Infinity` return 1 engine-wide and disagree with the engine's own Math.pow, exactly as claimed. No compile-time constant folding or upstream normalization mitigates it.

**Verifier correction:** Minor count fix only: the title says "duplicated five times; four copies are spec-incorrect", but the evidence correctly lists five raw powf sites and all five are spec-incorrect (generic path included). Tally: six exponentiation implementations, five wrong, one correct (private to Math.pow). Substance and severity of the finding are unaffected.

**Suggestion:** Create one `number_exponentiate(base, exp)` in runtime/abstract_operations (or numeric.rs) with the Math.pow special cases, and make all six sites (including Math.pow) call it. Failure demo: `print(1 ** NaN)` prints 1, should print NaN.

### [HIGH] Call-time function-body fast path is not gated by OptimizationMode and bypasses step accounting

- **Kind:** `microopt-debt` · **Location:** `src/runtime/function/fast_path.rs:243` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** `eval_bytecode_function_fast_path` (fast_path.rs:243-313) pattern-matches function-body bytecode at every call and executes via fast_function_load_binding/fast_number_binary with no `optional_optimizations_enabled()` check and no `charge_runtime_steps`/`step()` calls. Its only caller guard is `self.current_with_environments().is_empty()` (function/parameters.rs:287-289). Contrast: the compile-time FunctionFastPath IS gated (function/mod.rs:254) and DOES charge steps (fast_path.rs:146).

**Why it matters:** OptimizationMode::Disabled is the project's own verification instrument (AS-08 'disabled-mode equivalence'), but this dispatch surface runs identically in both modes, so disabled-mode testing cannot detect divergence in it. It also duplicates the same instruction patterns as the compile-time FunctionFastPath (two parallel matchers to keep in sync) and executes matched bodies with zero runtime-step charge, weakening the resource-accounting contract.

**Verifier notes:** Every cited fact verified: eval_bytecode_function_fast_path (fast_path.rs:243-313) re-matches body bytecode per call with zero charge_runtime_steps/step() calls (all charge sites enumerated repo-wide; none on this path), its sole caller guard is current_with_environments().is_empty() at parameters.rs:287-289 on the universal body-eval path with no optional_optimizations_enabled() gate anywhere upstream, while the compile-time FunctionFastPath is gated (mod.rs:254) and charges step_count in every arm (fast_path.rs:146-202) and the normal interpreter charges step() per instruction (execution.rs:367, 434). The claim is actually understated: docs/architecture-stabilization-plan.md records AS-08a completion evidence stating 'disabled mode bypasses ... function/callback fast paths', which this un-gated surface falsifies, and Disabled mode charges fewer steps than Enabled for identical matched bodies — a cross-mode divergence the on/off equivalence tests cannot catch since both modes execute this same matcher. The two matchers (243-313 vs 506-610) duplicate ~5 overlapping instruction patterns as claimed.

**Verifier correction:** Strengthening: AS-08a completion evidence in docs/architecture-stabilization-plan.md explicitly claims disabled mode bypasses function/callback fast paths, so this surface contradicts certified gate evidence, and in Disabled mode this matcher runs on every call (no compiled fast path intercepts first), making it the primary dispatch surface there. Nuance: the step-accounting hole is bounded (matched bodies are <=7 instructions and the caller's loop still charges one step per call instruction), so it weakens rather than unbounds the resource contract.

**Suggestion:** Gate it on optional_optimizations_enabled(), charge instructions().len() steps like the pre-setup path, and ideally delete it in favor of the compile-time FunctionFastPath, which already covers its patterns.

### [HIGH] Array.prototype.flatMap embeds a private, unguarded bytecode specializer for user callbacks

- **Kind:** `hack` · **Location:** `src/runtime/native/builtins/array/flatten.rs:260` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** `compile_flat_map_array_callback` (flatten.rs:260-327) pattern-matches the user callback's bytecode (LoadBinding/PushLiteral/NumberBinary/ArrayLiteral/Complete) inside the builtin, `eval_flat_map_array_number_binary` (flatten.rs:359-373) re-implements the numeric binary table (with the same powf bug), and `flat_map_array_source_for_binding` (flatten.rs:329-343) duplicates fast_path.rs's param_frames slot-matching. `eval_packed_numeric_array_flat_map` (flatten.rs:86-123) has no optional_optimizations_enabled() gate.

**Why it matters:** A builtin owns its own mini-JIT: it duplicates numeric semantics and binding-resolution logic that exist in two other places, runs even in OptimizationMode::Disabled (breaking the disabled-mode equivalence contract), and on a mid-loop bailout (flatten.rs:115-117 returns Ok(None) after self.step() and charge_runtime_steps per element) the generic path re-runs the whole loop, double-charging the step budget. This is workload-shaped code (packed all-number arrays + arithmetic-only callback returning an array literal) living outside the AS-08 isolation boundary.

**Verifier notes:** All four evidence bullets check out at the cited lines: flatten.rs:260-327 pattern-matches user-callback bytecode privately; flatten.rs:359-382 duplicates fast_path.rs:464-488's numeric table (same powf); flatten.rs:330-343 duplicates fast_path.rs:735-747's param-frame slot matching; and the path has no optional_optimizations_enabled() gate while being reachable via the generic dispatch (direct.rs:437-445 -> 482 -> array.rs:142) even in OptimizationMode::Disabled, contradicting optimizer.rs:7's documented contract that Disabled means "Execute the generic semantic paths". The finding is actually stronger than claimed: the private compiler omits the whole guard set the sanctioned specializer requires (fast_path.rs:491-504: not async/generator, no arguments use, no parameter defaults, no rest param, unique param names; plus self-binding at function/mod.rs:257 and with_environments at callback_fast_path.rs:26), creating latent semantic-divergence risk (e.g. async arrow or rest-param callbacks). One sub-claim is overstated: the mid-loop bailout at flatten.rs:115-117 is type-deterministic (all elements are numbers), so it always fires on the first iteration and double-charges at most one step() plus one pattern.step_count, not the whole loop; and "AS-08" appears nowhere in the repo (external audit taxonomy, though the equivalent contract is documented in optimizer.rs).

**Verifier correction:** Array.prototype.flatMap (src/runtime/native/builtins/array/flatten.rs:86-123, 260-343, 359-382) embeds a private bytecode specializer for user callbacks that (a) duplicates the central fast-path machinery (fast_number_binary at fast_path.rs:464-488, param_index_for_operand at fast_path.rs:735-747), (b) runs ungated even in OptimizationMode::Disabled via the generic dispatch (direct.rs:442 -> eval_array_native_function_kind -> array.rs:142), violating optimizer.rs:7's documented contract that Disabled executes only generic semantic paths (every comparable path is gated: callback_fast_path.rs:18, function/mod.rs:254), and (c) omits the sanctioned specializer's guard set (fast_path.rs:491-504: async/generator, class constructor, new-target mode, arguments use, parameter defaults, rest params, duplicate param names; plus self-binding and with_environments checks), leaving latent semantic-divergence risk for e.g. async or rest-param callbacks. Correction to the original note: the bailout at flatten.rs:115-117 is type-deterministic, so it always occurs on the first iteration; the step-budget double-charge is bounded to one step() plus one pattern.step_count, not the whole loop. "AS-08" is not a repo-documented invariant.

**Suggestion:** Either delete it or rebuild it on the shared FunctionFastPath/CallbackFastPath machinery with the mode gate, and make bailout happen before any accounting side effects.

### [HIGH] Global environment split across three stores kept coherent by manual sync calls at every cached write site

- **Kind:** `inelegant` · **Location:** `src/runtime/property/static_names/mod.rs:272` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** `sync_global_object_property_binding` (globals.rs:365-384) must be invoked after cached property writes at six sites in static_names/mod.rs alone (lines 272, 280, 311-313, 372-374, 380-382, 411-413) to mirror global-object writes back into `globals` var-binding cells; `builtin_globals` (core.rs:321-334 insert_global_builtin) is a third store of lazily materialized builtins, consulted via `builtin_value`'s 80-line name match (core.rs:34-112).

**Why it matters:** One concept (the global environment record) has three owners — object properties, var-binding cells, and builtin cells — with coherence enforced by remembering to call the sync helper on every new write path (the cache layer had to grow the call in five separate branches). Any future cached-write path that forgets the call silently desynchronizes `var` bindings from global-object properties. Lazy materialization by string name also means globals' existence depends on lookup order.

**Verifier notes:** All cited code exists as described: sync_global_object_property_binding (globals.rs:365-384) mirrors global-object writes into `globals` var-binding cells at exactly the six cited sites in static_names/mod.rs, plus three more uncited sites (assignment.rs:196/234, with_environment.rs:64); builtin_globals is a lazily materialized third store (insert_global_builtin, core.rs:321-334) consulted via the 79-line name match in builtin_value (core.rs:34-112, with a near-duplicate 66-line copy at core.rs:114-180). The sync is load-bearing, not belt-and-suspenders: BindingCell is Rc<Mutex<Binding>> separate from object property storage, and bare-identifier global var reads resolve directly to cells (static_binding_lookup.rs:368-385) without consulting the object, while the semantic write layer (semantic_object/mutation.rs, descriptor.rs) has no centralized global-object interception on writes — only reads are centrally intercepted. The predicted failure mode already exists structurally: Reflect.set(globalThis, ...) (reflect.rs:112-130) and Object.defineProperty(globalThis, ...) (descriptor.rs:55-141) write global-object properties with no cell sync, and no doc comment or test enforces the invariant.

**Verifier correction:** Claim verified and slightly understated: beyond the six sites in static_names/mod.rs there are three more manual sync sites (bytecode/ops/assignment.rs:196, 234; binding/with_environment.rs:64), a reverse-direction helper sync_global_object_binding_property (globals.rs:344) with its own two call sites, and a near-duplicate of the 80-line builtin_value match (direct_builtin_callable_value, core.rs:114-180). Generic write paths that reach the global object — Reflect.set (reflect.rs:112-130) and Object.defineProperty (semantic_object/descriptor.rs:55-141) — already lack the sync, so the "future path forgets the call" risk appears to be a present latent desync, not merely prospective. One softening: "globals' existence depends on lookup order" is true of internal store state (cell existence/slots in builtin_globals) but is largely masked from JS-observable behavior by consistent materialize-on-demand fallbacks (static_bindings.rs:388-399).

**Suggestion:** Make the global var-binding cells views over the global object's property slots (single ownership), or funnel every global write through one choke point that owns the mirroring, so the sync call cannot be forgotten.

### [MEDIUM] Two mirrored 100+ variant enums (NativeCallTarget vs NativeFunctionKind) with a 458-line hand-written bijection and a silent catch-all

- **Kind:** `inelegant` · **Location:** `src/runtime/native/function/call_target.rs:413` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** call_target.rs is entirely hand-maintained from_*/to_* tables mapping every NativeCallTarget to a NativeFunctionKind and back (call_target.rs:6-457). `from_string_call_target` ends with `_ => Self::String` (call_target.rs:413) instead of returning Option, so any future NativeCallTarget variant missing from the earlier tables silently maps to the String constructor kind.

**Why it matters:** The two enums encode the same concept (identity of a builtin) at two layers and can only drift apart; the compiler must be edited in api/native_call.rs, the mirror in call_target.rs, the registry slot in registry.rs, and both dispatch pyramids in direct.rs for every new builtin. The `_ => Self::String` fallback converts a forgotten table entry into wrong cached kind data (remember_static_property_native_call_kind stores it) rather than a compile error — the runtime id guard usually saves execution, but the cache then permanently records a String-kind entry for a non-String site.

**Verifier notes:** The structural facts check out: call_target.rs is 458 lines of hand-written from_*/to_* tables and line 413 is `_ => Self::String` (currently reached only by NativeCallTarget::String itself — a diff of all 158 variants against the from_* arms shows the tables are complete today). But the claimed harm is wrong: `direct_native_call_kind` (direct.rs:417-429) filters the computed kind with `native_function_id(kind) == id`, so a mismapped String kind for a non-String callee fails the filter, returns None, and is never cached; the fallback (direct.rs:47-52) caches the callee's true kind read from the function object itself, and cache reads re-validate the function id (`kind_if_current`, static_names/mod.rs:91). A forgotten table entry therefore degrades to a silently missed optimization (slow path), not "wrong cached kind data". Also "mirrored" overstates it: NativeFunctionKind has ~310 variants vs NativeCallTarget's 158 — an injection into a superset with a legitimate layering reason, not duplicate mirrors.

**Verifier correction:** call_target.rs:6-457 hand-maintains the NativeCallTarget→NativeFunctionKind injection, and `from_string_call_target` ends with `_ => Self::String` (line 413) doing double duty as the arm for NativeCallTarget::String. Contrary to the claim, a forgotten table entry cannot poison the kind cache: direct_native_call_kind (direct.rs:425-428) only accepts a kind whose registered function id equals the actual callee id, the fallback caches the callee's true kind from the function registry, and cache hits re-validate the function id — so drift fails safe as a silently missed fast path (perf only). The remaining issue is maintainability: each new direct-callable builtin must be edited in api/native_call.rs, call_target.rs, and direct.rs, and the catch-all hides an omission that an explicit `NativeCallTarget::String => Self::String` arm plus an Option return (or a macro generating both directions) would surface. Low severity: current tables are complete and the guards make drift non-corrupting.

**Suggestion:** Make from_string_call_target return Option and derive the bijection (macro or single table) so a missing arm is a compile error; longer term, collapse NativeCallTarget into NativeFunctionKind and let the compiler reference kinds directly.

### [MEDIUM] Hand-numbered 330-slot native-function registry with holes reserved in a different file and only runtime uniqueness checking

- **Kind:** `inelegant` · **Location:** `src/runtime/native/function/registry.rs:18` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** ~180 manually numbered `const *_SLOT: NativeFunctionSlot = NativeFunctionSlot::new(N)` constants (registry.rs:18-287) with gaps 58-65, 112, 289 consumed by error_constructor_slot_index in registry_error.rs:3-16, plus arithmetic slot bases (DATA_VIEW_GET_SLOT_BASE=227, ATOMICS_METHOD_SLOT_BASE=316); the kind→slot mapping is spread over ~10 const fns in registry.rs plus registry/object_slot.rs and registry/data_view_slot.rs.

**Why it matters:** Slot uniqueness is an unchecked cross-file invariant: if two kinds ever map to one index, `registry.get(kindB)` silently returns kindA's NativeFunctionId (used by direct_native_call_kind guards and by native_function_id short-circuits in global_function_value), and the error only surfaces if both happen to be inserted. Every new builtin requires manually picking the next free number while dodging bases reserved elsewhere — pure discipline-based integrity in a codebase that elsewhere insists on checked invariants.

**Verifier notes:** registry.rs:18-287 matches the claim exactly: hand-numbered constants with gaps 58-65/112/289 filled by error_constructor_slot_index in registry_error.rs:3-16, arithmetic bases (227/236/273/282/297/311/316) whose enum ranges I verified fit their gaps with zero slack (e.g., DisposableStack 282-288 abuts SuppressedError's cross-file 289; Atomics 316-329 abuts count 330), and the mapping spread over 13 const fns plus two submodules. get() (registry.rs:301-303) reads the slot without any kind check, so a collision silently returns the other kind's id; the only uniqueness enforcement is the runtime double-insert error (registry.rs:321-326), and since builtins are lazily materialized per VM (docs/project-plan.md), a collision where only one kind is registered silently reuses the wrong function via global_function_value's short-circuit (core.rs:313-315) and can pass direct_native_call_kind's identity guard (direct.rs:417-429). I found no compile-time assertion or test enforcing slot uniqueness anywhere in src/ or tests/.

**Verifier correction:** Claim verified in full; if anything it is slightly understated: every arithmetic slot range fits its reserved gap with zero slack (DisposableStack methods 282-288 sit one index below SuppressedError's 289 reserved in a different file), so adding a single enum variant to any method-kind enum silently overlaps a neighbor. Mitigations that keep this at medium rather than high: no collision exists today, the fixed 330-slot array bounds-checks with a runtime error rather than UB, and insert() does error on double registration with rollback support — but that check only fires when both colliding kinds are materialized in the same VM, which lazy builtin materialization does not guarantee under test.

**Suggestion:** Generate slot indices from a single exhaustive match (or an EnumCount-style derive) so uniqueness and density are compile-time facts; at minimum add a test iterating all kinds asserting injective slot assignment.

### [MEDIUM] builtin_value and direct_builtin_callable_value are near-identical 80-line dispatch tables that must be edited in tandem

- **Kind:** `inelegant` · **Location:** `src/runtime/native/core.rs:114` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** core.rs:34-112 (`builtin_value`) and core.rs:114-180 (`direct_builtin_callable_value`) are the same name→constructor match except the latter omits Atomics/JSON/Math/globalThis/Infinity/NaN/Performance/Reflect and a few others; both end with the same ErrorName fallback.

**Why it matters:** Duplicated semantics with an implicit, uncommented difference rule (callable vs namespace values). A new global must be added to both tables plus NativeCallTarget::from_binding_name (api/native_call.rs) — three name-keyed lists encoding one fact; missing one produces silent behavioral divergence between the direct identifier-call path and generic lookup rather than an error.

**Verifier notes:** The duplication is real: core.rs:34-112 and core.rs:114-180 are near-identical match tables differing only by the non-callable globals (Atomics, JSON, Math, Reflect, Performance, globalThis, Infinity, NaN), with no in-file comment stating the rule and no consistency test. However, the claimed failure mode is refuted: the only caller of direct_builtin_callable_value (unresolved_direct_builtin_callable, static_bindings.rs:328, gated on optional optimizations) is used at runtime/mod.rs:520, which on None falls back to get_or_materialize_binding_bytecode, which itself calls builtin_value (static_bindings.rs:395) — so omitting a new callable from the direct table produces an identical result via the slower generic path, not silent behavioral divergence. Likewise NativeCallTarget::from_binding_name (api/native_call.rs:196) is a deliberately smaller 19-entry compiler hint that already omits many builtins (Map, Set, Proxy, BigInt, typed arrays, etc.); omission there just means native: None and the generic native-kind path runs, again correctness-preserving. The maintenance cost of forgetting an entry is a missed optimization, not wrong behavior.

**Verifier correction:** builtin_value and direct_builtin_callable_value are genuinely duplicated ~60-line tables whose difference (callable-only subset) is encoded only in the function name, and they could be collapsed into one table plus a callable predicate. But the claimed hazard is overstated: the direct-call path falls back to builtin_value via get_or_materialize_binding_bytecode (runtime/mod.rs:520-527, static_bindings.rs:395), so a missing entry in the direct table costs only the fast path, with identical semantics. NativeCallTarget::from_binding_name is not a third copy of the same fact — it is an intentionally smaller compile-time hint list whose omissions are also correctness-preserving. Maintenance-burden inelegance, severity low.

**Suggestion:** Express one table of (name, kind, is_callable) entries and derive both functions from it.

### [MEDIUM] Benchmark-shaped peephole superinstructions (index-mask array accumulation, bit-and-literal chain terms)

## Optimization opportunities (post-cleanup)

### [MEDIUM] Every GC that frees one object invalidates all property caches globally and recounts the whole heap

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/object/base.rs:119` · **Status in project docs:** UNDOCUMENTED

**Evidence:** sweep_unmarked_objects bumps the single global prototype_lookup_version whenever removed > 0 (object/base.rs:97-119), which invalidates every CacheablePropertyLookup guard engine-wide (lookup.rs:33-37); the same sweep does a full O(live-heap) storage_payload_bytes recount (base.rs:104-115); collect_garbage additionally zeroes all native-call and call-value caches unconditionally (gc.rs:588, 623-638).

**Why it matters:** Correct but maximally blunt: cache warmth and GC frequency are inversely coupled, so allocation-heavy workloads pay a full re-warm of every inline cache after each collection. This is the direct cost of using raw reusable ids as cache keys.

**Suggestion:** Once ids carry generations (AS-05 built VM-bound identity for handles; extend to ObjectId/FunctionId in cache keys), guards can compare generation instead of relying on a global epoch bump, letting caches survive collections; track payload bytes incrementally per sweep instead of recounting.

### [LOW] Per-activation cache-handle allocation and ledger reservation on every script/function entry

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/property/static_names/cache.rs:24` · **Status in project docs:** UNDOCUMENTED

**Evidence:** StaticNameAtomCacheHandle::new allocates four boxed Cell slices per instantiation (cache.rs:24-51); with_static_name_caches reserves/commits/releases CacheEntry storage-ledger counts and pushes/pops three stacks per activation (static_names/mod.rs:48-72, 661-689); module/script paths build fresh handles per evaluation (module.rs:178, 565; script_execution.rs:106), discarding warm caches.

**Why it matters:** The cache infrastructure itself has per-call overhead (allocation + accounting) and scripts/modules rewarm from zero on each evaluation, while functions already demonstrate the better design (persistent per-function handles, function/mod.rs:50-51). Cheap win: intern the handle on the compiled script/function and only reserve ledger counts once.

**Suggestion:** Attach cache handles to CompiledScript/CompiledModule the way Function records already do, making activation push/pop a pointer copy with one-time accounting.
