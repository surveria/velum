# GC, roots, resource accounting, embedding API

> Детальный отчёт аудита архитектуры rs-quickjs. Подсистема: `gc-accounting-api`. Сгенерировано многоагентным аудитом 2026-07-12; каждая находка уровня medium/high прошла адверсариальную проверку отдельным агентом. Материал на английском (исходный текст аудита).

## Subsystem assessment

The GC/accounting/embedding subsystem is unusually disciplined for a prototype: a single direct-root visitor contract feeds both the marker and public diagnostics, arenas are non-moving with reserve-before-mutate sweeps, weak/ephemeron edges have a typed strength model, and the embedding surface (RetainedValue, OwnedValue, HostCall) enforces VM identity and generation checks throughout. Layering is mostly clean — optimization caches are invalidated before slot reuse and version-guarded, and specialized call paths validate callee identity with a genuine fallback to the semantic path, so the AS-08 claim of layered rather than duplicated semantics holds up under inspection. The two structural weaknesses are both about split ownership. First, the mark phase has one genuine completeness hole: object instance private slots live in a side table that the whole-heap diagnostic tracer visits but the per-object marking tracer does not, so values held only by #fields of live instances are swept — a high-severity, undocumented correctness gap. Second, the twenty-six-kind double-entry accounting is sound as a concept but is transcribed by hand twice: ~122 grow/release sites must mirror ~330 lines of recount formulas term-for-term, and I found one concrete divergence (the arguments-binding CacheEntry term exists on the grow side but not the recount side), which would hard-fail every snapshot and GC in a VM containing a function that uses `arguments` — and the accounting test suite never exercises that combination. The reconciliation is also entangled with the collector (two full-heap recounts per collection) rather than layered as a verify mode over an O(reclaimed) release. Kludge count is genuinely low — the documented AS program really did remove benchmark- and harness-shaped code; what remains is bureaucratic duplication (five hand-rolled counter visitors, dual Engine/Vm vs Runtime/Context embedding hierarchies, a kind-tag-plus-optional-payload CollectionData union) rather than semantic special-casing. The main foundational bets to revisit before adding an automatic GC trigger: fix the private-slot marking gap, derive both sides of the ledger from single per-record count functions, make transient-root completeness mechanically checkable (GC-stress mark at allocation points), and move Value to an O(1)-clone string representation, which would cheapen rooting, host calls, and ephemeron marking simultaneously.

## Strengths

- Fast paths genuinely layer over semantics rather than forking them: direct native calls validate the actual callee's registry kind against the compiler hint before dispatch and fall back to the common semantic call on any miss (runtime/native/function/direct.rs:26-56), matching the AS-08 contract; the compiler emits name-based hints (api/native_call.rs) but they are advisory only.
- Cache-invalidation ordering around GC is correct and clearly sequenced: identity-bearing caches (native-call and call-value caches, per-function static caches) are cleared before sweep (gc.rs:588, 623-638), object sweep bumps prototype_lookup_version so version-guarded property-lookup caches self-invalidate (object/base.rs:119, object/property/lookup.rs:36), and SlotArena documents that slot reuse is legal only after that invalidation (arena.rs:5-7).
- The sweep is panic-free by construction: free-list capacity is reserved before any removal (arena.rs:135), mark-bitmap length is validated against the arena, and every arithmetic step in GC, accounting, and numeric code is checked with typed errors — a consistent no-unwrap discipline across all ~4k lines reviewed.
- RetainedValue is a well-designed handle: non-cloneable, VM-identity checked, slot+generation checked, deterministic release() with a Drop safety net, and registry pointer equality guards against cross-registry forgery (retained_values.rs:234-242).
- TransientRootScope is leak-proof under early return: Drop-based removal, #[must_use], inactive fast path for all-primitive inputs, and scope-id keying tolerates any drop order.
- Root enumeration is a single shared visitor contract (visit_direct_roots) consumed by both the marker and the public counting snapshot, so the diagnostic API and the collector see the same root registry by construction (roots.rs:182-278).
- The weak-edge model is typed and principled: WeakSet keys and WeakMap ephemerons flow through a separate WeakEdgeVisitor with per-category strength validation (async_trace.rs:176-195), and dead weak entries are swept against the same reachability the marker computed.
- The project’s self-documentation is exceptional: the semantic-architecture-inventory and stabilization plan record the four-variant callable split, the side-table exotic-object pattern, the raw-Value non-durability, and cache/specialization boundaries with named AS owners and explicit anti-regression guards — most conceptual debt found in code is already on the books.

## Confirmed findings (adversarially verified)

### [HIGH] GC mark phase never traces object instance private slots — live #field values get swept

- **Kind:** `risk` · **Location:** `src/runtime/gc.rs:200` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** gc.rs drain_queue: `MarkTarget::Object(id) => { context.objects.visit_object_strong_edges(id, self)?; ... }` which (object/trace.rs:35-41) calls only `self.object(id)?.visit_strong_edges(visitor)`. Instance private slots live in a parallel table `ObjectHeap.private_slots: Vec<Vec<PrivateSlot>>` (object/base.rs:47) and are visited ONLY by the whole-heap diagnostic walker `ObjectHeap::visit_strong_edges` (object/trace.rs:20-31, `VmObjectEdgeKind::InternalSlot`), which the marker never uses. Sweep clears private_slots for unmarked objects (object/base.rs:92-96) but keeps them for marked objects whose slot values were never marked. Writes land there via `objects.add_private_slot`/`set_private_field` (private.rs:291-307, 259-268).

**Why it matters:** A value reachable only through a live instance's private field (`class C { #x = {} }`) is unreachable to the marker: `vm.collect_garbage()` reclaims its arena record while the owning instance survives. The stale ObjectId then either errors ('object id is not defined') or, after slot reuse, silently aliases an unrelated new object — type confusion in a safe-Rust engine. The asymmetry between the two traversal APIs (diagnostic one complete, marking one not) also means edge-count snapshot tests cannot catch it, and tests/gc_reachability_smoke.rs contains no class/private-field case.

**Verifier notes:** The marker's MarkTarget::Object arm (gc.rs:200) calls visit_object_strong_edges, which only traces Object's own fields (properties, array storage, prototype, primitives, proxy, buffer views) and never the parallel ObjectHeap.private_slots table (base.rs:47); the only traversal that visits those slots (ObjectHeap::visit_strong_edges, trace.rs:20-31) is called solely by the diagnostic object_edge_snapshot counter, and roots (roots.rs:182-278) do not cover them either. Sweep (base.rs:91-97) keeps private slots on marked survivors while reclaiming their unmarked targets into a reusable free list (arena.rs:26,93), and ObjectId is a bare usize with no generation (value/ids.rs:54), so a surviving instance's #field value is reclaimed and its stale id later errors with "object id is not defined" (heap.rs:421) or aliases a reused slot. The asymmetry is real: Function::visit_strong_edges (trace.rs:393-410), which the marker does use, traces function private_slots and class_private_slots, so this is an oversight rather than a design invariant, and tests/gc_reachability_smoke.rs contains no class/private-field case. Every cited location, the failure mechanism, and the severity check out; no guard or invalidation refutes the claim.

**Suggestion:** Make `visit_object_strong_edges(id, ...)` visit `private_slots[id.index()]` values (same InternalSlot edges the whole-heap walker emits), or better: have exactly one per-object edge enumeration that both the marker and the diagnostic snapshot are built from, so completeness cannot diverge. Add a GC test: retain an instance with `#x = {}` across collect_garbage and read #x.

### [HIGH] Ledger grow formula and observed recount disagree on `arguments` binding CacheEntry — reconciliation hard-fails

- **Kind:** `risk` · **Location:** `src/runtime/accounting.rs:441` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** Grow side (function/storage.rs:10-33, called once from function/mod.rs:190-198) charges `CacheEntry` with `... + usize::from(has_self_binding).saturating_mul(2) + usize::from(has_arguments_binding).saturating_mul(2)` and commits it to the ledger. Observed side `record_callable_storage` (accounting.rs:436-467) recounts per function: param_binding_ids/param_atoms/param_frames/self_binding*2/class_fields/class_private_slots/fast_path/properties-cache/scope_template — there is no arguments term (grep 'arguments' in accounting.rs: zero hits). `ensure_durable_storage_ledger_matches` (accounting.rs:258-269) requires exact equality for CacheEntry.

**Why it matters:** Any VM containing a function whose bytecode `uses_arguments()` with a compiled arguments binding carries ledger = observed + 2 permanently; the next `storage_snapshot()`, `teardown_report()`, `finish()`, or `collect_garbage()` returns Err('CacheEntry storage ledger mismatch'). tests/vm_storage_accounting_smoke.rs and gc_reachability_smoke.rs never eval a function using `arguments`, so the hole is invisible to the reconciliation suite. Whether it fires in production or the suite simply lacks coverage, it is empirical proof that the 26-category double-entry design fails open: one missed mirror term at one of ~122 mutation sites bricks GC for the whole VM.

**Verifier notes:** The grow formula (function/storage.rs:25, sole caller function/mod.rs:190-198) durably commits +2 CacheEntry per function with a compiled arguments binding, but the observed recount in record_callable_storage (accounting.rs:416-484) has no arguments term (zero grep hits for "arguments" in accounting.rs), and no other ledger site or release path compensates — I checked every VmStorageKind::CacheEntry site outside accounting.rs and each is mirrored by a distinct observed term. CacheEntry is in LEDGER_ENFORCED_KINDS and ensure_durable_storage_ledger_matches (accounting.rs:258-269) requires exact equality, so any VM containing a function like `function f(){ return arguments.length }` (parser sets arguments_binding whenever uses_arguments, plumbed through bytecode/types.rs) permanently fails storage_snapshot, teardown_report, finish, and collect_garbage — GC checks the `before` snapshot first (accounting.rs:280) so it can never reclaim its way out. The coverage claim also holds: vm_storage_accounting_smoke.rs and gc_reachability_smoke.rs never use `arguments`, and arguments_object_smoke.rs only calls eval, which never triggers ledger enforcement (GC is explicit-API-only).

**Verifier correction:** One nuance: plain eval keeps working (no automatic GC or ledger enforcement during execution), so "bricks GC for the whole VM" applies to the GC/accounting/teardown API surface, not to script execution; the "fails open across ~122 sites" framing is extrapolation beyond the verified defect. The concrete mismatch, its permanence, and the test-coverage hole are all confirmed as stated.

**Suggestion:** Add the missing `usize::from(function.arguments_binding.is_some()).saturating_mul(2)` observed term (or delete the grow term); then derive both sides from ONE function (e.g. a per-record `fn cache_entry_count(&self)` used at grow, release, and recount) so the formulas cannot drift. Add an accounting test that snapshots after evaluating a function using `arguments`.

### [HIGH] Double-entry storage accounting maintained by hand at ~122 sites across 31 files with magic per-record multipliers

- **Kind:** `inelegant` · **Location:** `src/runtime/accounting.rs:9` · **Status in project docs:** documented (AS-plan)
- **Verification verdict:** confirmed

**Evidence:** `grow_count`/`release_count`/`reserve_count`/`release_count_on_drop` appear at ~122 call sites in 31 files (disposable_stack.rs, collections.rs, generator.rs, promise/mod.rs, object/heap.rs, bytecode/continuation.rs, ...). Each site must pick the right VmStorageKind and magnitude (e.g. `saturating_mul(2)` = atom+frame pairs, function/storage.rs:23-25), and accounting.rs must mirror every one of them term-for-term in ~330 lines of recount formulas (record_binding_storage/record_callable_storage/record_cache_storage/record_association_storage). LEDGER_ENFORCED_KINDS covers only 18 of 26 kinds — Atom/HeapString/Symbol/HostCallback/Object/ByteBuffer/OutputEntry/SourceRecord are snapshot-only, so enforcement is asymmetric by category.

**Why it matters:** The invariant 'ledger == full recount' is distributed, unenforced by types, and validated only at snapshot/GC/teardown time — the mismatch (finding above) surfaces far from the offending site with no provenance. The recount duplicates ownership knowledge that the owners themselves already have; CacheEntry alone aggregates ten unrelated things (param metadata, self/arguments bindings, class fields, fast-path presence, property caches, scope templates, atom indexes, native registry ids). This is sound double-entry bookkeeping in intent, but the entries are hand-written twice, so it verifies transcription discipline rather than semantics.

**Verifier notes:** Every factual element checks out: ~121 genuine call sites across 31 files (126 grep hits minus definitions), saturating_mul(2) multipliers at function/storage.rs:23-25 whose mirror terms are split between accounting.rs:441 and separate suspended-phase counters, ~330 lines of hand-written recount formulas in accounting.rs:310-641, LEDGER_ENFORCED_KINDS covering exactly 18 of 26 kinds with the 8 excluded kinds exactly as named, and reconciliation invoked only at snapshot/GC/teardown (api/embedding.rs:473-496, gc.rs:587-610) with a provenance-free error message (accounting.rs:263-265). The 'why bad' reasoning holds — release_count_on_drop's own doc comment (storage_ledger.rs:119-128) confirms the design leans on deferred snapshot reconciliation as the safety net, and CacheEntry does aggregate 10+ unrelated sources. Two mitigations soften but do not refute: the 18/26 asymmetry is plausibly principled (the 8 snapshot-only kinds each have a single authoritative owner container, making a ledger redundant), and there is partial type assistance via #[must_use] VmStorageReservation plus a dedicated smoke-test suite exercising the invariant.

**Verifier correction:** Confirmed as described. One evidence bullet is weaker than framed: the 18/26 enforcement asymmetry is arguably by design, not an oversight — the 8 snapshot-only kinds (Atom/HeapString/Symbol/HostCallback/Object/ByteBuffer/OutputEntry/SourceRecord) each have a single authoritative owner container whose count is read directly, and their limits are still enforced at mutation sites via ensure_storage_totals; the ledger exists precisely for the distributed-ownership kinds. Partial mitigations exist (typed #[must_use] reservations with stale-detecting commit, RAII releases, and smoke tests exercising the invariant), but the core defect stands: the ledger==recount invariant spans ~121 hand-mirrored call sites plus ~330 lines of recount formulas, magic multipliers must be transcribed identically in two places (and for the ×2 binding pairs, across different lifecycle-phase counters), and violations surface only at snapshot/GC/teardown with no provenance about the offending site.

**Suggestion:** Move the count/byte contribution into each owning type (one method per record type used by grow, release, and recount), or make the ledger the single source of truth with RAII charge guards owned by the records themselves, and demote the full recount to a debug/consistency-check mode. Split CacheEntry into narrower kinds or stop enumerating heterogeneous members by formula.

### [MEDIUM] Drop-path ledger release uses saturating_sub — silent desync deferred to an unrelated failure site

- **Kind:** `risk` · **Location:** `src/runtime/storage_ledger.rs:127` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** `release_count_on_drop` does `count.set(count.get().saturating_sub(released_count))` and ignores a missing kind cell (`let Ok(count) ... else { return; }`). Callers: TransientRootScope::drop (transient_roots.rs:180-184), RetainedValueState::discard (retained_values.rs:128-129).

**Why it matters:** An over-release in a destructor clamps at zero instead of erroring, and an under-release accumulates; the doc comment says 'snapshot reconciliation still detects any violated invariant' — but that detection fires later as a generic '{kind} storage ledger mismatch' from storage_snapshot()/collect_garbage(), pointing nowhere near the destructor that lied. Combined with per-kind exact-equality enforcement, one buggy Drop turns every subsequent snapshot and GC of the VM into a permanent error state with no recovery path.

**Verifier notes:** The code matches the citation: release_count_on_drop (src/runtime/storage_ledger.rs:119-128) saturates and ignores a missing cell, and detection is deferred to the generic per-kind mismatch in ensure_durable_storage_ledger_matches (src/runtime/accounting.rs:258-269) with no resync path, so a desync would permanently fail snapshot/GC. However, the auditor's failure scenario is unreachable in current code: both callers derive the released count from the same structure the snapshot observes, under the same lock (TransientRootScope::drop releases the actual vec-removal count that grow_count paired 1:1 with pushes; discard releases 1 only when flipping a Some slot to None under a generation guard), so the ledger always equals the observed count and saturating_sub can never clamp. The missing-cell branch is dead code (index() is a total match 0..=25 into a 26-element array), the deferral is the explicitly documented contract for error-less destructors, and the mismatch error names the kind plus tracked/observed counts, localizing to 2-3 mutation sites. What remains is a legitimate but minor diagnosability trade-off contingent on a hypothetical future destructor bug, not a defect in the code as written.

**Verifier correction:** Downgraded to low: the described over/under-release cannot occur with the current callers because drop-path release counts are structurally derived from the same state the snapshot observes (no independent number that can drift), the silent missing-cell return is unreachable, and deferred reconciliation is the documented design for error-less destructors. The surviving concern is narrow: if a future Drop-path accounting bug were introduced, the ledger mismatch would surface non-locally and permanently poison storage_snapshot()/collect_garbage() with no resync path. A debug_assert in release_count_on_drop (asserting released_count <= count.get()) would restore local detection in debug builds at zero release-path cost.

**Suggestion:** Have drop-path releases record the discrepancy (e.g. a poisoned/underflow flag with the kind and count) so the eventual reconciliation error can name its source; consider debug_assert-style logging in checked builds.

### [MEDIUM] Ephemeron fixpoint clones the entire ephemeron list every iteration and never retires satisfied entries

- **Kind:** `inelegant` · **Location:** `src/runtime/gc.rs:184` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** Reachability::capture loop: `loop { marker.drain_queue(context)?; let mut added = false; for (key, value) in marker.ephemerons.clone() { if marker.weak_key_is_reachable(&key) { added |= marker.mark_value(&value)?; } } if !added { break; } }` — `ephemerons` is Vec<(Value, Value)> and Value clones can deep-copy Strings/JsBigInt.

**Why it matters:** Each pass allocates and deep-clones all E ephemeron pairs (done only to appease the borrow checker), and pairs whose values are already marked are rescanned forever. A WeakMap chain where entry i's value keys entry i+1 forces E passes: O(E^2) pair visits plus E full-vector clones. Standard ephemeron marking retires satisfied pairs and keeps a pending list, which is both faster and clearer about the algorithm's invariant.

**Verifier notes:** src/runtime/gc.rs:181-192 contains exactly the cited loop: each fixpoint pass calls `marker.ephemerons.clone()` (a full Vec<(Value,Value)> allocation done solely because the for-loop would otherwise hold an immutable borrow while `mark_value` needs &mut self), and `ephemerons` is append-only — populated in `visit_ephemeron` (line 558) from WeakMap entries and never retired, so pairs with already-marked values are rescanned every pass until global fixpoint. A reverse-ordered chain of E WeakMap entries (value_i is key_{i+1}) does force ~E passes, i.e. O(E^2) pair visits plus ~E full-vector clones; no guard, dedup, or pending-list exists anywhere in the file. The one factual error in the evidence: Value clones deep-copy only Value::String(String) — JsBigInt is Rc<BigInt> and JsString is Rc-backed, so those clone as refcount bumps. Severity medium is fair for an inelegance in the GC mark core, slightly tempered by GC being embedder-invoked only (no allocation-triggered collection).

**Verifier correction:** Core claim verified: the ephemeron fixpoint clones the entire Vec<(Value,Value)> every pass (borrow-checker workaround) and never retires satisfied pairs, giving O(E^2) pair visits and ~E vector clones on an adversarially ordered WeakMap chain. Minor correction to the evidence: JsBigInt (Rc<BigInt>) and JsString (Rc<StringData>) clone as cheap refcount bumps; only Value::String(String) — possible as a WeakMap value — is a deep copy. Also note GC is only explicitly invoked via collect_garbage/heap_reachability_snapshot (no automatic collection), so the pathological case requires an embedder-triggered GC over an unfavorably ordered chain; the per-pass full-vector clone, however, happens on every capture regardless.

**Suggestion:** Store ephemerons as indices/ids rather than cloned Values; each pass, drain into a `pending` vec, mark values for reachable keys, and push back only still-unsatisfied pairs (swap two vectors, no cloning).

### [MEDIUM] collect_garbage performs two complete O(heap) accounting recounts inline, entangling the audit with the collector

- **Kind:** `inelegant` · **Location:** `src/runtime/gc.rs:587` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** collect_garbage: `let before = self.owner_storage_snapshot()?;` ... sweep ... `let after = self.owner_storage_snapshot()?; self.release_collected_storage(&before, &after)?;`. Each owner_storage_snapshot (accounting.rs:271-273 → record_storage_counts) walks every function's params/properties/class fields, every collection's entries, every iterator's items, every activation frame, every cache table. release_collected_storage then re-verifies ledger equality twice more (accounting.rs:280, 295). ObjectHeap::sweep_unmarked_objects additionally recomputes all object/buffer payload bytes from scratch after any sweep (object/base.rs:100-119).

**Why it matters:** The GC's ledger reconciliation is not incremental: instead of releasing what the sweep actually removed (the sweep knows exactly which records died), it diffs two full-heap recounts. A collection is therefore mark O(live) + sweep O(slots) + 2×recount O(heap) + payload recompute O(objects), and the audit is load-bearing (release_collected_storage errors abort the GC report) rather than a checked invariant layered over an O(reclaimed) release. This shape blocks any future incremental/generational evolution because accounting correctness is defined only in terms of whole-heap recounts.

**Verifier notes:** The mechanics are all real: collect_garbage (src/runtime/gc.rs:587,609-610) brackets the sweep with two owner_storage_snapshot calls, each a genuine O(heap) walk (accounting.rs:394-641 iterates every function's properties/params/class-field caches, all activation frames, collection entries, iterator items, and cache tables; object/accounting.rs:47-57 walks every object's properties), and release_collected_storage is load-bearing since the arena sweep (arena.rs:127) drops records with no ledger hooks, making the before/after diff the sole ledger decrement path; object/base.rs:101-118 also recomputes all payload bytes from scratch when removed > 0. However, the claim omits three mitigations: (1) the audit-as-load-bearing shape is a documented design choice — storage_ledger.rs:115-118 states snapshot reconciliation is the detection backstop for error-less destructor paths (release_count_on_drop saturates silently), i.e. deliberate dual-entry bookkeeping, not accidental entanglement; (2) collect_garbage is reachable only via the explicit embedding API (src/api/embedding.rs:463), and mark+sweep already visit every live record and slot, so the recounts are a ~2-3x constant factor on an already O(heap) stop-the-world operation, not an asymptotic regression (the accounting.rs:280/295 equality checks the evidence cites are O(18) comparisons, not extra recounts); (3) "blocks any future incremental/generational evolution" is overstated — every non-GC path already releases exact per-kind counts (function/properties.rs:624-625, collections.rs:268-281, execution_storage.rs, binding/scope.rs:417-429) and the per-record footprint helpers an O(reclaimed) release would need already exist, so this is a contained refactor, not a blocker.

**Verifier correction:** collect_garbage does perform two full O(heap) accounting recounts (gc.rs:587,609) and releases ledger counts by diffing them rather than tallying what the sweep freed, and ObjectHeap::sweep_unmarked_objects recomputes all payload bytes from scratch (object/base.rs:101-118). But this is a documented dual-entry scheme — snapshot reconciliation is the stated backstop for error-less destructor releases (storage_ledger.rs:115-118) — and GC is explicit-only (api/embedding.rs:463) and already O(heap) for mark+sweep, so the recounts add a constant factor, not a new complexity class. The two ledger equality checks (accounting.rs:280,295) are O(#kinds) comparisons, not additional recounts. An O(reclaimed) release with the recount demoted to a debug-mode audit would be cleaner and the per-record footprint helpers for it already exist, but nothing structurally blocks that refactor.

**Suggestion:** Have sweep return per-kind release tallies (it already returns per-arena counts) and release those directly; keep the before/after full recount as an optional verify mode (feature flag or debug builds).

### [MEDIUM] Functions cannot be WeakMap/WeakSet keys and the collector hard-codes that deviation

- **Kind:** `risk` · **Location:** `src/runtime/native/builtins/weak_collections.rs:454` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** `can_be_held_weakly` returns true only for `Value::Object(_)` and unregistered `Value::Symbol`; `Value::Function/NativeFunction/HostFunction` are rejected (spec: any function object CanBeHeldWeakly). The GC mirrors this: `weak_key_is_reachable` (gc.rs:478-493) classifies Function/NativeFunction/HostFunction keys as never-reachable, so even if weak collections ever accepted them their entries would be swept while the callee is live.

**Why it matters:** `new WeakSet().add(function(){})` and caching patterns like `weakMap.set(callback, state)` throw TypeError — a real-world compat break rooted in the four-way callable Value split (Function/NativeFunction/HostFunction are not ObjectIds). The identity split itself is documented debt with AS-02/AS-05 owners, but this weak-key consequence is not listed in Known Property Divergences, and the GC's weak-key matcher now encodes the deviation in a second place that must be remembered when callables are unified.

**Verifier notes:** The cited code matches the claim exactly: `can_be_held_weakly` (weak_collections.rs:454-460) returns true only for `Value::Object(_)` and unregistered `Value::Symbol`, and it guards all seven weak entry points (set/add/has/delete/get/getOrInsert/getOrInsertComputed); since `Value::Function/NativeFunction/HostFunction` are distinct non-Object variants (src/value/kind.rs:8-21, `type_name()` = "function"), `new WeakSet().add(function(){})` deterministically throws the "WeakMap and WeakSet keys must be objects or symbols" TypeError, while spec CanBeHeldWeakly accepts any Object including functions. The GC mirror is also real and slightly worse than "mirrored": `weak_key_is_reachable` (gc.rs:478-493) hard-codes false for the three function variants even though the Reachability marker carries `functions`/`native_functions`/`host_functions` liveness bitmaps (gc.rs:167-169) it could have consulted, and it feeds both ephemeron marking (gc.rs:185) and weak-entry sweeping (gc.rs:654). The documentation claims hold too: the four-way callable split is documented AS-02/AS-05 debt (docs/semantic-architecture-inventory.md:54-57, 214-215), but the "Known Property Divergences" section (line 217-238) omits the weak-key consequence, no test in tests/weak_collections_smoke.rs or the quickjs differential corpus exercises a function-valued weak key, and collection_set (src/runtime/collections.rs:231) stores raw Values with no ToObject normalization that could rescue it. The only softening points are that the GC's function arms are currently dead in practice (the front-door check rejects functions first, which the auditor already framed conditionally) and that the divergences section is nominally scoped to property dispatch — but no document anywhere records this deviation, so medium severity for a silent, deterministic real-world compat break rooted in known debt is fair.

**Suggestion:** Record the divergence in the inventory now; when the AS-02/AS-05 unification lands, make weak_key_is_reachable delegate to the same CanBeHeldWeakly predicate the builtins use rather than keeping two hand-synced value-kind matches.

### [MEDIUM] 81 transient-root sites defend against a collection that currently cannot happen, so their completeness is untestable

- **Kind:** `risk` · **Location:** `src/runtime/transient_roots.rs:25` · **Status in project docs:** documented (AS-plan)
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** collect_garbage is reachable only via `Vm::collect_garbage`/`Context::collect_garbage` (grep: no internal caller); it needs `&mut Context`, host callbacks get no Context access, so GC can never run while an eval/transient scope is on the stack. Yet 81 call sites create TransientRootScope (with per-value Mutex lock, clone, try_reserve, and TransientRoot ledger traffic), e.g. resource_scope.rs:86-92, host.rs:512-513.

**Why it matters:** The mechanism is sound future-proofing for allocation-triggered GC (AS-05b1c documented), but today it is pure cost with unverifiable benefit: a forgotten scope at any allocation point is unobservable until an internal GC trigger is introduced, at which point every missed site becomes a latent use-after-free-shaped bug. Root-set completeness for categories TransientOperand/Call/Temporary rests on manual site discipline with no mechanical check (unlike the ledger, nothing reconciles 'values live across an allocation' against 'values rooted').

**Verifier notes:** The factual core holds: grep confirms `collect_garbage` (src/runtime/gc.rs:585) has no internal caller — only `Vm::collect_garbage` (src/api/embedding.rs:463) reaches it — and host callbacks receive only identity/registry/root-snapshot/values (src/api/host.rs:522), never `&mut Context`, so GC provably cannot run while any of the ~79-84 transient-scope sites is live; the per-site cost (Mutex lock, per-value clone/try_reserve/ledger grow in `add_values`) is real. However, two "why bad" pillars are wrong: (1) "completeness is untestable / a forgotten scope is unobservable" is contradicted by `HostCall::root_snapshot` (host.rs:304, captured at host.rs:521 while scopes are active) and tests/transient_root_snapshot_smoke.rs, whose six re-entrant tests assert positive TransientOperand/Call/Temporary counts at operand, call, iterator, descriptor, Proxy-result, and class-key probe points plus exact cleanup on success and error — site-level regressions are observable today; (2) "a forgotten scope at any allocation point" misstates the documented contract: AS-05b1c (architecture-stabilization-plan.md:1909-1910) states the future collector may start only after the relevant instruction/call scope is installed and that raw allocation helpers are NOT collector safepoints, so the discipline surface is the centralized per-instruction/call-entry registration, not 81 ad hoc allocation sites. What survives is the narrower point that no mechanical check reconciles "values live across a safepoint" against "values rooted," so exhaustive completeness rests on convention until an internal trigger (plus a stress mode) exists.

**Verifier correction:** The mechanism is documented deliberate future-proofing (architecture.md:300 defines `Vm::collect_garbage` as the only stop-the-world safepoint; AS-05b1c restricts future collector starts to installed instruction/call scopes, explicitly excluding raw allocation helpers as safepoints), and it is partially verifiable today: host callbacks observe live transient root counts via `HostCall::root_snapshot`, and tests/transient_root_snapshot_smoke.rs covers six re-entrant categories with exact-cleanup assertions, while the TransientRoot storage ledger ties the registry into tested accounting invariants. The residual risk is forward-looking only — no global reconciliation proves every value live across a safepoint is rooted — with zero correctness exposure today. Fair severity is low, with the actionable note that introducing any internal GC trigger must be gated on a collect-at-every-safepoint stress mode.

**Suggestion:** Before ever adding an internal GC trigger, add a stress mode that runs a full mark (not sweep) at every allocation point and asserts all reachable-from-Rust ids were marked (QuickJS-style GC-stress). Until then, consider gating the runtime cost of scope bookkeeping behind the same optimization/diagnostic mode switch.

## Additional low-severity findings (not separately verified)

### [LOW] Association storage category is a grab-bag: whole generator arena, prototype anchors, symbol registry, and object slots share one bucket

- **Kind:** `inelegant` · **Location:** `src/runtime/accounting.rs:599` · **Status in project docs:** UNDOCUMENTED

**Evidence:** record_association_storage counts under VmStorageKind::Association: collection_object_slots, symbol registry entries, promise_object_slots, `self.generators.len()` (every generator record!), generator_object_slots, seven `usize::from(prototype.is_some())` runtime anchors, async_generator_request_count, and iterator_symbol presence.

**Why it matters:** Promise and Collection records get their own storage kinds while Generator records are counted as 'Associations'; boolean prototype-anchor presence and full generator state are dimensionally different things summed into one limit-enforced number. The category exists to close the 26-kind taxonomy, not to describe ownership, so its limit (VmStorageLimits::max_count(Association)) cannot express any meaningful policy, and future readers cannot tell what releasing 'an Association' means.

**Suggestion:** Give generators their own kind (mirroring Promise), fold the fixed prototype anchors out of accounting entirely (they are bounded constants), and keep Association for the object-to-side-store slot tables it originally named.

### [LOW] Transient scope teardown is O(total active roots) per scope via retain(); Rc<parking_lot::Mutex> in a single-threaded VM

- **Kind:** `inelegant` · **Location:** `src/runtime/transient_roots.rs:179` · **Status in project docs:** UNDOCUMENTED

**Evidence:** TransientRootScope::drop: `state.roots.retain(|root| root.scope != self.scope)` — a full scan of the shared roots vec for every scope drop. Registry state is `Rc<Mutex<TransientRootState>>` (parking_lot Mutex under an Rc, which is !Send); RetainedValueRegistry uses the same pattern (retained_values.rs:197).

**Why it matters:** N nested or sequential scopes each holding a few values cost O(N^2) on unwind; scopes drop LIFO in practice (Rust locals), so a mark/truncate stack discipline would be O(1) amortized — the scope-id+retain generality pays for an out-of-order case that the ? -based control flow cannot produce. The Mutex exists only because RefCell::borrow can panic (banned); the result is lock traffic on every root push/visit and a re-entrancy deadlock (instead of a loud panic) if a visitor ever re-enters the registry.

**Suggestion:** Track (scope_id, start_index) and truncate on drop, falling back to retain only if out-of-order drop is detected; replace Mutex with a non-blocking try-borrow cell that surfaces Err on re-entry.

### [LOW] String-concat fast path threads a compiler-fed `final_result` flag and a magic 24-byte over-reserve through Value semantics

- **Kind:** `microopt-debt` · **Location:** `src/runtime/values.rs:13` · **Status in project docs:** UNDOCUMENTED

**Evidence:** `STRING_CONCAT_INTERMEDIATE_EXTRA_CAPACITY: usize = 24`; string_concat_step/string_concat_static_step take `final_result: bool` and call reserve_string_concat_tail to pre-reserve 24 extra bytes on intermediate results, mutating the left String in place. Guards (requires_generic_add, has_exact_utf8, string-operand check) fall back to the generic `add`.

**Why it matters:** The guards are complete (non-primitives and Symbols correctly reject before ToPrimitive side effects could diverge), so this is not a semantics fork — but it is a workload-shaped knob (string += in loops) with a hand-tuned constant, and the is-this-the-last-concat bit is compiler-plumbed state whose only purpose is capacity hinting. It survives AS-08's specialization audit because it lives inside the semantic owner, which is exactly how benchmark shaping re-accretes.

**Suggestion:** If kept, justify the constant with a measurement note and consider growing geometrically from the observed operand instead of a fixed 24; alternatively drop `final_result` and rely on String's own amortized growth.

### [LOW] Four hand-rolled counter visitors duplicate identical enum-index/checked-add boilerplate; docs say fourteen root categories, code has fifteen

- **Kind:** `inelegant` · **Location:** `src/runtime/roots.rs:5` · **Status in project docs:** UNDOCUMENTED

**Evidence:** RootCounter (roots.rs:128-171), CallableEdgeCounter and ObjectEdgeCounter (trace.rs:219-285), AsyncEdgeCounter (async_trace.rs:161-196), and StorageCounter (accounting.rs:199-245) each reimplement the same pattern: a const ALL array, a hand-written index() match that mirrors declaration order, get_mut + checked_add + total. trace.rs also carries `consume_reference`/`consume_weak_reference` (trace.rs:427-457) — const no-op exhaustive matches whose only job is to break compilation when a variant is added. Meanwhile docs/semantic-architecture-inventory.md (Public Handle Boundary) still says 'fourteen direct-root categories' while ROOT_KIND_COUNT = 15 (ModuleBinding added).

**Why it matters:** Five copies of the same counting machinery (~400 lines) invite index-order drift (each index() must match its ALL array by hand), and the exhaustive-no-op-match trick is a clever hack standing in for a derive. The 14-vs-15 drift shows the documented embedder contract already lags the code.

**Suggestion:** One generic `KindCounter<K: KindIndex>` (a tiny trait with COUNT and index(), or a macro emitting ALL+index together) replaces all five; fix the inventory's root-category count.

## Optimization opportunities (post-cleanup)

### [MEDIUM] Every host function call walks the entire direct-root set to build a diagnostic snapshot nobody may read

- **Kind:** `perf-opportunity` · **Location:** `src/api/host.rs:521` · **Status in project docs:** UNDOCUMENTED

**Evidence:** eval_host_function: `let values = args.to_owned_values();` (deep-clones every argument Value, including inline Strings) then `let roots = self.root_snapshot()?;` unconditionally before invoking the callback. root_snapshot → visit_direct_roots (roots.rs:182-278) iterates all global/builtin/local binding cells, all modules, all activation frames, all promise jobs, retained and transient roots, symbol registry, native registry — purely to populate the `HostCall::root_snapshot()` count struct.

**Why it matters:** Host-call cost is proportional to total VM root population, not to the call. A script calling a registered host function in a loop pays a full root-set traversal (plus argument deep-clones and a transient-root scope with lock+ledger traffic per value) per invocation. The snapshot is a diagnostic; making it eager on the hot embedding boundary is optimization-backwards.

**Suggestion:** Make HostCall::root_snapshot lazy — hand the callback a capability that computes the snapshot on demand (the Context is unavailable to the callback anyway, so capture what's needed or thread a closure), and pass borrowed args instead of cloned Vec when the callee contract allows.

### [LOW] RetainedValue slot allocation is a linear scan over all slots; the slot vec never shrinks

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/retained_values.rs:66` · **Status in project docs:** UNDOCUMENTED

**Evidence:** RetainedValueState::retain: `for (index, entry) in self.slots.iter_mut().enumerate() { if entry.value.is_some() { continue; } ... }` before pushing a new slot; there is no free list. Slots whose generation counter would overflow are skipped forever.

**Why it matters:** An embedder holding H live handles pays O(H) per retain — O(K·H) to create K handles — in an API explicitly recommended for all durable values ('Embedders must use retained handles', gc.rs doc). The generation-checked design is right; only the allocation strategy is quadratic.

**Suggestion:** Keep a Vec<usize> free list (the SlotArena in arena.rs already implements exactly this pattern and could be reused), making retain O(1).

### [MEDIUM] Value::String(String) makes every value copy a deep string clone; hot boundaries clone whole argument vectors

- **Kind:** `perf-opportunity` · **Location:** `src/api/host.rs:511` · **Status in project docs:** UNDOCUMENTED

**Evidence:** Value has both `String(String)` (inline owned Rust String) and `HeapString(JsString)` variants (value/kind.rs:14-15). eval_host_function does `args.to_owned_values()` cloning every arg; TransientRootScope::add_values clones each rooted value; the ephemeron fixpoint clones (Value,Value) pairs; retained_values clone on read (retained_values.rs:101-104). runtime_value() converts String→heap lazily (values.rs:22-27), so inline strings flow through the interpreter and get memcpy'd at each clone.

**Why it matters:** Once the accounting/rooting foundation is fixed, the single biggest representational win is making Value cheap to copy: today every rooting, boxing, argument-passing, and GC bookkeeping step that clones a Value can allocate. The dual representation also forces every string consumer to handle two cases (see has_exact_utf8/concat paths in values.rs) — a leaky split between 'compiler literal string' and 'runtime string'.

**Suggestion:** Move to one interned/refcounted string representation in Value (Rc<str>/JsString everywhere, with the string heap owning dedup), making Value clone O(1) and deleting the String-vs-HeapString case analysis from ~every string operation.

### [LOW] Mark-state uses per-kind Vec<bool> plus BTreeSets rebuilt per collection; symbols/strings marked through ordered sets

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/gc.rs:157` · **Status in project docs:** UNDOCUMENTED

**Evidence:** Reachability::capture allocates nine `vec![false; slot_count]` bitmaps plus `BTreeSet<SymbolId>` and `BTreeSet<StringId>` every run; mark_symbol/mark_string do O(log n) BTreeSet inserts; snapshot() recounts bitmaps with `true_count` per kind.

**Why it matters:** Minor today, but symbols and strings are dense arena ids like everything else — BTreeSets add log-factor marking and allocation churn on what should be a bitmap index test. Once GC becomes automatic (the plan's endpoint), allocation-per-collection and per-mark log factors matter.

**Suggestion:** Give StringHeap/SymbolTable slot-indexed mark bitmaps like the other arenas (they already expose sweep_unmarked over id sets), and reuse a scratch Reachability allocation across collections.
