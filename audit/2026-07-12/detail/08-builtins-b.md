# Builtins: TypedArray, Date, Promise, RegExp, collections, Proxy, Atomics

> Detailed architecture-audit report for Velum. Subsystem: `builtins-b`. Generated by a multi-agent audit on 2026-07-12; every medium/high finding was adversarially verified by an independent agent reading the cited code.

## Subsystem assessment

The builtins under review sit atop the AS-02/AS-03 semantic facades and mostly respect them: receiver branding, conversions, calls, and iterator protocol all delegate to the shared owners, and the buffer subsystem (ArrayBuffer/SharedArrayBuffer/TypedArray/DataView/Atomics) is the strongest area, with one element-kind owner, one ByteBuffer view API over the local/shared storage split, and live detach/resize observation. The main structural weaknesses are three misshapen physical owners that the AS program has not yet reached: the Map/Set backing store is a linear tombstone Vec (every operation O(n), inserts O(n²), never compacted), the 'collection' arena is a by-convention union that also hosts disposable stacks and six unrelated iterator-state flavors because ledger accounting categories are frozen, and iterator objects encode their internal state as an observable NUL-prefixed own property with a freshly allocated prototype per iterator. Below the protocol layer there is a stratum of duplicated abstract operations — SpeciesConstructor exists six times, RegExpExec is bypassed entirely (exec overrides are ignored by match/search/matchAll), and Promise combinator internal slots are modeled as '[[...]]'-named JS properties updated through the full property machinery. RegExp recompiles its pattern per execution and re-encodes input per match-loop iteration; Date is the expected hack magnet, with a rigid parse subset that cannot round-trip the engine's own toString, local time hard-wired to UTC, and i64/f64 conversions done by formatting strings — the same lint-dodge that makes every Uint8ClampedArray write allocate a String. Snapshot-vs-live iteration is split-brained: Map/Set are correctly live, but typed-array iterators and RegExp matchAll are eager snapshots that diverge observably from spec and do unbounded up-front work. None of the undocumented items contradict the layering rules — they are physical-representation and duplicated-operation debt inside the runtime layer — so the stabilization order remains sound; the highest-value next steps are a hash-backed keyed-collection store, a shared SpeciesConstructor/RegExpExec owner, typed side-store state for Promise combinators and iterators, and a compiled-regex cache, all of which the existing facades make safe to do.

## Strengths

- Buffer lifecycle is genuinely unified: one ByteBuffer facade hides the Rc<RefCell> (local) vs Arc<RwLock> (shared) split, views observe detach/resize live because they share the state cell, and Atomics correctly holds one exclusive guard across each full read-modify-write — the two-worlds design is well-contained rather than leaky.
- Typed-array element-kind dispatch has a single owner: TypedArrayElementKind::read/write centralizes all eleven kinds, the Number/BigInt content-type discriminator is checked at construction, species, and set boundaries, and DataView reuses the same discriminator — no per-type copy-paste of conversion semantics.
- Proxy internal methods follow the spec shape faithfully: every trap miss falls back to the shared semantic owners (semantic_property_read, Set, semantic_construct), Symbol keys survive round trips, and post-trap invariant validation is factored into one proxy_invariants module.
- ES2025 iterator helpers are properly lazy with careful close semantics; the plan/act split (HelperPlan/ModeAction) cleanly separates state borrows from JavaScript re-entry, and abrupt completions consistently close underlying iterators with correct error precedence.
- Map/Set iterators are live (cursor over index-stable entries), matching spec insertion/deletion-during-iteration behavior including clear-then-add; the tombstone representation, whatever its cost, is semantically right.
- Resource accounting discipline is pervasive and honest: every arena insert reserves ledger counts with rollback on failure, all arithmetic is checked, and iterator/collection charges reconcile after GC.
- Almost every builtin routes conversions and calls through the shared AS-03 abstract-operation owners (to_index, to_integer_or_infinity, semantic_is_callable/constructor, iterator_step/close), so the consolidation program demonstrably reached this subsystem.

## Confirmed findings (adversarially verified)

### [HIGH] Map/Set backing store is a linear Vec with SameValueZero scans and permanent tombstones (O(n) per op, O(n²) workloads)

- **Kind:** `inelegant` · **Location:** `src/runtime/collections.rs:207` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** collection_get/collection_has (lines 207-228) do `entries.iter().flatten().find(|(k,_)| same_value_zero(k,key))`; collection_set (231-252) scans all entries before every insert; collection_delete (254-276) replaces the slot with None and nothing ever compacts the Vec (only the WeakMap/WeakSet GC sweep uses retain); collection_len (203-205) recounts via iter().flatten().count() on every `size` read; collection_entry_at_or_after (302-318) skips tombstones by index on every iterator/forEach step.

**Why it matters:** Every Map/Set operation is O(live+dead entries), so building an n-entry Map is O(n²) and a Map used as a queue (alternating set/delete) grows memory and per-op cost without bound. set_operations.rs amplifies it: eval_set_union inserts each element via collection_set, making Set.prototype.union O(n·m). This is the semantic owner for all keyed collections, so the cost is foundational, not a fast-path detail.

**Verifier notes:** The cited code matches the claim exactly: collection_get/has/set/delete (src/runtime/collections.rs:207-276) all do linear same_value_zero scans over a Vec<Option<(Value,Value)>>, delete writes None tombstones, collection_set always pushes and never reuses None slots, collection_len/logical_entry_count recount on every size read, and the only compaction in the codebase (sweep_dead_weak_entries via gc.rs:640) is gated to WeakMap/WeakSet. set_operations.rs (lines 28-118) builds results via collection_set/collection_has, confirming the quadratic set-operation amplification. The one mitigating rationale — live iterators hold raw index cursors, so tombstones are load-bearing for spec-correct iteration — explains the design but does not refute the cost, and the claim is actually understated in one respect: collection_delete releases the storage-ledger CollectionEntry charge while the tombstone slot persists, so a set/delete workload grows real memory without bound while evading the engine's own resource accounting.

**Verifier correction:** Confirmed as written, with two refinements: (1) tombstones are not an oversight but load-bearing for LiveCollectionIteratorState index cursors (collection_live_iterator.rs:74-112), so any fix must repair/version live-iterator cursors rather than simply compacting; (2) the memory issue is worse than stated — collection_delete releases the ledger CollectionEntry charge while the None slot persists, so tombstone memory is invisible to the VM storage ledger. Minor nit: collection_entry_at_or_after's skip(cursor) is O(1) on slice iterators, so per-step iterator cost is proportional to the tombstone gap, not the whole prefix (full traversal is still O(live+dead)).

**Suggestion:** Replace with a hash index keyed by a SameValueZero-compatible hash over an insertion-ordered entry arena (the classic compact-list + hash design real engines use). The existing index-stable tombstone list already provides spec-correct live-iterator semantics; add the hash as a side index and compact when dead ratio exceeds a threshold, fixing iterator cursors via a remap.

### [MEDIUM] RegExp Symbol.match/matchAll/search/test bypass the receiver's `exec` property, so exec overrides and subclasses are ignored

- **Kind:** `risk` · **Location:** `src/runtime/native/builtins/regexp.rs:646` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed

**Evidence:** regexp_match_text (646-653) and eval_regexp_prototype_symbol_search (204-222) call self.regexp_exec, which goes straight to regexp_receiver_data + regexp_find_utf16 (568-588); regexp_test_code_units (537-557) likewise. The spec's RegExpExec must Get(R,'exec') and call it if callable. eval_regexp_prototype_symbol_match_all (655-688) even constructs a fresh internal RegExp (create_regexp_object_from_utf16) and drives that instead of the receiver.

**Why it matters:** This duplicates matching semantics below the observable protocol: `re.exec = ...` or `class MyRe extends RegExp { exec(){...} }` changes exec but not match/replace/search/split behavior, diverging from every conforming engine. It is exactly the 'parallel path reproduces dispatch outside the widest facade' failure mode the AS program exists to eliminate, but no AS item names RegExpExec.

**Verifier notes:** The core mechanism is verified: RegExp[Symbol.match] (via regexp_match_text, regexp.rs:646-653), [Symbol.search] (212), [Symbol.replace] (via string_regexp_exec_match), test (regexp_test_code_units, 537-557), and [Symbol.matchAll] (655-688, which builds a fresh internal regexp ignoring SpeciesConstructor) all route to internal regexp_exec_code_units/regexp_test_utf16 with no Get(R,"exec") dispatch, so `re.exec = fn` or `RegExp.prototype.exec = fn` overrides are ignored by those methods, diverging from spec RegExpExec. However, the claim is wrong about split: regexp/split.rs:141-166 (regexp_split_exec) implements a fully spec-shaped RegExpExec — Get(splitter,"exec"), callable check, call, builtin fallback, TypeError otherwise — and split.rs:50-79 even honors constructor/Symbol.species, so [Symbol.split] behavior DOES follow exec overrides exactly as conforming engines do. The subclass repro also fails as stated: create_regexp_object_from_utf16 (248-269) ignores new.target and always installs the builtin prototype, so `class MyRe extends RegExp` instances never see MyRe.prototype.exec anyway (a separate upstream deviation). The doc framing is accurate: the widest-facade/parallel-path terminology exists in docs/semantic-architecture-inventory.md and no AS item names RegExpExec.

**Verifier correction:** RegExp[Symbol.match]/[Symbol.matchAll]/[Symbol.search]/[Symbol.replace] and RegExp.prototype.test bypass the spec's RegExpExec protocol (never Get(R,"exec")), so exec overrides via own property or prototype patching are ignored by those methods — but NOT by [Symbol.split]: regexp/split.rs:141-166 already implements conforming RegExpExec dispatch (Get "exec", call if callable, builtin fallback) plus SpeciesConstructor, proving the correct pattern exists in-tree while the other four methods remain parallel paths. Drop split from the affected list. The subclass repro is invalid as written because RegExp construction ignores new.target (create_regexp_object_from_utf16 always uses the builtin prototype), a distinct deviation; use `re.exec = fn` or `RegExp.prototype.exec = fn` as the repro. String.prototype.match/search/replace additionally skip Symbol-protocol dispatch entirely (string_regexp.rs:42-118), widening the divergence beyond the cited file.

**Suggestion:** Introduce one RegExpExec abstract operation (Get 'exec', call if callable, validate object-or-null, else built-in exec) and route all Symbol.* methods through it; keep the internal matcher as the guarded backend for the unmodified-receiver case.

### [MEDIUM] RegExp string iterator and %TypedArray% iterators are eager snapshots, not lazy live iterators

- **Kind:** `risk` · **Location:** `src/runtime/native/builtins/regexp.rs:194` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** eval_regexp_prototype_symbol_match_all (194-202) computes ALL matches up front via regexp_match_all_results and wraps them in a snapshot collection iterator (create_tagged_collection_iterator_object). typed_array/methods.rs eval_typed_array_iterator (288-310) materializes every key/value/entry (allocating one array per entry for entries()) before returning the iterator. Each loop iteration in regexp_match_all_results also re-encodes the input to UTF-16 (regexp_exec, regexp.rs:564) and advance_utf16_index (725-745) re-encodes the whole string per empty match — O(n) per step, O(n²) overall.

**Why it matters:** Spec %RegExpStringIterator% and %ArrayIterator% over typed arrays are lazy: they observe lastIndex mutation, buffer detach/resize, and element writes at each next() call, and do bounded work per step. The snapshot diverges observably (a detach after .values() should make next() throw; here stale values keep flowing) and does unbounded eager work even if the consumer takes one element. Map/Set already have a correct live-iterator mechanism (LiveCollection), so this is a two-regimes split of one concept.

**Verifier notes:** Every cited code location matches the claim: eval_regexp_prototype_symbol_match_all (regexp.rs:194-202) eagerly exhausts all matches via regexp_match_all_results (655-688) into a snapshot Vec wrapped by create_tagged_collection_iterator_object (map_set.rs:559), with regexp_exec (564) re-encoding the input to UTF-16 per iteration and advance_utf16_index (726) re-collecting the whole string per empty match (O(n) per step, O(n²) worst case, though metered by charge_runtime_steps). eval_typed_array_iterator (typed_array/methods.rs:288-310) materializes every key/value/entry eagerly, and install.rs:136 aliases Symbol.iterator to values, so for-of snapshots too. The divergence is reachable: the engine supports ArrayBuffer transfer (detach) and resize (array_buffer.rs:44-48, 101-109, 156+), out-of-bounds is checked only at iterator creation (methods.rs:260-266), and the snapshot has no back-reference or invalidation hook, so post-detach next() yields stale values instead of the spec-required TypeError. The regimes split is real and understated — plain Arrays use lazy LiveArray (array/iterator.rs:151-195) and Map/Set use LiveCollection, making three regimes; docs/semantic-architecture-inventory.md:169 records the snapshot state as inventory, not a guarding invariant.

**Verifier correction:** One sub-point of the rationale is overstated: spec %RegExpStringIterator% species-clones the regexp at creation, so mutations to the original regexp's lastIndex after matchAll are not observed even per spec, and since this engine never does dynamic exec dispatch, the regexp-side snapshot produces the same match values as a lazy iterator would. The regexp defect is therefore eager unbounded (metered) O(n²) work plus premature limit errors ("RegExp matchAll result exceeded array limit" thrown at matchAll() even if the consumer takes one element), not stale values. The observable value-level divergence (stale elements flowing after detach/resize/writes instead of TypeError/live reads) applies to the typed-array iterator, which is fully confirmed and reachable via ArrayBuffer.prototype.transfer/resize. Also, the regimes split is wider than claimed: plain Arrays already use a lazy LiveArray iterator in addition to Map/Set's LiveCollection, so typed arrays and RegExp matchAll are the outliers among three coexisting iterator regimes.

**Suggestion:** Extend the existing live-iterator arena variants (as AS-09s did for Array) with a typed-array-view variant (owner object + index, re-validating bounds per step) and a RegExpStringIterator variant (receiver + input + global/unicode + done), retiring snapshot use for these built-ins.

### [MEDIUM] Uint8ClampedArray rounding converts the float to a String and parses it back on every element write

- **Kind:** `hack` · **Location:** `src/runtime/object/typed_array.rs:790` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** to_uint8_clamp: `rounded.to_string().parse::<u8>().unwrap_or(0)` (line 790) after computing ties-to-even rounding in f64; date/support.rs has the same idiom twice: date_integer_from_number `format!("{integer:.0}").parse::<i64>()` (lines 122-127) and integer_to_number `value.to_string().parse::<f64>()` (129-134), invoked on every Date getter/setter/component conversion.

**Why it matters:** These are lint-dodges around the forbidden `as` truncating cast (note the deliberate #[allow(cast_possible_truncation)] used for to_float32 right below). Formatting to a heap String and reparsing per element write / per Date operation is orders of magnitude slower than a checked conversion, and `.unwrap_or(0)`/error mapping silently launders any future logic error into a wrong value or misleading 'runtime' error instead of an invariant failure.

**Verifier notes:** All three cited sites exist verbatim: `rounded.to_string().parse::<u8>().unwrap_or(0)` at src/runtime/object/typed_array.rs:790 (invoked per element write via the Uint8Clamped arm at line 374), and `format!("{integer:.0}").parse::<i64>()` / `value.to_string().parse::<f64>()` at src/runtime/native/builtins/date/support.rs:122-134, reached from time_clip, integer_component, date_value_to_number, and component getters, i.e. essentially every Date operation. Cargo.toml denies cast_possible_truncation/cast_sign_loss (plus pedantic/nursery), and to_float32 directly below line 790 carries a targeted #[allow(clippy::cast_possible_truncation)], confirming the string roundtrips are workarounds for the forbidden `as` cast rather than a necessary technique. In every case the value is already range-guarded before formatting (clamped to integer-valued [0,255]; MAX_TIME_MS_NUMBER checked by both callers; i64 decimal always parses as f64), so the failure branches (unwrap_or(0) and the Error::runtime mappings) are dead code that would silently launder any future invariant break — exactly as the auditor argues. No hidden guard or documented invariant refutes the claim; severity medium is fair for a heap-allocating format/parse on hot conversion paths that is correctness-neutral today.

**Verifier correction:** Confirmed as written; two clarifications. (1) The date file's actual path is src/runtime/native/builtins/date/support.rs (cited line numbers are exact). (2) The silent `unwrap_or(0)` is partly a consequence of `unwrap_used`/`expect_used`/`panic` also being denied in Cargo.toml, but the codebase's own precedent (the #[allow] on to_float32) shows the intended fix: a checked or documented cast with a targeted allow, which would eliminate both the per-operation String allocation and the dead error-laundering branches.

**Suggestion:** Since rounded is proven in [0,255], use `u8::try_from(rounded as u32)` under a scoped, commented allow (precedent exists in the same file), or a small shared checked f64→integer helper (e.g. compare-and-subtract or f64::to_bits based) used by both typed arrays and Date. One owner for 'exact f64↔integer conversion' would delete three ad-hoc string round-trips.

### [MEDIUM] Date.parse accepts only a rigid YYYY-MM-DD[THH:MM:SS[.fff]][Z] subset, silently accepts garbage fractions, and cannot parse the engine's own toString output

- **Kind:** `risk` · **Location:** `src/runtime/native/builtins/date/support.rs:213` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** parse_date_string (213-236) requires exactly 4-digit year, month and day present (no '2020' or '2020-05', no ±YYYYYY extended years); parse_time_part (249-277) requires all of hh:mm:ss (spec allows THH:mm) and only strips a literal 'Z' (no ±hh:mm offsets). parse_millisecond_fraction (289-303) returns Ok(0) on a non-digit, so '2020-01-01T00:00:00.xyz' parses as a valid date. format_date_time_string output ('Sat Jul 12 2026 10:00:00 GMT+0000 (UTC)') is unparseable by parse_date_string, so new Date(String(d)) is Invalid.

**Why it matters:** The spec requires Date.parse to accept the full Date Time String Format plus the outputs of toString/toUTCString/toISOString; here valid inputs are rejected, invalid inputs are accepted (fraction band-aid), and Date round-tripping through its own string form is broken. Combined with MAX_COMPONENT_ABS=1e6 in integer_component (144-151) rejecting spec-valid large components, Date construction semantics live in arbitrary local cutoffs rather than the shared numeric owners.

**Verifier notes:** Every factual assertion checks out at the cited locations: parse_date_string (support.rs:213) is the sole parser for Date.parse and new Date(string), and parse_date_part/parse_time_part reject spec-conforming forms (date-only "2020"/"2020-05", THH:mm, ±hh:mm offsets, ±YYYYYY expanded years) — a hard violation since conforming strings must be interpreted. parse_millisecond_fraction (289-303) does return Ok(0) on non-digits, so "...T00:00:00.xyz" and even ".12x" parse as valid with 0 ms (and take(3) ignores any garbage after three digits). format_date_time_string output ("Sun Jul 12 2026 10:00:00 GMT+0000 (UTC)") splits on the 'T' inside "GMT" and fails parse_date_part, so Date.parse(d.toString()) and Date.parse(d.toUTCString()) are NaN, breaking the spec's round-trip expectation; toISOString round-trip also breaks for expanded years. MAX_COMPONENT_ABS (support.rs:144-151) maps to DateValue::Invalid at every caller, rejecting spec-valid calls like new Date(2020,0,1,0,0,0,86400000) or setMilliseconds(2e6). No fallback parser, guard, or documented invariant exists anywhere in the date module or tests.

**Verifier correction:** One nuance: accepting garbage fractions is technically spec-permitted (non-conforming strings may fall back to implementation-defined behavior), so only the rejection of conforming strings is a strict violation — but the acceptance is clearly accidental (silently truncates ".12x" to 0 ms) rather than a deliberate heuristic. Also the breakage is slightly broader than claimed: characters after the third fraction digit are never validated at all ("...00.123garbage" parses as valid), and toISOString round-trip fails for expanded-year dates because format_iso_year emits ±YYYYYY which the parser cannot read.

**Suggestion:** Implement the spec's Date Time String Format grammar exactly (optional month/day, optional time with optional seconds/fraction, ±hh:mm offsets, extended years), reject trailing garbage, and add a parser for the engine's own toString/toUTCString forms; delete MAX_COMPONENT_ABS in favor of letting MakeDay/MakeTime/TimeClip do the range enforcement they already implement.

**Verifier notes:** The code facts are accurate: get_timezone_offset returns Number(0.0) for valid dates (mod.rs:426-435), local getters alias UTC getters (mod.rs:354-424), local/UTC setter kinds share one dispatch arm (mod.rs:189-217), and format_time_only_string hard-codes "GMT+0000 (UTC)" (support.rs:424). **Verifier correction:** Local-time-is-UTC is a deliberate, documented, test-pinned deviation (docs/project-plan.md:648), not silent divergence, and the distinct DateFunctionKind variants preserve the structure needed to add real timezone support later. The only real defect is that Date setters (src/runtime/native/builtins/date/setters.rs, e.g. lines 206-237) short-circuit ToNumber coercion of later arguments when an earlier component coerces to NaN or the receiver's date is invalid, skipping observable valueOf calls the spec requires; that edge-case coercion-order deviation is not covered by the documented deviation note. Optionally, the UTC-only contract could also be surfaced in docs/architecture.md or the semantic inventory rather than only in a project-plan tranche row.

**Suggestion:** Record the UTC-only-timezone decision explicitly (docs + an embedder-configurable timezone offset hook, even if only UTC is shipped), keep local/UTC as distinct dispatch arms delegating to one parameterized implementation, and make setters convert all provided arguments in spec order before validity checks.

Per-iterator prototypes diverge from spec (%MapIteratorPrototype% must be one shared object: Object.getPrototypeOf(m.entries()) !== Object.getPrototypeOf(m.entries()) here) and cost 2 objects + 2 native functions per iterator.

**Verifier notes:** The cited code matches the claim exactly: map_set.rs:52 defines the "\0CollectionIteratorState" constant; define_collection_iterator_state (655-673) installs it as an ordinary non-enumerable own data property whose native-function value encodes the iterator arena id in its kind, and collection_iterator_receiver_state (756-779) recovers that id via the general semantic_own_property_descriptor path. ### [MEDIUM] Promise combinator internal state is modeled as '[[...]]'-named JS object properties driven through the full semantic property machinery

- **Kind:** `inelegant` · **Location:** `src/runtime/native/builtins/promise.rs:28` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** Constants '[[PromiseCapabilityResolve]]', '[[PromiseAllValues]]', '[[PromiseAllRemaining]]' etc. (28-34) are defined as ordinary properties on heap objects (create_promise_internal_state, 609-616). change_promise_all_remaining (644-667) does get_named → Number → finite_nonnegative_integer_to_usize → define_non_enumerable_object_property for every element settlement; begin_combinator_element (combinators.rs 442-465) does two get_named + one define per element call; set_promise_all_value (669-690) interns index.to_string() per element.

**Why it matters:** Spec internal slots (remaining-count, already-called flag, values list) are redundant state stored in the slowest possible representation: each counter update round-trips usize→f64→property→f64→usize through prototype-chain-capable lookups. Nothing prevents future code from making these objects reachable (they are ordinary heap objects with Object.prototype behind them), at which point user code could corrupt combinator invariants. The engine already has a traced typed side-store pattern (PromiseCombinatorElement native-kind carries state/index) — the design stops halfway.

**Verifier notes:** The code matches the claim's description exactly (constants at promise.rs:28-34, JS-object state at 609-616, number round-trip counter at 644-667, per-element get/define traffic in combinators.rs:442-465). However, the 'why bad' reasoning overlooks key context: (1) the state ObjectIds live only inside the Rust-side NativeFunctionKind enum payload and are GC-traced via StrongEdgeReference::Object (native/function/mod.rs:224-231) — they are never JS-reachable, and all read properties are defined as own properties before user code runs, so own-property shadowing makes Object.prototype pollution inert; the corruption risk is admitted to be hypothetical future code. (2) The pattern is not 'stopped halfway' — it is the uniform idiom for closure-like native-function state across the whole promise subsystem (PromiseCapabilityExecutor, PromiseFinallyFunction, combinator elements), and using heap objects buys GC tracing plus max_objects/max_object_properties resource accounting for free in this resource-metered VM. (3) 'Slowest possible' is overstated: get_named hits the own property without a chain walk, and per-element cost is dominated by spec-mandated Get/Call machinery; index.to_string() is needed anyway to populate the spec-required values array.

**Verifier correction:** The factual description is accurate and the stringly-typed "[[...]]" keys with per-access runtime type-check error paths (e.g. "Promise.all remaining state is not numeric" — structurally impossible states handled as runtime errors) plus the usize↔f64 counter round-trip are genuine inelegance. But the state objects are unreachable from JS (ObjectId held only in the Rust NativeFunctionKind payload, own properties defined before user code runs, DefineOwnProperty bypasses prototype setters), the pattern is the deliberate uniform idiom for native-function closure state across the promise subsystem (capability executors, finally functions, combinator elements) that gets GC tracing and resource-limit accounting for free, and the per-element overhead is marginal next to the spec-mandated Get/Call machinery. Downgrade to low: a style/robustness nit, not a medium architectural defect.

**Suggestion:** Move combinator shared state into a typed Context-side record (remaining: usize, already_called: bitset/bool, values: ObjectId, resolve: Value) traced like other Promise associations; the element native-function kind already carries the id. This deletes the '[[...]]' property namespace and the per-settlement property traffic.

## Additional low-severity findings (not separately verified)

### [LOW] Byte-buffer size limit is conflated with the per-object property-count limit

- **Kind:** `inelegant` · **Location:** `src/runtime/native/builtins/typed_array.rs:472` · **Status in project docs:** UNDOCUMENTED

**Evidence:** check_byte_buffer_length: `if length > self.limits.max_object_properties { return Err(Error::limit(format!("typed array byte length exceeded {}", self.limits.max_object_properties))) }` (472-480), used by ArrayBuffer/SharedArrayBuffer construction and transfer. typed_array_iterable_values (typed_array/methods.rs 244) and the Symbol.match loop (regexp.rs 178) also use max_object_properties as an element/match-count cap.

**Why it matters:** A property-count knob silently bounds raw byte allocation and match counts; an embedder raising object-property limits inadvertently raises buffer memory limits and vice versa. AS-05b2c built a dedicated per-kind payload-byte policy (VmStorageKind::ByteBuffer max_payload_bytes, already enforced in resize/grow paths) — construction bypasses it in favor of this proxy limit, so the same buffer is guarded by two unrelated regimes depending on how it grew.

**Suggestion:** Make construction check the same VmStorageKind::ByteBuffer payload policy that resize/grow use, and give iteration caps their own named limit.

### [LOW] Atomics word write takes the wrong bytes on big-endian targets

- **Kind:** `risk` · **Location:** `src/runtime/native/builtins/atomics.rs:471` · **Status in project docs:** UNDOCUMENTED

**Evidence:** write_atomic_word: `let raw_bytes = raw.to_ne_bytes(); let source = raw_bytes.get(..size)...` (458-476). The operand value lives in the low bits of the u64; on a big-endian host to_ne_bytes puts those in the trailing bytes, so slicing the first `size` bytes writes zeros/high bytes for 1/2/4-byte elements. read_atomic_word composes correctly, so reads and writes disagree.

**Why it matters:** A silent endianness assumption inside otherwise carefully checked code; the project's no-unsafe/no-cast discipline is aimed exactly at this class of latent wrongness. All current practical targets are little-endian, so tests cannot catch it, which is what makes it a foundation risk rather than a live bug.

**Suggestion:** Convert through the element-width integer first (e.g. u16/u32 try_from(raw & mask) then .to_ne_bytes()), mirroring read_atomic_word's per-width composition, or reuse TypedArrayElementKind::write's per-kind encoding.

### [LOW] Proxy ownKeys invariant validation and duplicate detection are quadratic in key count

- **Kind:** `inelegant` · **Location:** `src/runtime/native/builtins/proxy.rs:566` · **Status in project docs:** UNDOCUMENTED

**Evidence:** validate_proxy_own_property_keys: for each target key, `!trap_keys.contains(target_key)` (566) and for non-extensible targets `target_keys.iter().any(|key| !trap_keys.contains(key))` (575-576) — O(n·m) Value equality scans. proxy_key_list_from_value: `if keys.contains(&key)` per trap key (613) — O(n²).

**Why it matters:** Both loops are shown: outer iteration over target_keys/trap keys with an inner linear contains over trap_keys. A proxy over an object with thousands of keys (a common reactive-store shape) makes every Object.keys/for-in/getOwnPropertyNames O(n²), and this sits on the semantic [[OwnPropertyKeys]] path, not an optional fast path.

**Suggestion:** Build one hash set of trap keys (string keys by interned atom, symbols by id) and use it for both duplicate detection and the two invariant scans.

### [LOW] Atomics.wait returns 'not-equal' from an agent that can never suspend, and converts timeout after the memory read

- **Kind:** `risk` · **Location:** `src/runtime/native/builtins/atomics.rs:169` · **Status in project docs:** documented (AS-plan)

**Evidence:** eval_atomic_wait (169-188): reads the current word, returns the string 'not-equal' if it differs, converts timeout afterwards (184-186), and only then throws 'the current agent cannot suspend'. The spec converts value then timeout, then checks AgentCanSuspend and throws TypeError before touching memory — a non-suspendable agent must always throw, never return 'not-equal'.

**Why it matters:** Order-of-effects divergence on the concurrency primitive: observable ToNumber side effects fire in the wrong order relative to the read, and the value-dependent 'not-equal' result from a single-threaded agent misrepresents the agent capability model that AS-09ag otherwise carefully documents (agent cluster and wait queues deferred to the embedder).

**Suggestion:** Reorder: convert expected value, convert timeout, then check the (currently constant) can-suspend capability and throw TypeError before any memory access; keep the memory-compare path behind the future embedder-coordinator gate AS-09ag reserves.

## Optimization opportunities (post-cleanup)

### [HIGH] RegExp pattern is recompiled by the regress engine on every exec/test, and compiled once more at construction only to be thrown away

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/native/builtins/regexp/engine.rs:28` · **Status in project docs:** UNDOCUMENTED

**Evidence:** regexp_find_utf16 (line 28) and regexp_test_utf16 (line 61) each call `compile_regexp(pattern, flags)?` per invocation; validate_regexp_pattern_utf16 (lines 15-17) compiles and `.map(drop)`s the result at construction. RegExpValue stores only pattern UTF-16 + flag text; regexp.rs parse_regexp_flags() re-parses the flag string from scratch on every getter/exec/test call (regexp.rs:497, 539, 570).

**Why it matters:** Every `r.test(s)` in a loop pays full regex compilation; a global-match loop in Symbol.match/matchAll compiles the same pattern once per match. The inventory records the transient design as a cross-VM-safety fact, but no AS item owns a VM-local cache, so the cost is unowned. Flag re-parsing is the same pattern in miniature: derived state recomputed because the physical owner stores only source text.

**Suggestion:** Cache the compiled regress::Regex (and parsed RegExpFlags) in RegExpValue behind Rc, invalidated only by RegExp.prototype.compile's replace_regexp_value. This stays VM-local (RegExpValue lives in the ObjectHeap), so the documented no-cross-VM-compiled-state constraint is preserved.

### [LOW] ArrayBuffer.transfer copies the full payload and then discards the moved-out bytes that detach() already returns

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/native/builtins/array_buffer.rs:187` · **Status in project docs:** UNDOCUMENTED

**Evidence:** eval_array_buffer_transfer: `let mut bytes = source.copy_bytes(0, current_length)?; bytes.resize(new_length, 0); ... self.objects.detach_array_buffer(id)?;` (187-195) — detach_array_buffer returns Result<Vec<u8>> (object/typed_array.rs:635-647) whose payload is dropped, so transfer is O(n) copy where the spec models a zero-copy move.

**Why it matters:** Transfer's whole purpose is ownership transfer without copying; the primitive that makes it free (detach returning the Vec) already exists in this codebase and is called two lines later with its result thrown away. Large-buffer transfer (the main use case) pays a needless full copy plus accounting churn.

**Suggestion:** Detach first, take the returned Vec, resize it, and wrap it in the new ByteBuffer (adjusting payload-byte accounting once). Same pattern applies to construct-from-same-content-type typed arrays and set_typed_array_from_typed_array, which currently box every element into Value instead of copying bytes when content types and element kinds allow.

### [MEDIUM] All bulk typed-array operations go element-at-a-time through ObjectHeap lookup + RefCell borrow; with_exclusive_bytes_mut exists but only Atomics uses it

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/native/builtins/typed_array/methods.rs:458` · **Status in project docs:** UNDOCUMENTED

**Evidence:** eval_typed_array_fill (448-465) loops set_typed_array_value per index — each call does objects.object(id) lookup, clones the view, borrows the RefCell, bounds-checks (object/typed_array.rs 691-701, 180-211). write_typed_array_values (706-715, sort write-back) and set_typed_array_from_typed_array (400-429) do the same, additionally boxing every element as Value::Number/BigInt via typed_array_view_values (typed_array.rs 517-527). grep shows with_exclusive_bytes_mut (object/typed_array.rs:213) is called only from atomics.rs. typed_array_property_index (object/typed_array.rs 720-747) also allocates a formatted String per indexed property access to canonicalize the key.

**Why it matters:** The per-element overhead (heap lookup + Rc/RefCell traffic + Option<Value> boxing) dominates every fill/set/sort/copy on the very type whose reason to exist is bulk numeric throughput. The safe one-borrow bulk primitive already exists; the element-kind owner (read/write per kind) already centralizes conversion, so a bulk path would not duplicate semantics.

**Suggestion:** Route fill/set/copyWithin/sort write-back and same-kind set() through one borrow of with_exclusive_bytes_mut with a typed inner loop (or byte memcpy for same element kind), after performing all observable conversions first; add an ASCII-digit fast path to typed_array_property_index before the f64-format canonicalization.

### [LOW] Iterator result objects and per-call ephemeral native functions are allocated afresh on every protocol step

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/native/builtins/iterator_stepping.rs:29` · **Status in project docs:** UNDOCUMENTED

**Evidence:** create_iterator_result_object (29-56) interns 'value' and 'done' keys and builds a two-property object through the generic create path on every next() of every helper/collection iterator; eval_collection_iterator_next_state (map_set.rs 725-754) re-interns both keys per step. Each Promise combinator element allocates one state object plus one or two ephemeral native functions per input element (combinators.rs 345-440); every iterator object allocates its own next native function (map_set.rs 614).

**Why it matters:** Steady-state iteration cost is dominated by allocator and intern traffic that a fixed shape would eliminate. This is the kind of representation win that becomes safe precisely because AS-02/AS-03 centralized the semantics: the result-object shape is now produced in one place.

**Suggestion:** Cache the interned value/done keys and a prebuilt two-slot shape on Context (shapes already exist in object/shape.rs) so next() is one arena push with a known shape; share singleton next functions per iterator brand once iterator state moves off per-object hidden properties.
