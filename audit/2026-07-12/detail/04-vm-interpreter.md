# VM interpreter core (runtime/bytecode)

> Detailed architecture-audit report for rs-quickjs. Subsystem: `vm-interpreter`. Generated by a multi-agent audit on 2026-07-12; every medium/high finding was adversarially verified by an independent agent reading the cited code.

## Subsystem assessment

The interpreter is a hybrid: a flat pc-driven dispatch loop (run_bytecode_state) over mostly linear instructions, with structured constructs (loops, switch, try, for-in/of, with, destructuring) executed by Rust recursion over nested BytecodeBlocks. Resumability for await/yield is achieved not by flattening but by an elaborate 'park and re-execute' model — activation-owned continuation frames hold parked interpreter states, a stack of structured-control records with phase enums, and a single resumed-child (block, completion) slot; on resume the whole instruction tree is re-executed down to the suspension point with phase checks skipping completed work. This model works and is defended by strong invariant checks, but it spreads suspension-awareness through every construct via three inconsistent 'am I resuming' mechanisms (control-record phases, state suspend phases, destructure-continuation presence), and the catch-parameter analysis shows this inconsistency can corrupt the scope stack. Layering discipline is mostly real: fast paths are optimizer-gated with generic fallbacks, and abstract operations are reused rather than duplicated — the main exceptions are the triplicated numeric table, a prototype-skipping linear `in` op, and a legacy property-write path selected by heuristics. The peephole 'linear plan' tier is broader than the docs' 'one workload-shaped reduction' admission suggests: half a dozen mask-indexed multi-instruction windows are benchmark signatures, though all evaluate through shared semantics. The deepest foundation problems are cost-model ones — per-instruction transient-root cloning with global rescans on drop, Rc<Mutex> binding cells that allocate on every read, per-loop-entry plan recompilation, and string-keyed argument readback — which together dominate per-instruction cost and will nullify further micro-optimization until the AS-06-planned explicit frames and slot-indexed locals land. The one high-risk semantic split is the mirrored builtin-global binding-cell/global-property state with manual bidirectional sync at ~10 sites, which the plan's own stop-the-line rules would flag but the inventory does not record.

## Strengths

- Suspension/resume is genuinely re-entrant without side-effect replay: structured-control records are checked out/parked with a cursor discipline (enter/checkout/park/finish in continuation.rs) that detects double-checkout, park mismatch, and unsettled synchronous pops with explicit invariant errors rather than silent corruption.
- The optimization layer is uniformly gated on one optimizer switch (optional_optimizations_enabled) and almost every fast path — array index members, string concat, numeric quickening, linear plan compilation, the reduction plan — declines to a generic semantic path on guard miss, matching the AS-08 contract (the linear `in` op is the one exception found).
- Resource accounting is taken seriously inside fast paths, not just around them: the numeric array reduction charges per-iteration step costs and writes its state back before surfacing a limit error, and storage-ledger reservations use reserve/commit/rollback patterns (e.g. bound-function creation).
- The no-unwrap/no-unchecked-indexing discipline holds throughout 13k lines: every stack pop, tail slice, and index is checked with typed runtime errors, and error-to-Throw conversion is centralized in bytecode_error_completion with consistent source-span annotation.
- Destructuring as an explicit resumable task stack (DestructureContinuation) is a clean design: spec evaluation order falls out of task push order, suspension points re-push the current task, iterator closing walks remaining Array tasks, and every task enumerates its GC roots.
- Abstract operations are genuinely reused from the bytecode layer: iterator step/close, ToPrimitive/ToNumber/ToBoolean, equality, GetMethod, and semantic [[Get]]/[[Set]]/construct dispatch are called rather than re-implemented in most instruction handlers.

## Confirmed findings (adversarially verified)

### [HIGH] Catch parameter with awaited default double-pushes the catch lexical scope on resume

- **Kind:** `risk` · **Location:** `src/runtime/bytecode/control/try_catch.rs:209` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** eval_bytecode_catch gates scope creation on `let resumes = state.is_resuming(); if !resumes { self.push_lexical_scope()?; ...hoist... }`. But when suspension happens inside the catch-param destructure (e.g. `catch ({a, b = await p})` in an async function), eval_resumable_destructure stores the continuation via state.store_destructure_continuation (state.rs:320-332), which creates the suspend record with phase=Running and never sets an awaiting/child-suspended phase — the actual Awaiting state lives in the separate child activation for the default-expression block. is_resuming() (state.rs:181-185) checks `phase != Running`, so on the resume re-execution of the Try instruction, `resumes` is false and lines 210-221 push a SECOND catch scope and re-hoist all param bindings.

**Why it matters:** First-pass bindings (e.g. `a`) were initialized in scope 1, which was never popped (suspension returned early at line 228-230). The resume pass hoists fresh uninitialized TDZ cells in scope 2 and completes the destructure there, so the body reading `a` hits a TDZ ReferenceError, and the single pop_lexical_scope at line 231 removes only scope 2 — scope 1 leaks and misaligns every subsequent scope pop in the function. The root cause is that 'am I resuming' is answered by three different mechanisms (suspend phase, destructure-continuation presence, control-record phase) and this site checks only one. The instruction-level DestructurePattern path is safe only because run_bytecode_state marks ChildSuspended (execution.rs:377-380); the catch param runs outside any instruction loop so nothing marks the state.

**Verifier notes:** try_catch.rs:209-222 gates catch-scope creation solely on state.is_resuming() (phase != Running), but a suspension inside the catch-param destructure exits via store_destructure_continuation (state.rs:320-332), which creates the suspend record with the default phase Running (state.rs:363-374) and never marks Awaiting/ChildSuspended — the Awaiting phase lives in the child activation for the default-expression block. On resume, the parked catch slot state (control_continuation.rs:428,559) reaches line 209 untouched (no begin_run/prepare_run on that path), so resumes=false and a second scope is pushed with fresh TDZ hoists; the first scope was never popped (early return at 228-230), producing the claimed TDZ ReferenceError on already-bound names and a leaked lexical scope after the single pop at line 231. prepare_suspended_run (state.rs:123) even acknowledges Running+destructure as a suspended state, and body-side suspensions do mark the slot state (execution.rs:377-380, linear/segment.rs:130-131), confirming the phase is the intended resume signal that this one exit path fails to set.

**Verifier correction:** Claim verified end-to-end; also reachable via yield in generator catch-param defaults and awaited computed keys, and no test covers it. One side detail is imprecise: the instruction-level DestructurePattern path is safe not because of the ChildSuspended mark (begin_run/prepare_suspended_run reset it to Running before re-execution) but because eval_bytecode_destructure_instruction keys off has_destructure_continuation (destructure.rs:52) and pushes no scopes. Minimal fix: gate try_catch.rs:209 on state.is_resuming() || state.has_destructure_continuation() (or make is_resuming treat a stored destructure continuation as resuming, matching prepare_suspended_run at state.rs:123).

**Suggestion:** Make suspension marking uniform: store_destructure_continuation should set a suspend phase (or eval_bytecode_catch should treat has_destructure_continuation()/has_scope_disposal() as 'resuming', matching the check eval_bytecode_catch_scope already does at line 246). Better: move catch-scope setup into the phase-driven Try control record like every other construct, instead of inferring it from BytecodeState internals.

### [HIGH] Builtin-global bindings are mirrored state with hand-maintained bidirectional sync at ~10 scattered sites

- **Kind:** `risk` · **Location:** `src/runtime/bytecode/ops/assignment.rs:193` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** The same JS global exists twice: as a BindingCell and as a global-object property. Cell-to-property sync: static_bindings.rs:359-386 (assign_bytecode_cell -> sync_builtin_binding_global_property). Property-to-cell sync: assignment.rs:192-197 and 231-235 (`if self.is_global_object_id(*id) { self.sync_global_object_property_binding(...) }`), plus six sites in runtime/property/static_names/mod.rs, with_environment.rs:64, and native/builtins/typed_array.rs:186. Each write path must remember to call the sync manually.

**Why it matters:** Classic redundant state that can desynchronize. Any semantic write path that reaches the global object without passing these bytecode-layer wrappers — Reflect.set(globalThis, ...), Object.defineProperty(globalThis, ...), a Proxy whose target is the global, object spread onto the global, or any future built-in — updates the property but not the cell. The damage is amplified by the linear-plan layer: compiled loop plans cache BindingCells at compile time (linear/mod.rs get_binding_bytecode captures `cell`), so a desynced cell becomes persistently wrong reads inside hot loops with no invalidation hook. The AS-02 unified [[Set]] boundary was built precisely so one owner sees all writes, yet global-binding coherence is enforced outside that boundary.

**Verifier notes:** Every cited site checks out: property-to-cell sync calls exist at assignment.rs:193-197/231-235, six sites in property/static_names/mod.rs, and with_environment.rs:64, while cell-to-property sync lives at static_bindings.rs:359-386 and typed_array.rs:186 — roughly 11 hand-placed sites. The unified [[Set]]/[[DefineOwnProperty]] boundary (Context::set -> semantic_reflect_property_write, mutation.rs:207; semantic_define_own_property_update, descriptor.rs:55) contains no global sync hook, so Reflect.set (object_static.rs:427), Object.defineProperty, proxy dispatch, and the internal set_property_value_with_accessors (property/accessor.rs:25-46, an uncited additional bypass) all update the property without the cell. A concrete bidirectional desync is code-traceable: global_object_property_descriptor (globals.rs:306-342) lazily materializes the property from the cell, after which Reflect.set leaves the identifier-read cell stale, and identifier assignment leaves the materialized property stale (cell-to-property sync effectively fires only for globalThis). No guard, invalidation mechanism, or documented invariant refutes this, and test262 coverage (~10.6%) does not pin the behavior.

**Verifier correction:** Core claim fully verified, with two evidence details corrected: (1) "object spread onto the global" is not a real bypass (spread targets a fresh object) — Object.assign(globalThis, ...) is the real equivalent and does bypass; (2) the linear-plan amplification is overstated: BindingCells are shared interior-mutable cells, so plan-captured cells are the same cells ordinary identifier reads use — plan caching reads the same stale cell rather than creating a new desync class. The defect is actually slightly worse than claimed in one respect: cell-to-property sync (sync_global_object_binding_property) no-ops for every name except globalThis, so var-global cells assigned via identifiers never refresh an already-materialized global-object property, and an additional uncited bypass exists at set_property_value_with_accessors (src/runtime/property/accessor.rs:25-46).

**Suggestion:** Make one representation authoritative: back builtin-global bindings BY the global object property (a binding location that reads/writes through [[Get]]/[[Set]] of the global), or route all global-object [[Set]]/[[DefineOwnProperty]] through a single hook that updates the cell. Delete the per-call-site sync calls.

### [MEDIUM] Numeric binary semantics implemented three times; quickened copies drift together from spec (Pow)

- **Kind:** `microopt-debt` · **Location:** `src/runtime/bytecode/linear/numeric_chain.rs:561` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** `apply_number_binary` (numeric_chain.rs:561-593) re-implements the complete Add/Sub/Mul/Div/Rem/Pow/bitwise/shift table on f64. The identical table exists in `eval_bytecode_number_binary` (bytecode/ops/mod.rs:197-241, optimizer-gated) and in the generic owner `numeric_binary` (runtime/numeric.rs:81). All three use `left.powf(right)`, which follows IEEE pow — `powf(1.0, f64::INFINITY) == 1.0` — while ES Number::exponentiate requires NaN when abs(base)==1 and the exponent is ±Infinity, so `1 ** Infinity` is wrong in every copy.

**Why it matters:** AS-08's own rule (inventory: optimized paths 'may select operands... but may not redefine' semantics) is violated: the quickened layer redefines the arithmetic table instead of layering over one owner. The shared spec bug demonstrates the failure mode concretely — a future Pow fix must be found and applied in three places or the fast and slow paths diverge observably. instruction_window/same_bytecode_binding helpers are likewise copy-pasted across linear/mod.rs, numeric_chain.rs, property_chain.rs.

**Verifier notes:** All three cited copies exist exactly as claimed and each uses bare left.powf(right) (numeric_chain.rs:572, ops/mod.rs:212, numeric.rs:81), with no guard for the ES Number::exponentiate rule anywhere upstream in the operator paths — so 1 ** Infinity observably returns 1 instead of NaN in every path. The claim actually understates the debt: two further copies of the same f64 table exist in function/fast_path.rs:479 and native/builtins/array/flatten.rs:373 (five total), and the codebase demonstrably knows the spec rule because math_pow in native/builtins/math.rs:612-617 implements exactly the required abs(base)==1 && exponent.is_infinite() → NaN guard for Math.pow only, proving the divergence between the guarded builtin and the unguarded operator copies. Minor inaccuracies: the same_bytecode_binding duplicates live in linear/mod.rs, numeric_chain.rs, and function/fast_path.rs (not property_chain.rs, which duplicates only instruction_window), and no AS-08 inventory document is findable in the repo to verify the quoted rule. Severity medium is fair: a real cross-path spec bug on an obscure edge case plus a five-way maintenance hazard.

**Verifier correction:** Numeric binary f64 table is duplicated five times, not three: runtime/numeric.rs:81 (generic owner), bytecode/ops/mod.rs:212 (quickened), bytecode/linear/numeric_chain.rs:572 (linear chain), function/fast_path.rs:479, and native/builtins/array/flatten.rs:373 — all use bare powf, so 1 ** Infinity yields 1 instead of the spec-required NaN in every path. Math.pow (native/builtins/math.rs:612-617) already implements the correct guard, making the operator/builtin divergence directly observable and showing the fix pattern that must currently be replicated in five places. Helper duplication detail corrected: instruction_window is copied in linear/mod.rs, numeric_chain.rs, and property_chain.rs; same_bytecode_binding in linear/mod.rs, numeric_chain.rs, and function/fast_path.rs.

**Suggestion:** Collapse to one `#[inline]` pure fn in runtime/numeric.rs that all three call; fix Pow's ES special cases there once. Deduplicate the window-matching helpers into the linear module root.

### [MEDIUM] Linear 'in' fast path answers own-property only, skipping the prototype chain the generic path walks

- **Kind:** `microopt-debt` · **Location:** `src/runtime/bytecode/linear/in_operator.rs:146` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** confirmed

**Evidence:** eval_in_array_index_mask_binding: `if let Some(has_property) = self.has_own_array_index_for_in(&object, index)? { state.stack.push(Value::Bool(has_property)); return Ok(()); }`. has_own_array_index_for_in -> ObjectHeap::has_own_array_index_if_array (object/array/bytecode.rs:61-74) returns Some(false) whenever the array lacks the OWN index. The generic `in` (eval_bytecode_in -> has_cached_dynamic_property_value) performs full [[HasProperty]] including the prototype chain.

**Why it matters:** Incomplete guard on a workload-shaped superinstruction: for `(i & mask) in arr` compiled into a linear segment, an indexed property inherited from the prototype (Object.defineProperty(Array.prototype, 20, ...) or Object.setPrototypeOf(arr, {20: x})) yields `false` where the language requires `true`. Some(false) should have been the fall-through case, exactly as eval_dynamic_array_index_member does (returns None -> generic get). This is the precise category AS-08 claims to have eliminated: a fast path whose guard miss does NOT reach the generic semantic path.

**Verifier notes:** The code matches the claim: src/runtime/bytecode/linear/in_operator.rs:146 treats Some(false) from has_own_array_index_for_in as a definitive answer and pushes it, while that helper (src/runtime/bytecode/in_operator.rs:33-45 -> ObjectHeap::has_own_array_index_if_array, src/runtime/object/array/bytecode.rs:61-74) checks only Object::get_own_array_index on the receiver — its sole gate is array_length being Some, with no prototype-shape guard or invalidation. The generic path the fast path replaces (eval_bytecode_in -> has_cached_property_lookup_value -> has_property -> ObjectHeap::has -> prototype_has_in_chain, src/runtime/object/prototype.rs:41-67) does a full prototype-chain walk, so `(i & mask) in arr` returns false where the spec (and the engine's own generic path) returns true whenever the index exists only on the prototype chain (e.g. Object.setPrototypeOf(arr, {5:1}) or Array.prototype[5]=x). No mitigating invariant exists: OptimizationMode::Enabled (the default, which gates linear segments at segment.rs:28) is documented as "verified quickening... guarded specialized paths", i.e. semantics-preserving, and the analogous GET fast path (array_index_value_if_array -> array_property_value_by_index, src/runtime/object/array/mod.rs:337-356) explicitly walks the prototype chain, confirming the in-operator fast path is the outlier. Minor nits: the evidence path omits the runtime/ directory component, and eval_dynamic_array_index_member handles prototypes by walking the chain itself rather than "returning None -> generic", but neither affects the substance; medium severity is fair given the exotic trigger (indexed properties on an array's prototype chain) versus a silent wrong boolean in the default optimization mode.

**Verifier correction:** Evidence path should read src/runtime/object/array/bytecode.rs:61-74. Parity example is slightly imprecise: eval_dynamic_array_index_member's array hit path preserves prototype semantics by walking the chain inside array_property_value_by_index (src/runtime/object/array/mod.rs:337-356), not by falling back to the generic get; the fix for `in` could either fall through on Some(false) or walk the chain the same way.

**Suggestion:** Return None (fall back to eval_bytecode_in) when the own index is absent; only answer directly on Some(true).

### [MEDIUM] ForOf control record's transient-root set is a fixed 4-slot array that silently truncates iterator roots

- **Kind:** `risk` · **Location:** `src/runtime/bytecode/control_continuation.rs:356` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** BytecodeControlRecord::transient_root_values builds `[Some(last), values.first().copied(), values.get(1).copied(), resume...]` from the ForOf iterator's roots. But AsyncIteratorContinuation::root_values (abstract_operations/iterator/async_iteration.rs:54-58) yields up to THREE values: source iterator, cached `next` method, and the parked `closing` completion value. With a Protocol source, slots 0-1 are consumed by iterator+next and the closing throw/return value is dropped from the root set while run_bytecode_for_of_action / drive_async_for_of_close executes nested user code (the `return()` call).

**Why it matters:** The rooting contract (AS-05/AS-07) is structurally violated by a hard-coded arity: the full parked-record enumeration (root_values, lines 189-245) is complete, but the checked-out transient variant loses values whenever a record's dynamic root count exceeds the array literal. Today GC is only embedder-invoked so this is latent, but the moment allocation-triggered collection lands (the plan's stated direction), an async for-of closing with a pending throw value can have that exception object collected mid-close. Fixed-size root arrays are a trap for every future field added to a record.

**Verifier notes:** The code shape is as claimed: transient_root_values (control_continuation.rs:345-377) hard-codes 4 slots and drops the async iterator's third root (the parked `closing` completion, async_iteration.rs:54-58) for Protocol sources, while its twin has_traceable_transient_roots (lines 379-406) checks all roots — a real inconsistency, and checked-out records are absent from the parked enumeration (checkout_control uses Option::take). But the claimed failure scenario does not hold: the only path entering run_bytecode_for_of_action with `closing` already set is the post-await resume leg (resume_for_of_close → drive_async_for_of_close → resume_async_iterator_close), and resume_async_iterator_close takes &self — it can run no user code and perform no allocation, so the truncated-scope window contains no possible GC point even under future allocation-triggered collection. During the return() call the auditor cites, `closing` is set (async_iteration.rs:270) only AFTER the transient scope has snapshotted its values (guard at line 267 requires closing.is_none() on entry; scopes clone values at creation, transient_roots.rs:148-171), so the value is unprotected there due to scope-creation timing, not slot truncation — a dynamically sized root array would not fix that leg either.

**Verifier correction:** The fixed 4-slot array in BytecodeControlRecord::transient_root_values does truncate the async ForOf iterator's `closing` root and already disagrees with has_traceable_transient_roots (which can create a scope because of a value the scope then omits) — worth fixing for consistency and as future-proofing against new record fields. However, no reachable window exists where the truncation leaves a live value unrooted at a possible collection point: the only checked-out path with `closing` set runs resume_async_iterator_close, which takes &self and cannot execute user code or allocate. The adjacent, genuinely riskier latent gap is different from the one claimed: on first entry to async_iterator_close, the closing completion is stored after the transient scope is snapshotted, so it is unrooted (regardless of array size) while the user return() method runs — that timing gap, not the arity, is what would need addressing before allocation-triggered GC lands.

**Suggestion:** Make transient_root_values delegate to the same complete enumeration as root_values (a small Vec or iterator chain), and add a debug assertion that the transient set is a superset of the record's completion/iterator values.

### [MEDIUM] Await-resume child routing keyed by deep structural equality of bytecode blocks

- **Kind:** `inelegant` · **Location:** `src/runtime/bytecode/continuation.rs:246` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed

**Evidence:** has_resumed_child: `self.resumed_child.as_ref().is_some_and(|resumed| &resumed.0 == block)`. BytecodeBlock derives PartialEq over `Rc<[BytecodeInstruction]>` + `Rc<[SourceSpan]>` (bytecode/block.rs:31-35). BytecodeInstruction contains f64 literals, so the types are PartialEq-only (no Eq) and Rc comparison cannot short-circuit on pointer identity — every match is a deep recursive comparison of the instruction tree (nested blocks included). take_resumed_bytecode_child is consulted by eval_bytecode_block, scoped blocks, with, nullish-coalescing, pattern defaults, etc.

**Why it matters:** Identity is the semantic requirement (deliver this completion to the exact block that suspended), but the implementation uses structural equality: correctness silently depends on the invariant that re-execution of the parent instruction reaches the same syntactic block before any structurally identical sibling, and each delivery pays an O(block-size) recursive compare. It also forces PartialEq to be derived through the entire instruction hierarchy solely for this mechanism.

**Verifier notes:** The cited code is exactly as claimed: has_resumed_child (src/runtime/bytecode/continuation.rs:245-249) routes await-resume completions via derived structural PartialEq; BytecodeBlock derives PartialEq over Rc<[BytecodeInstruction]>+Rc<[SourceSpan]> (src/bytecode/block.rs:31-35); BytecodeInstruction (src/bytecode/types.rs:439) is PartialEq-only because PushLiteral(Value) contains Value::Number(f64) with a manual PartialEq (src/value/kind.rs:46-51), so std's Rc ptr_eq fast path (Eq-specialized) never fires and a match is a full deep recursive compare of two handles that in fact always point to the same allocations (stored block comes from program_block(), a clone of the same Rcs the parent re-presents on re-dispatch — identity IS the semantic requirement, and Rc::ptr_eq is the codebase idiom elsewhere, e.g. scope.rs:650). Grep confirms continuation.rs:248 is the sole structural == on any bytecode type, so PartialEq is derived through the hierarchy solely for this call. However, two mitigations weaken the claim: (1) the deep compare runs only when a resumed_child is actually stored — has_resumed_child is a cheap None-check on every ordinary eval_bytecode_block call, so the O(block-size) cost is per suspension-delivery, not per consult; (2) spans participate in equality and from_parts rejects unknown source ids, so two distinct syntactic sibling blocks essentially always differ in span tables — the feared misroute needs identical instructions AND identical spans (same compiled source range) plus a handler consulting a different-but-equal block first, and every current consumer (execution.rs:178, control.rs:205/244, parameter/pattern defaults) consults take_resumed_bytecode_child as its first action with its single sub-block, so no concrete misrouting path exists today.

**Verifier correction:** Confirmed as an inelegance: identity routing implemented as deep structural PartialEq, with the entire instruction hierarchy deriving PartialEq solely for continuation.rs:248, and each await-resume delivery into a nested block paying an O(instruction-tree) recursive compare of two clones of the same Rc allocations (Rc::ptr_eq on the instructions field would be O(1), semantically exact, and matches the codebase idiom). Two corrections: the compare is paid once per suspension delivery, not on every eval consult (has_resumed_child is a None-check in the common path), and the sibling-misrouting hazard is effectively guarded because span tables (source-id + offsets, unknown ids rejected in from_parts) participate in equality, pinning equal blocks to the same compiled source range — the fragility is that this guard is incidental and undocumented, not that misrouting is reachable today.

**Suggestion:** Key the resumed child by Rc pointer identity (Rc::ptr_eq on the instructions slice) or assign blocks a compile-time BlockId; drop PartialEq from the instruction tree.

### [MEDIUM] Completion enum conflates spec completions, engine suspension outcomes, and overlapping return variants

- **Kind:** `inelegant` · **Location:** `src/runtime/control/completion.rs:8` · **Status in project docs:** UNDOCUMENTED
- **Verification verdict:** partially-confirmed (corrected severity: low)

**Evidence:** Completion has 10 variants: the 5 spec completion types plus Suspended(PromiseId), GeneratorStart, Yielded, YieldedIteratorResult — driver-internal outcomes that BytecodeOutcome (execution.rs:14-34) then re-encodes. ReturnDirect exists solely to distinguish `return;` from `return expr;` (compiler/mod.rs:304-312) and is handled identically to Return everywhere except two async-generator sites. Four projection methods (into_result, into_function_result, into_call_completion, into_native_value_result) each re-enumerate all 10 variants with subtly different mappings, and every control construct in bytecode/ writes exhaustive 10-arm matches (control.rs:354-368, structured_do_while.rs:57-90, instruction_stack.rs:229-235, for_of.rs:76-84, destructure.rs:589-597, ...).

**Why it matters:** The AS-04 target was typed separation of JavaScript completion from engine control, but the suspension outcomes added by AS-06 were folded INTO the JS completion enum, so every consumer must now reason about (and root, and route) four extra variants that are not JS semantics, and 'which projection do I call' is an asymmetric per-call-site decision. Adding one suspension kind touches dozens of matches. Overlapping variants (Return/ReturnDirect, Yielded/YieldedIteratorResult) encode caller context as data.

**Verifier notes:** The structural facts are accurate: Completion has 10 variants including 4 suspension outcomes that BytecodeOutcome (execution.rs:14-45, 376-414) re-encodes 1:1, four projection methods each re-enumerate all variants, and every cited exhaustive match exists. However the "why bad" reasoning overreaches: AS-04's documented target (architecture-stabilization-plan.md:973) was separating JS completion from engine FAILURES, and that boundary is intact — every projection converts escaped suspensions into Error::runtime, a typed guard. The docs explicitly record the deliberate choice that suspension outcomes "are typed completions rather than a parallel interpreter", the suspends_execution() helper (completion.rs:29) already groups the 4 variants at 15+ routing sites, and ReturnDirect is not mere caller context: it marks a return value that must not be re-awaited by async-generator resumption (constructed at async_generator.rs:412 for already-awaited resume-returns, distinguished at async_generator.rs:527 and async_delegate.rs:34/43), just as Yielded vs YieldedIteratorResult carry spec-distinct payloads (raw iterator-result identity plus inner-await flag, async_generator.rs:505-519).

**Verifier correction:** Real but deliberate and guarded verbosity, not a boundary violation. The dual Completion/BytecodeOutcome encoding and four hand-written projections with near-identical suspension escape arms are genuine duplication, and adding a suspension kind does touch many matches — but exhaustive matching is compile-enforced, each control site genuinely decides park vs propagate vs error, suspends_execution() already abstracts the grouping where routing is uniform, and the docs record choosing typed completions over a parallel interpreter/second channel. ReturnDirect and YieldedIteratorResult encode observable async-generator semantics (skip re-await; preserve iterator-result identity), not redundant caller context. A fair framing is a low-severity cleanup opportunity: e.g., a shared suspension sub-enum or a single escape-arm helper the four projections delegate to.

**Suggestion:** Split into `Completion` (spec 5) and a driver-level `Outcome { Done(Completion), Suspended(...), Yielded(...) }` — BytecodeOutcome already is that type; push it down so construct drivers match 5+1 instead of 10. Replace ReturnDirect with Return carrying an `explicit_value: bool` only where the two async-generator sites need it, or normalize at the generator boundary.

### [MEDIUM] While/do-while/for loop drivers are three hand-rolled copies of the same phase machine with inlined completion routing

## Optimization opportunities (post-cleanup)

### [HIGH] Per-instruction transient-root scope clones the whole operand stack into a global Mutex<Vec> and rescans it on drop

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/bytecode/execution.rs:359` · **Status in project docs:** UNDOCUMENTED

**Evidence:** run_bytecode_state creates a transient_root_scope for EVERY instruction over `state.root_values()`/`synchronous_root_values()` (whole operand stack + last + suspend roots). transient_roots.rs:148-171 then, per traceable value: locks a parking_lot Mutex (inside an Rc, single-threaded), calls storage_ledger.grow_count per value, and clones the Value into a shared Vec. Drop (lines 174-186) runs `roots.retain(|root| root.scope != self.scope)` — a scan of the ENTIRE global transient-root vector, including roots owned by all enclosing scopes.

**Why it matters:** Per-instruction cost is O(traceable stack values) clones + ledger ops, plus O(total live transient roots) on drop. Code holding N object values on the operand stack (building an N-element array/object literal, N-argument call) executes each of its ~N instructions at O(N) — accidental O(N^2) with a mutex lock per value in the innermost dispatch loop. Since GC is currently only embedder-invoked via &mut Vm (api/embedding.rs:463) and can never run mid-instruction, this per-step rooting buys accounting precision at enormous cost. This one mechanism likely dominates the interpreter's fixed per-instruction overhead and will mask every other optimization.

**Suggestion:** Root the operand stack structurally: the BytecodeState is reachable from the activation frame while running (as parked states already are), so GC root capture can walk live states directly instead of copying values into a side vector per instruction. Reserve transient scopes for genuinely detached temporaries; make scope drop O(own values) by recording a range instead of retain-scanning.

### [HIGH] Every binding read allocates a Vec, locks a Mutex, and clones — including in the 'fast' linear ops

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/bytecode/linear/mod.rs:529` · **Status in project docs:** UNDOCUMENTED

**Evidence:** All lifted linear ops read through `cell.value(binding.name())` (e.g. LoadBinding at linear/mod.rs:526-530, every peephole op, the reduction plan). BindingCell is `Rc<Mutex<Binding>>` (binding/scope.rs:530) and BindingCell::value (scope.rs:560-575) builds a `visited: Vec` for alias-cycle detection, pushing a clone of the cell on the FIRST iteration even in the common non-alias case, then locks the mutex and clones the whole Binding state per read.

**Why it matters:** The interpreter's most frequent operation — reading a local — costs a heap allocation (Vec push), an Rc clone, a mutex lock (pointless under Rc's single-threaded constraint), and a Value clone. The peephole/superinstruction layer exists to make loops fast, but it inherits this cost on every lifted load/store, capping what any dispatch-level optimization can achieve. Alias chains are rare and statically knowable; cycle detection per read is paying for the exotic case on every access.

**Suggestion:** This is the representation change that the AS-06 explicit-frame plan already points at: slot-indexed local frames with plain Vec<Value> storage and Cell-free access for non-captured bindings; keep Rc cells only for captured/upvalue bindings; move alias-cycle detection to alias-creation time so reads are a single indexed load.

### [MEDIUM] Spread and apply argument lists are read back through per-index string-keyed property lookups

- **Kind:** `perf-opportunity` · **Location:** `src/runtime/bytecode/spread.rs:146` · **Status in project docs:** UNDOCUMENTED

**Evidence:** spread_call_arguments: `for index in 0..len { args.push(self.get_named(packed, &index.to_string())?); }` — the engine builds the packed array itself in eval_bytecode_collect_spread_args and then immediately re-reads it with a heap-allocated decimal string and a named property lookup per element. Same pattern in call/bound.rs create_list_from_array_like:118-121 for the generic apply path. eval_bound_function (bound.rs:156) additionally clones the whole BoundFunction (args Vec included) on every call.

**Why it matters:** CollectSpreadArgs -> packed array -> string-keyed readback is a double materialization with O(n) string allocations and dynamic lookups for data the engine had contiguously moments earlier; the indexed fast read (array_index_value_if_array) used elsewhere in this same subsystem is bypassed. Spread-heavy and apply-heavy code pays several times the necessary cost.

**Suggestion:** Pass spread arguments as a Vec<Value> through the instruction (or read back via the indexed array accessor); make eval_bound_function borrow the bound record or stash target/this before extending args.
